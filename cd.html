§ 3.1

SDD Recap — Synthesized vs Inherited, S- vs L-Attributed

A Syntax-Directed Definition (SDD) is a generalisation of a context-free grammar in which each grammar symbol carries a set of attributes, and each production carries a set of semantic rules that compute attribute values. SDDs are the high-level specification of translation; semantic rules describe what to compute, not when.

Two Kinds of Attributes

AttributeComputed fromDirection in tree
SynthesizedChildren of the nodeBottom-up (leaves → root)
InheritedParent and/or left-siblingsTop-down / left-to-right

S-Attributed Definition

An SDD with only synthesized attributes. Easy to implement on a bottom-up parser by keeping attribute values on the parser's value stack — at every reduction A → α, A's attributes are computed from α's attributes already on the stack.

L-Attributed Definition

An SDD is L-attributed if every inherited attribute of Xj in a production A → X1 X2 … Xn depends only on:

  • The attributes of the symbols to the left of Xj: X1, …, Xj−1 (this is what the L stands for), and
  • The inherited attributes of A.
Theorem. Every inherited attribute in an L-attributed definition can be computed by a pre-order (depth-first, left-to-right) traversal of the parse tree. This is the natural order of evaluation.
Every S-attributed SDD is also L-attributed (synthesized attributes trivially satisfy the L-attributed restriction). The converse is not true.

Worked Example 1 — Classify the SDD

Problem 1

Classify the following SDD as S-attributed, L-attributed, or neither.

A → B C   { A.s = B.s + C.s ;  C.i = B.s }
B → b      { B.s = 1 }
C → c      { C.s = C.i + 2 }
Solution

A.s is synthesized (uses B.s, C.s — both children).
C.i is inherited; it depends only on B.s, which is the attribute of a left-sibling of C in the production A → B C.
There are no inherited attributes that depend on right-siblings or on synthesized attributes of the parent. ⇒ L-attributed, but not S-attributed (because of the inherited C.i).

Worked Example 2 — Why this SDD is NOT L-attributed

Problem 2
A → L M   { L.i = l(M.s) ;  M.i = m(L.s) ;  A.s = f(M.s) }
A → Q R   { R.i = r(A.i) ;  Q.i = q(R.s) ;  A.s = f(Q.s) }

Is the SDD L-attributed? Justify.

Solution

Consider L.i = l(M.s) in A → L M. L is the left-most symbol; M is to its right. L's inherited attribute depends on M's synthesized attribute — i.e., on a right-sibling. This violates the L-attributed restriction.
Similarly Q.i = q(R.s) in A → Q R depends on a right-sibling's synthesized attribute. ⇒ The SDD is not L-attributed.

S-Attributed SDD — Building a Syntax Tree

A syntax tree condenses the parse tree by keeping only operators and operands. Two helpers are used:

  • new Node(op, left, right) — interior node for a binary operator.
  • new Leaf(token, attr) — leaf for an identifier or number.

Solution

Extends with if, while, Seq, conditions and assignment expressions:

§ 3.5

Translation Schemes (SDT)

A Syntax-Directed Translation Scheme (SDT) is a context-free grammar in which semantic actions — written between { } — are inserted on the RHS of productions. SDTs are more implementation-oriented than SDDs because they fix the order of evaluation. Yacc uses translation schemes.

Design Rules for SDTs

  • If the action uses only synthesized attributes, place it at the end of the production.
  • For an L-attributed SDD, the inherited attribute of a symbol on the RHS must be computed in an action that appears before that symbol.
  • A synthesized attribute on the LHS can only be computed when all attributes it references have been computed — usually placed at the end.

Problematic SDT — Why the Position Matters

The grammar with embedded actions is parsed first ignoring the actions. The actions become extra leaves in the parse tree; a pre-order traversal visits each leaf and runs its action.

Problem 1 — Trace the SDT for input 5 + 4 − 2
E → T R
R → + T { print('+'); } R1
R → − T { print('−'); } R1
R → ε
T → F
F → digit { print(digit.lexval); }
Output

Pre-order on the parse tree of 5+4−2 visits the digits before the operator action attached after each T:

5  4  +  2  −

That is the postfix of the original expression. Notice the action is placed after the T, not after the operator token — this is the trick that converts infix to postfix during parsing.

Problem 2 — Infix-to-prefix SDT
L → E n
E → { print('+'); } E + T
E → T
T → { print('*'); } T * F
T → F
F → ( E )
F → digit { print(digit.lexval); }
Output for 3 * 4 + 5

Pre-order on the action-decorated tree prints the operator action before visiting the operand sub-trees, giving the prefix form: + * 3 4 5.

Converting an SDD to an SDT

Procedure

  1. Place the action computing the inherited attributes of a symbol before that symbol on the RHS.
  2. Place the action computing the synthesized attributes of the LHS at the end of the production.
  3. Build the parse tree (ignoring actions); add the actions as extra leaves; perform a pre-order traversal to execute them.

§ 3.7

Postfix SDT & LR-Parser Implementation

A postfix SDT is one in which all actions are placed at the end of every production. Postfix SDTs correspond exactly to S-attributed SDDs and can be implemented during bottom-up parsing by extending the parser stack with a value stack.

Extended Parser Stack

For every A → α reduction, before the reduction the values of α already sit on the value stack at positions s[top], s[top−1], …. The action computes A's attribute and replaces those slots:

// E → E1 + T
{ stack[top-2].val = stack[top-2].val + stack[top].val ; top = top - 2 ; }

Implementing L-Attributed SDD on an LR Parser — Marker Non-Terminals

An LR parser only fires actions on a reduction. To execute an action in the middle of a production, introduce a fresh non-terminal M whose only production is M → ε; attach the embedded action to that ε-production:

A → α { a } β        becomes        A → α M β
                                     M → ε { a }
Problem 1 — Postfix SDT desk calculator on a parser stack
E → E1 + T  { stack[top-2].val += stack[top].val ; top -= 2 ; }
E → T
T → T1 * F  { stack[top-2].val *= stack[top].val ; top -= 2 ; }
T → F
F → ( E )   { stack[top-2].val = stack[top-1].val ; top -= 2 ; }
F → digit
Why it works

Every action involves only synthesized values that already lie on the stack at positions top, top−1, top−2. After computing the new value into the position of the LHS, the pointer is decremented to expose it as the new top.

Problem 2 — While-loop with markers
S → while ( M C ) N S1
M → ε  { L1 = newlabel() ;
          L2 = newlabel() ;
          C.true  = L2 ;
          C.false = S.next ; }
N → ε  { S1.next = L1 ; }
S → while ( M C ) N S1   { S.code = L1 || C.code || L2 || S1.code ; }
Trace (high-level)
  1. Parser shifts while (; reduces M → ε ⇒ creates labels and inherited values, pushing them as M's record onto the parser/value stack.
  2. C is parsed; its inherited C.true, C.false were taken from M's slot.
  3. Parser shifts ); reduces N → ε ⇒ S1.next = M's L1 (read from s[top-3]).
  4. S1 is parsed.
  5. Final reduction S → while ( M C ) N S1 reads code/labels off the stack and emits the loop body.
The marker-non-terminal trick changes the grammar slightly but does not change the language; every L-attributed SDT can be made into an LR-implementable postfix-style scheme this way.
§ 3.8

Intermediate Code Generation — Overview

Intermediate Code (IC) is the program representation that sits between the high-level source and the machine target. The Intermediate Code Generator takes an annotated syntax tree from the semantic analyser and produces IC for the optimiser/code generator.

Why an Intermediate Form?

  • Retargeting: N front-ends + M back-ends require only N + M components rather than N × M.
  • Enables machine-independent optimisations.
  • Cleanly separates analysis (front end) from synthesis (back end).

Three Ways to Classify an IR

AxisExamples
High-level (close to source)Syntax tree, DAG, Java bytecode
Low-level (close to machine)Three-Address Code, SSA
Language-specificJava bytecode, Pascal P-code
Language-independentThree-address code
GraphicalSyntax tree, DAG
LinearTAC (quad/trip), bytecode, postfix
A good IR is fairly independent of source and target — this maximises retargetability. High-level IRs make source-level optimisations easy; low-level IRs help with register allocation and instruction selection.
§ 3.9

Directed Acyclic Graph (DAG)

A DAG is a syntax tree in which every identical subexpression is represented by a single shared node. Use it to identify common subexpressions, eliminate redundant computations, and find values that can be hoisted out of a block.

SDD to Build a DAG

Same SDD as for the syntax tree, but the constructors check first:

if (an identical node already exists)
    return that existing node ;
else
    create and return a new node ;

Assignments of the form x := y need not create a new node — they only relabel an existing one.

Applications

  • Detect common subexpressions at the block level.
  • Reduce the number of computations — calculate once, reuse everywhere.
  • Identify names whose value is computed outside the block but used inside.
  • Identify names whose computed value is live on exit from the block.
Problem 1 — DAG for ((x+y) − ((x+y)·(x−y))) + ((x+y)·(x−y))
Solution

Identify common subexpressions: (x+y), (x−y), and the product (x+y)·(x−y) all appear twice.

+ / \ - * <-- shared product node /| |\ + | x y / \| x y - / \ x y (x+y) and (x-y) are each single shared leaf-pair nodes.

The DAG has nodes: x, y, +(x,y), −(x,y), *(+,−), −(+, *), +(−(+, *), *) — only 7 distinct nodes instead of the syntax tree's many duplicates.

Problem 2 — DAG for a + a · (b − c) + (b − c) · d
Solution

Common subexpression: (b − c) appears twice. Build it once and reuse.

+ / \ + * / \ |\ a * | d |\| a - /\ b c <-- (b - c) is a single shared node

Number of distinct interior nodes drops from 5 to 4 because the − node is shared.

§ 3.11

TAC for 1-D & 2-D Arrays

Address Calculation Formulas

1-D : addr(A[i]) = A + W · (i − LB)
2-D row-major : addr(A[i][j]) = A + W · [ N · (i − Lr) + (j − Lc) ]
2-D column-major : addr(A[i][j]) = A + W · [ (i − Lr) + M · (j − Lc) ]

where W = element width, N = number of columns, M = number of rows, Lr = lower-row index, Lc = lower-column index.

Default assumptions in slides: row-major, integer arrays, W = 4 bytes; if size is unspecified, treat the array as m × n.

Operator Conventions Used in the Slides

  • Unary x = −y ⇒ op = "−", arg2 = NULL.
  • Copy x = y ⇒ op = "=", arg2 = NULL.
  • param a ⇒ op = "param", arg2 = result = NULL.

§ 3.13

Static Single-Assignment (SSA) & the φ-Function

SSA is an IR in which every variable is assigned exactly once but may be used many times. Each original variable is split into versions x1, x2, …, and special φ-functions are inserted at control-flow merge points to choose the correct version depending on which incoming edge was taken.

The φ-Function

  • x = φ(x1, x2, …, xk) chooses among k incoming definitions.
  • Number of arguments = number of incoming control-flow edges to that block.
  • The argument selected corresponds to the path actually taken at run-time.
  • φ is purely notational: real machines do not execute it; back-ends lower it back to copies.
SSA makes data-flow optimisations like constant propagation, dead-code elimination, and value numbering dramatically simpler — there is exactly one definition per name, so def-use chains are explicit. Used by GCC, LLVM, HotSpot, SUIF.

§ 3.14

Control-Flow Graph · Basic Blocks · CFG → SSA

Basic Block

A basic block is a maximal sequence of consecutive instructions such that control can enter only at the first instruction (the leader) and leave only at the last instruction. Every instruction belongs to exactly one basic block.

Rules for Identifying Leaders

  1. The first three-address instruction is a leader.
  2. The target of a conditional or unconditional jump is a leader.
  3. The instruction immediately after a (conditional or unconditional) jump is a leader.

A basic block runs from a leader up to (but not including) the next leader.

Constructing the CFG

  • Nodes = basic blocks (numbered B1, B2, …); add an ENTRY and EXIT super-node.
  • Edge B → B' if (a) the last instruction of B jumps to the leader of B', or (b) B' textually follows B and B does not end in an unconditional jump.
  • Useful for register allocation, instruction selection, machine-independent optimisation.

§ 4.1

Code Optimization — Introduction & Classification

Code optimization is a program transformation that improves code by making it consume fewer resources (CPU, memory, energy) or run faster — i.e., the replacement of a sequence of instructions by a faster sequence that does the same thing.

Criteria for a Good Optimization

  • Must be semantics-preserving.
  • Must speed up programs on average.
  • Must be worth the effort — no dramatic slow-down of the compilation process.

Techniques Used

  • Control-flow analysis — identifies loops in the CFG (loops are prime candidates for improvement).
  • Data-flow analysis — collects information about variable use across the program.
  • Always remember: the biggest performance wins usually come from improving the source algorithm, not the compiler.

§ 4.2

Machine-Independent Optimizations

Constant Folding

Evaluate constant expressions at compile time instead of run time. Includes algebraic identities like 0·x = 0, 1·x = x, and string concatenations like "dog"+"house" = "doghouse".

i = 30 * 20 * 10        →   i = 6000
x = 5 + 4 / 2           →   x = 7

Constant Propagation

Substitute the value of a known constant variable into its uses. Often combined with constant folding and dead-code elimination.

int a = 10; int b = 45 - a/2; return b * (200/a + 2);
   → int a = 10; int b = 40;     return 40 * 22;
   → return 880;

Common Subexpression Elimination (CSE)

Replace identical re-computations of an expression with a single temporary, when none of the operands change between computations. Relies on data-flow analysis.

a = b * c + g ;             t1 = b * c ;
d = b * c * e ;             a = t1 + g ;
                            d = t1 * e ;

Copy Propagation

After a copy u = v, replace later uses of u by v wherever possible — typically a clean-up pass after CSE.

y = x ; z = 3 + y ;     →   z = 3 + x ;

Dead-Code Elimination (DCE)

Remove instructions whose results are never used, and any code that is unreachable.

Strength Reduction

Replace an expensive operation with a cheaper, equivalent one.

ExpensiveCheaper
x ^ 2x * x
x * 2x + x  or  x << 1
x / 2x * 0.5  or  x >> 1

Packing Temporaries

Replace several distinct temporaries by a single one when their lifetimes do not overlap (live ranges disjoint).

t1 = a + a ;          t1 = a + a ;
t2 = t1 + b ;     →   t1 = t1 + b ;
c  = t2 * t2 ;        c  = t1 * t1 ;

§ 4.3

Loop Optimizations

Most execution time is spent in loops, so loops are the prime targets for optimisation. Four major techniques:

1. Loop-Invariant Detection & Code Motion

A loop invariant is a computation whose value doesn't change during the loop. Move it before the loop (this is code motion).

while (i <= limit - 2)             t = limit - 2 ;
   ...                       →     while (i <= t)
                                      ...

If the loop runs n times, n+1 evaluations of limit-2 become 1.

2. Loop Unrolling (Loop Unwinding)

Replicate the body to reduce iteration count and the number of branch / test instructions. Also exposes more instruction-level parallelism for the back-end.

for (i=0; i<100; i++) g();    →    for (i=0; i<100; i+=2) { g(); g(); }

Caveat: excessive unrolling bloats code and hurts the instruction cache.

3. Induction-Variable Detection & Strength Reduction in Loops

An induction variable grows by a fixed amount per iteration. Replace the multiplication by addition.

L: i = i + 1                Initialize: t1 = 4 * i
   t1 = 4 * i                L: t1 = t1 + 4         // strength reduction
   t2 = a[t1]                   t2 = a[t1]
   if t2 < v goto L             if t2 < v goto L

4. Combined Example

for (i=0; i<10; i+=2) {           x = -6 ;
    x = i * 3 ;            →      for (i=0; i<10; i+=2) {
    a[i] = y - x ;                    x = x + 6 ;        // strength reduction
}                                      a[i] = y - x ;
                                  }

§ 4.5

Local Optimization Using a DAG

Algorithm

  1. Build a DAG for the basic block: every interior node is an operator, leaves are initial values; identical sub-expressions share a node.
  2. Each node is labelled with the list of names whose value it currently holds. The latest name assigned is the "primary" label.
  3. Walk the DAG and re-emit code, skipping nodes whose value is dead on exit and reusing nodes for repeated sub-expressions.

Global optimisation looks across blocks. Typical steps:

  1. Constant-propagate & constant-fold.
  2. Identify and remove unreachable blocks (DCE).
  3. Detect loop invariants and hoist them.
  4. Apply CSE across blocks where data-flow analysis confirms operands haven't changed.
  5. Reduce strength of induction-variable multiplications.
  6. Final clean-up: copy propagation + DCE.

§ 4.7

Next-Use Information & Algorithm

Next-use information tells, for each use of a variable in a basic block, which following statement (if any) will use the same value next. It is needed for register allocation: once a value has no further use, its register can be freed and reassigned.

Assumptions

  • All named variables are live on exit — they will be stored back to memory.
  • All temporaries are dead on exit.

Algorithm (Two-Pass, per Basic Block)

  1. Scan forward, find the end of the block. For every variable X used in the block create symbol-table fields X.live = FALSE and X.next_use = NONE.
  2. Scan backward; for each tuple (i): x = y op z:
    • Copy x, y, z's current symbol-table live / next-use fields into the tuple's instruction-info area. (This is what we report for that statement.)
    • Then update the symbol table: set x.live = FALSE, x.next_use = NONE (x is being defined here, killing earlier values), and set y.live = TRUE, y.next_use = i; same for z.

For each instruction, after processing, the table reports the live/next-use state for the operands at the point just before that instruction executes.

§ 4.8

Live-Variable Analysis (Data-Flow)

A variable x is live at a program point p if there exists a path starting at p along which the value of x is used before being redefined. Otherwise x is dead at p. Liveness is associated with edges of the CFG, not nodes.

Applications

  • Register allocation — a register can be reused for another variable as soon as its current variable is dead.
  • Dead-code elimination — if x is not live after assignment x = …, the assignment is redundant.

Data-Flow Values per Block

  • use[B]: variables that are read in B before being defined.
  • def[B]: variables that are defined in B before being read.
  • IN[B]: live on entry to B.
  • OUT[B]: live on exit from B.

Backward Data-Flow Equations

OUT[B] = ⋃S ∈ successors(B) IN[S]
IN[B] = use[B] ∪ ( OUT[B] − def[B] )

Algorithm

  1. Compute use[B] and def[B] for every block.
  2. Initialise IN[B] = ∅ and OUT[B] = ∅ for every block.
  3. Iterate the equations (typically in reverse-postorder of the CFG) until nothing changes — the fixed point.

§ 4.9

Code Generation — Issues & Three Primary Tasks

The Code Generator is the final phase. Input: the IR from the front-end + symbol table. Output: a semantically equivalent target program (absolute, relocatable, or assembly). The most important criterion is correctness; quality (speed, size, energy) is the next priority.

Why Code Generation Is Hard

  • Producing an optimal target program for a given source program is undecidable.
  • Sub-problems like register allocation and instruction ordering are NP-complete.

Three Primary Tasks

  1. Instruction selection — choose target machine instructions to implement IR statements.
  2. Register allocation & assignment — decide which values live in which registers.
  3. Instruction ordering — schedule the instructions for execution.

Five Design Issues

  1. Input to the code generator — IR (TAC, bytecode, syntax tree, DAG) plus symbol table; assumes the front-end has done lexical, syntactic and type checking and inserted any coercions.
  2. Target program — absolute, relocatable, or assembly; choice depends on flexibility vs immediacy.
  3. Instruction selection complexity depends on:
    • (a) Level of the IR — high-level IRs need template expansion; low-level IRs map almost 1-to-1.
    • (b) Nature of the ISA — uniformity of data-types and completeness of the instruction set.
    • (c) Desired code quality — naive selection produces redundant loads/stores.
  4. Register allocation — register-pair constraints (some ops use even/odd register pairs for multiplication and division), variable register-usage conventions, etc. Optimal allocation is NP-complete.
  5. Evaluation order — order of evaluation affects register pressure; picking a best order is also NP-complete in general.

Code Generation for Procedures (Static Allocation)

With static allocation, the activation record of every procedure lives at a compiler-determined memory address. A procedure call is a branch + a memorised return address; a return is a branch through that saved address.

Memory Layout (Slide Convention)

  • CPU registers occupy 0 bytes in the encoded instruction.
  • Opcodes, immediate constants, memory addresses, memory operands each occupy 4 bytes in the encoded instruction.
  • The first slot of an activation record stores the return address.

caller :  ST <callee_AR>, <return_addr>        // 8 bytes

callee :  ...

§ 4.12

Code-Generator Algorithm — Register & Address Descriptors

The algorithm processes one TAC statement at a time and uses two tables:
  • Register descriptor — for each register, which variable(s) are currently held in it. Initially empty.
  • Address descriptor — for each variable, the set of locations (register, memory, stack) that currently hold its value.

getReg(I) — Choose a Register

  1. If a register is empty, return it.
  2. Else if some occupied register R holds value y, R holds only y, and y has no next use, return R (free without spill).
  3. Else find an occupied register R; spill by emitting ST M, R for each address M in y's address descriptor, then return R.
  4. Else (no register can be obtained) operate directly on memory.

Per-Statement Bookkeeping

For instruction x = y op z:

  1. Use getReg to obtain Rx, Ry, Rz.
  2. Emit LDs for y (and z) if not already in their registers.
  3. Emit OP Rx, Ry, Rz.
  4. Update register descriptor: Rx now holds x and only x. Update address descriptor: x's location is Rx; remove x from any other register's descriptor.
  5. If y or z has no next use and is not live on exit, mark its register as free.

At block exit, emit ST M, R for every variable that is live on exit and held only in register R.

Spilling

When all registers hold useful values and a new register is needed, the system selects a register whose value is needed furthest in the future, stores it back to memory, and reuses the register. The stored value is reloaded later.

§ 4.14

Activation Trees & Activation Records

Activation, Activation Tree

  • An activation = one execution of a procedure.
  • An activation tree represents all the activations during a particular run; the root is main's activation.
  • Properties: each node = one activation; node a is the parent of b iff control flowed from a to b; nodes ordered left-to-right by lifetime; one child must finish before the next starts.
  • The tree depends on input; the static analogue is the call graph.

Stack–Tree Correspondence

  • Sequence of procedure calls ⇔ pre-order traversal.
  • Sequence of returns ⇔ post-order traversal.
  • Flow of control as a whole ⇔ depth-first traversal.
  • While control is at node N, the live (open) activations are exactly N and its ancestors; their activation records are exactly the ones on the run-time stack at this moment.

Activation Record (Stack Frame) — General Layout

+-----------------------+ | Actual parameters | +-----------------------+ | Returned value | +-----------------------+ | Control link | → caller's AR +-----------------------+ | Access link | → enclosing scope's AR (for nested procs) +-----------------------+ | Saved machine status | PC, registers +-----------------------+ | Local data | +-----------------------+ | Temporaries | +-----------------------+

Calling-Sequence Layout Principles

  1. Values communicated between caller and callee at the beginning of the callee's AR — close to the caller's AR.
  2. Fixed-length items (control link, access link, machine status) in the middle so the same code can save/restore them.
  3. Items whose size isn't known until run-time at the end of the AR.
  4. The top_sp register points to the end of the fixed-length region — variable-length data lives "above" top_sp.
Problem 1 — Activation tree for the recursive Fibonacci Fib(3)
int main() { Fib(3); }
int Fib(int n) {
    if (n <= 1) return n;
    return Fib(n-1) + Fib(n-2);
}
Activation tree
main | Fib(3) / \ Fib(2) Fib(1) / \ Fib(1) Fib(0)

Pre-order: main → F(3) → F(2) → F(1) → F(0) → F(1). Post-order = order of returns. Depth-first = order of execution. Maximum stack depth = 3 (main, F(3), F(2), F(1)).

§ 4.15

Calling / Return Sequences · Variable-Length Data

Calling Sequence — Caller / Callee Division

  1. Caller evaluates the actual parameters and stores them at the start of the callee's AR.
  2. Caller stores the return address into the callee's AR.
  3. Caller stores the old value of top_sp into the callee's control link field, then increments top_sp past the caller's locals/temporaries and the callee's parameters/status fields.
  4. Callee saves register values and other status info.
  5. Callee initialises its local data and begins execution.

Return Sequence

  1. Callee places the return value next to the parameters (close to the caller).
  2. Callee restores top_sp and registers from the saved machine-status area, then branches to the return address.
  3. Caller knows where the return value sits relative to its top_sp and may use it.

Variable-Length Data on the Stack

For arrays whose length depends on a parameter (e.g., VLAs in C), the compiler stores only a pointer in the AR; the actual array is laid out above the fixed part of the AR. Access uses two pointers:

  • top_sp — end of the current AR's fixed-length region; gives O(1) access to locals via fixed offsets.
  • top — actual top of stack; the next AR begins here.

On return, top_sp is restored from the saved control link; top = top_sp − (size of fixed fields).

Problem 1 — Compute return-address offset for a recursive call

Suppose int fact(n) contains exactly two action blocks of 20 bytes each before the recursive call fact(n-1) at offset 40 in fact's code, then a 20-byte action after the call. The call+save sequence is ST AR, ret_addr ; BR fact — 8 bytes each.

Layout
offset 0:  ACTION                  (20)
       20: ACTION                  (20)
       40: ST <AR>, 56              (8) — save return addr 56
       48: BR <fact_entry>          (8)
       56: ACTION                  (20) — return point
       76: BR *<AR>                  (4)

The literal "56" is the address of the instruction just after the BR — that's the return point.

Problem 2 — Why VLAs need top vs top_sp

Consider void p(int n) { int a[n]; q(); }.

Discussion

p's AR contains just a pointer to a. The actual array sits above p's fixed fields, between p's AR and q's AR. top_sp points to end of p's fixed area — local references like a, n, return-value are at constant negative offsets from top_sp regardless of n. top points to where q's AR will start — it equals top_sp + sizeof(local fixed data) + n × 4. When q is called, top_sp is moved to the new fixed-area end; on q's return, top_sp is reloaded from q's control link, and top is recomputed from the fixed-area size of q.

§ 4.16

Access to Non-Local Data — Access Links & Displays

Nesting Depth

  • Procedures not nested within others have nesting depth 1. (All C functions have depth 1.)
  • If procedure p is defined immediately inside a procedure at depth i, then p has depth i + 1.

Why Non-Local Access Is Hard

If a procedure p at depth np uses a name x declared in an enclosing procedure q at depth nq < np, then which activation of q is the right one is a run-time decision (q or p might be recursive). The compiler needs additional info: access links or a display.

Access Links

Each AR has an access link: a pointer to the most recent AR of the immediately-enclosing procedure. Following access links forms a chain through ARs at progressively lower nesting depths — the dynamic embodiment of the static scope chain.

Setting the Access Link When q Calls p

CaseAction
nq < np (p nested immediately in q)p.access_link = pointer to current AR of q.
nq ≥ npBoth q and p are nested inside some common r at depth np − 1. Follow q's access link nq − np + 1 times to reach r's AR; that becomes p.access_link.
nq = np (sibling)p.access_link = q.access_link.

Procedures as parameters. The caller, which knows the static context of the procedure being passed, stores the correct access link alongside the procedure pointer; the callee uses both as a closure when it eventually invokes the procedure parameter.

Displays — Faster Lookup

A display is an array d indexed by nesting depth: d[i] always points to the highest AR currently on the stack for any procedure of depth i. Looking up a non-local at depth i is O(1): one indirection through d[i].

Maintenance: when a procedure of depth i is entered, save the previous d[i] in the new AR and overwrite d[i] with a pointer to the new AR. On return, restore d[i].

Trade-Off

Access LinksDisplays
Lookup costVariable — proportional to depth difference; long chains are costly.O(1) constant.
Maintenance costVariable — recomputed at every call.Constant per call/return (just save and restore one pointer).
Problem 1 — Compute access-link chain for a Pascal-style program

Procedures: sort (depth 1) contains readarray, exchange, quicksort (all depth 2). quicksort contains partition (depth 3). At one moment the run-time stack is: sort, quicksort(1,9), exchange. Show the access links.

Solution
  • sort.access_link = ⊥ (depth 1, nothing encloses it).
  • quicksort.access_link = pointer to sort's AR.
  • exchange.access_link = pointer to sort's AR (sibling rule: same depth as quicksort, so they share the access link).

If exchange now references a variable declared in sort, the compiler emits load access_link ; load offset_of_x — exactly one indirection.

Problem 2 — Display rebuild on partition's call

Just before partition is called from quicksort: stack contains sort, quicksort. Display: d[1] = sort, d[2] = quicksort.

Solution
  1. partition has depth 3. Save the old d[3] (which is ⊥) into partition's AR.
  2. Push partition's AR; set d[3] = pointer to partition's AR.
  3. Inside partition, accessing a variable of sort is a single load through d[1]; accessing a variable of quicksort is a single load through d[2].
  4. On return, restore d[3] from the saved field in partition's AR (back to ⊥) and pop the AR.
§ 4.17

Register Allocation by Graph Colouring

Idea. Build an interference graph where each node is a variable and each edge connects variables that are simultaneously live. Allocating k registers is equivalent to k-colouring the graph — adjacent nodes must get different colours. Determining if a graph is k-colourable is NP-hard in general, but greedy heuristics (Kempe's, Chaitin's) work well in practice.

Pipeline

  1. Run liveness analysis on the CFG to obtain a live range for every variable.
  2. Build the interference graph: edge (x, y) iff x and y are alive at the same program point.
  3. Try to colour with k = number of available registers. If any node ends up uncolourable, spill a candidate (insert loads/stores), recompute, and retry.

Kempe-style Greedy

  • Find a node with degree < k. Push it on a stack and remove it from the graph (its neighbours can keep it from getting any of their colours).
  • Repeat until the graph is empty (success) or no such node exists (potential spill — pick a node, mark for spill, remove it, continue).
  • Pop the stack and assign each node a colour different from its already-coloured neighbours.
Problem 1 — Build interference graph & colour with 2 registers

Live ranges:
a: [1, 5]   b: [2, 4]   c: [3, 6]   d: [5, 7]

Solution

Pairs that overlap: a-b (2-4), a-c (3-5), b-c (3-4), c-d (5-6). a-d touch only at point 5 (a's last, d's first) — typically not interfering. So edges: a–b, a–c, b–c, c–d.

a───b \ / c───d

Triangle a-b-c — needs at least 3 colours. With only k=2 registers, the graph is not 2-colourable; one of {a, b, c} must be spilled.

Spill b (smallest live range) ⇒ remaining graph a-c-d is a path → 2-colourable: a=R1, c=R2, d=R1. b is spilled to memory; loaded into a free register only when needed.

Problem 2 — Re-do the next-use block of §4.7 with 2 registers
(1) x = y + z
(2) z = x * 5
(3) y = z - 7
(4) x = z + y
Interference graph

From §4.7's table:
At (1) y, z live; at (2) x, z live; at (3) z, y live; at (4) z, y live.

Pairs: (y, z) at (1) & (3); (x, z) at (2). Edges: y-z, x-z. No edge x-y — they never overlap.

x───z───y

This is a path; 2-colourable. x = R1, y = R1, z = R2 works — x and y share R1 because their live ranges are disjoint. No spill, just 2 registers needed.

Quick-Reference Cheat Sheet

SDD / SDT One-Liners

  • S-attributed: only synthesized → bottom-up evaluation, postfix SDT.
  • L-attributed: synthesized + inherited that depend on left-siblings or parent only → pre-order traversal.
  • Convert SDD → SDT: inherited rules go before the symbol; synthesized rules go at the end.
  • Implement L-attributed on LR via marker non-terminals M → ε.

Intermediate Code Generation

  • Forms: syntax tree, DAG, TAC (quad/trip/indir), SSA, bytecode.
  • Address calc: 1-D A + W·(i−LB); 2-D row-major A + W·[N·(i−Lr)+(j−Lc)].
  • Quadruples: easy reorder, but temporaries clutter symbol table.
    Triples: compact, but code-immobile.
    Indirect triples: best of both (extra pointer list).
  • SSA: φ-functions at every CFG join with multiple defs of the same variable.

Optimization Toolbox

  • Block-local: strength reduction → constant propagation/folding → CSE → copy propagation → DCE.
  • Loop: invariant detection & code motion · induction-variable strength reduction · unrolling.
  • Global: data-flow analysis on the CFG; eliminate unreachable blocks.

Code Generation

  • Three primary tasks: instruction selection, register allocation, instruction ordering.

  • Slide byte-counting convention: opcode/imm/addr/operand each 4 bytes; registers 0 bytes.

Register Allocator (Descriptors)

  • Register descriptor: which variable(s) live in each register.
  • Address descriptor: where each variable's value can be found (register set + memory).
  • getReg() — try empty register → free a register holding a dead-only variable → spill the furthest-future register.
  • At block exit, ST every live-on-exit variable held only in a register.

Run-Time Environments

  • Memory layout: Code | Static | Heap → ↑ ↓ ← Stack (downward-growing).
  • Activation record fields: actual params · return value · control link · access link · saved status · locals · temporaries.
  • Activation tree ⇄ run-time stack: pre-order calls, post-order returns, depth-first overall flow.
  • Non-local data: access links (variable cost, simple) or display (constant cost, indexed by nesting depth).
  • Nesting depth: outermost = 1; nested directly inside depth-i procedure = i+1.

Register Allocation by Graph Colouring

  • Nodes = variables, edges = "simultaneously live" interference, colours = registers.
  • k-colouring is NP-hard; Kempe's algorithm and Chaitin's algorithm work in practice.
  • Spill = move a variable out of a register to memory when no colour is available.
