Advanced RAG Overview

RAG (Retrieval-Augmented Generation) combines document retrieval with language generation. The model first retrieves relevant documents from an external knowledge base, then uses them as context to generate a grounded response. This reduces hallucination and allows access to up-to-date or private knowledge without retraining the model.
Fig: RAG PipelineUser Query → Retriever (search knowledge base) → Retrieved Documents → LLM (generate response using docs as context) → Grounded Answer

Limitations of Basic (Naive) RAG

1. Lack of Domain Specificity

Standard RAG models use general-purpose retrievers that are not optimized for specialized domains. When a medical professional asks "What are the latest treatment options for HER2-positive breast cancer?", a general retriever may fetch generic cancer articles instead of HER2-specific treatments from authoritative sources like PubMed or FDA publications. The generator may then hallucinate plausible but incorrect or outdated information.

Fix: Use curated retrievers restricted to domain-specific databases, fine-tune generators on domain data, implement re-ranking to prioritize authoritative sources.
2. Multi-turn Conversation Failure

Basic RAG systems have no memory mechanism. In a legal contract analysis, if a user first asks "Does this contract allow termination for breach of confidentiality?" and follows up with "What are the specific conditions?", the system may not link the follow-up to the previous context. Without conversational memory, it retrieves generic termination conditions instead of confidentiality-specific clauses.

Fix: Conversational memory (history-aware retrieval), step-wise retrieval that refines based on prior answers, re-ranking tied to conversation thread.
3. Inconsistent Relevance and Quality of Documents

Naive RAG may rank popular blog posts above expert reports from VC firms or financial analysts. For "What are the key risks of investing in AI startups?", it might prioritize trending articles over specialized investment risk reports, retrieve outdated sources, or miss critical financial risks like liquidity concerns and regulatory hurdles.

Fix: Fine-tuned retriever with domain-specific embeddings, source prioritization (filter for credible institutions), query expansion to retrieve more focused documents.
4. Poor Retriever-Generator Synergy

In naive RAG, the retriever and generator operate independently. The retriever fetches top-ranked documents without considering what the generator actually needs for a coherent answer. For a drug discovery query, the retriever might fetch articles with only generic AI mentions while the generator needs specific breakthrough details like AlphaFold's protein structure prediction. The result is vague, shallow answers even though relevant papers existed in the knowledge base.

Fix: Retriever-aware generation (re-rank and summarize before passing to LLM), query reformulation by the generator, feedback loops where the generator guides the retriever to fill gaps.
5. Inefficient Handling of Large-Scale Data

Naive RAG suffers from slow response times when knowledge bases grow large. Keyword matching or basic semantic search may miss documents phrased differently from the query. The system is not adaptable to changing contexts without manual intervention.

The five limitations of naive RAG: (1) Lack of domain specificity, (2) Multi-turn conversation failure, (3) Inconsistent document quality, (4) Poor retriever-generator synergy, (5) Inefficient scaling. Know an example for each.

Re-Ranking with Cross-Encoders

Initial retrieval (by BM25 or dense methods) returns a large set of documents that vary in relevance. Re-ranking reorders these documents so the most relevant ones come first before passing to the generator.

Bi-Encoders (Fast, Less Precise)

Independently encode query and document into separate vectors. Compare using cosine similarity or dot product. Process thousands of documents quickly, but lose context because they don't consider query-document interaction directly. Used in: Sentence Transformers, FAISS, DPR.

Bi-Encoder FlowQuery → Encoder → Vector_Q
Document → Encoder → Vector_D
Score = cosine(Vector_Q, Vector_D)
Cross-Encoders (Slow, More Accurate)

Process query AND document together through a single Transformer (like BERT). The model sees the full interaction between query and document, producing a direct relevance score. Much more accurate but computationally expensive. Used for re-ranking after initial retrieval.

Cross-Encoder Flow[Query + Document] → Transformer → Relevance Score (e.g., 0.92)

Query Expansion

Enriches the user's query with additional terms to increase retrieval coverage.

  • Synonym Expansion: "Treat a cold" → cold remedy, flu treatment, cure for cold.
  • Embedding-Based Expansion: BERT finds semantically related words. "Bakery" → pastry shop, bread store.
  • LLM Paraphrasing: LLM generates alternative queries. "Best laptop for coding" → top programming laptops, laptops for developers, high performance coding laptops.

Advanced Filtering & Context Distillation

Metadata-based filtering: Filter documents by date, author, domain, or document type. In legal/medical contexts, ensures only the most recent or authoritative sources are used.

Content-based filtering: Exclude documents below a semantic similarity threshold or missing key phrases related to the query.

Context distillation: Summarize or condense retrieved documents to focus the LLM on the most important information. Especially useful when retrieved documents contain too much irrelevant content or when queries involve complex multi-step reasoning.

Addressing Hallucination in Advanced RAG

  • Grounding on retrieved documents: Condition the model to only generate based on actual document content, not pre-trained knowledge.
  • Context conditioning: Filter irrelevant parts of retrieved documents before passing to the model.
  • Feedback loops: Check generated outputs against retrieved documents for accuracy before returning to the user.

Evaluating Advanced RAG

  • Accuracy: Correctness of retrieved and generated responses.
  • Relevance: How well results answer the query. Metrics: MRR (Mean Reciprocal Rank), Precision@K.
  • MRR: Calculates the reciprocal rank of the first correct result and averages across queries. If the correct answer is at position 3, the reciprocal rank is 1/3.
  • Latency: Response time, critical for real-time systems.
  • Coverage: Ability to handle diverse and domain-specific queries.

BM25, Hybrid Retrieval & Multi-Hop (with Numericals)

BM25 (Best Matching 25)

BM25 is a probabilistic ranking function for information retrieval. It scores documents based on three factors: how often query terms appear in the document (TF), how rare those terms are across the entire corpus (IDF), and a normalization for document length so longer documents don't get unfair advantage.
BM25 Formula
BM25(D, Q) = Sigma over all query terms t of: IDF(t) x [ f(t,D) x (k1 + 1) ] / [ f(t,D) + k1 x (1 - b + b x |D|/avgD) ]

Where each variable means:

  • f(t, D) = frequency of term t in document D (how many times the word appears)
  • |D| = length of document D (total number of words)
  • avgD = average document length across the entire corpus
  • k1 = term saturation parameter (default 1.5). Controls how much additional occurrences of a term increase the score. High k1 = term frequency matters more linearly. Low k1 = diminishing returns faster.
  • b = document length normalization parameter (default 0.75). If b=1, full length normalization. If b=0, no length normalization.
IDF Formula
IDF(t) = log10( (N - nt + 0.5) / (nt + 0.5) ) Where: N = total number of documents in the corpus nt = number of documents containing term t

IDF captures how rare a term is. A term appearing in most documents has low IDF (less informative). A term appearing in very few documents has high IDF (more discriminating).

Length Normalization Factor
LNF = 1 - b + b x (|D| / avgD)

When |D| = avgD, LNF = 1 (no effect). When |D| > avgD, LNF > 1 (penalizes long documents). When |D| < avgD, LNF < 1 (boosts short documents).

Hybrid Retrieval: BM25 + Dense Retrieval

Hybrid Retrieval combines the precision of keyword-based BM25 with the semantic understanding of dense vector retrieval, using a weighting factor to balance both.
Hybrid Score = α x BM25_score + (1 - α) x Cosine_similarity Where α typically = 0.5 (equal weight to both)
AnalogyThink of FAISS (dense retrieval) as a magic compass that senses the general direction of treasure (semantic similarity). Elasticsearch/BM25 is like a detailed treasure map with exact locations (keyword matches). Hybrid search uses both together for the best results.

Multi-Hop Retrieval

Multi-Hop Retrieval retrieves information iteratively across multiple documents to answer complex queries requiring multiple reasoning steps. Instead of getting all info in one shot, the system connects facts from different sources.
Techniques for Multi-Hop
  • Graph-Based Retrieval: Uses knowledge graphs to connect related entities. Instagram → acquired by → Facebook → acquisition year → 2012.
  • Iterative Retrieval: Step-by-step: retrieve doc, extract new info, use it to retrieve another doc, continue until final answer. Example: "Capital of the country where 2008 Olympics were held?" → Step 1: Olympics = China → Step 2: Capital of China = Beijing.

Generation Enhancement Techniques

Even with good retrieval, the generation step can fail due to noisy/irrelevant documents, context overload, hallucinated reasoning, or poor multi-document synthesis. Advanced RAG introduces several techniques to improve generation quality.

1. Chain-of-Thought (CoT) Prompting
Guide the model to reason step-by-step before producing the final answer.

Instead of jumping to an answer, the LLM generates intermediate reasoning steps. Example: Query "Why did company revenue increase?" → (1) identify revenue data, (2) analyze trends, (3) explain causes. This improves accuracy on complex queries and reduces hallucination by making reasoning explicit.

2. Self-RAG (Self-Reflection Mechanism)
The LLM evaluates its own output using a critique module with special reflection tokens.

Process: (1) LLM generates an answer, (2) Critique module checks if information is supported by evidence, relevant to the query, and useful, (3) If the answer fails any check, the model revises it. This significantly reduces hallucinations and increases factual grounding.

3. Extract-Then-Generate (Ext2Gen)
Separate extraction from generation: extract key sentences first, filter irrelevant content, then generate.

Instead of passing entire retrieved documents to the LLM, extract only the relevant passages. This reduces noise in context and makes generation more robust to retrieval errors.

4. Iterative RAG (ITER-RETGEN)
Multiple cycles: Query → Retrieve → Generate → Refine → Retrieve Again

The system refines information progressively, filling knowledge gaps and improving reasoning for complex queries. Especially useful for multi-hop or multi-step questions.

5. FLARE (Flexible Retrieval Augmented Response Generation)
The LLM requests additional information during generation when it detects knowledge gaps.

While generating an answer, if the model encounters uncertainty, it triggers additional retrieval dynamically rather than producing an incomplete answer.

6. NLI Verification (Natural Language Inference)
An NLI model verifies generated answers against retrieved evidence.

After generation, an NLI model classifies whether the response is: (a) supported by evidence, (b) contradicts the evidence, or (c) is unrelated. This improves factual consistency.

7. KARE-RAG (Knowledge-Aware Refinement)
Structured knowledge representation + preference optimization + contrastive training.

Enhances how models process retrieved knowledge through structured representation and contrastive training to maintain semantic consistency.

Other Techniques

  • Prompt compression: Remove irrelevant context before generation to reduce noise.
  • Context fusion: Combine multiple retrieved documents into coherent input.
  • LLM fine-tuning: Train models to ignore noisy retrieval results.
  • Feedback loops: Incorporate human or system feedback to improve future generation.

Chunking Techniques

Chunking is breaking large documents into smaller, manageable segments to improve retrieval accuracy, context handling, avoid token limit issues, and enable efficient indexing in vector databases.

Why Chunking Matters

  • Improved Retrieval Accuracy: Retrieve only the most relevant information instead of searching whole documents.
  • Better Context Handling: Smaller, well-structured chunks improve LLM processing.
  • Token Limit Management: Prevents exceeding LLM context windows.
  • Efficient Storage: Optimized search and retrieval from vector databases.

Handling Multi-Turn Conversations

  • Conversation history tracking: Store key queries and responses for context.
  • Context windowing: Dynamically focus on recent or critical exchanges.
  • Retrieval-based memory: Selectively retrieve relevant past interactions for long conversations.
SEC 3.5

Generative AI Lifecycle

The GenAI project lifecycle has 4 core principles: Scope (define the problem), Select (choose or train a model), Adapt & Align (prompt engineering, fine-tuning, RLHF, evaluation), Deploy & Optimize (quantization, distillation, RAG, deployment).
GenAI Lifecycle FlowScope → Select Base Model → Adapt (Prompt/Fine-tune/RLHF) → Evaluate → Optimize → Deploy

Step 1: Scope

Clearly define the LLM task. Specific tasks (NER, classification, summarization) need less compute and data. Broad tasks (multi-purpose chatbots, creative generation) need more. Consider: Will the task evolve? How will performance be measured?

Step 2: Select Base Model

Popular choices: GPT-3.5, T5, BERT, LLaMA, Falcon. Use an existing model when you need faster development and cost-efficiency. Train from scratch only for highly specialized domains with insufficient pre-trained coverage. Choose smaller models for cost-sensitive applications.

Step 3: Adapt & Align

  1. Prompt Engineering: Zero-shot, one-shot, few-shot prompting. Fastest, no retraining needed, but limited for domain-specific tasks.
  2. Fine-Tuning: Supervised learning on task-specific labeled data. Full Fine-Tuning updates all weights (expensive but powerful). PEFT (LoRA, Adapters, Prefix-Tuning) updates fewer parameters to save cost.
  3. RLHF (Reinforcement Learning from Human Feedback): Aligns model outputs with human preferences. Used for safety, quality, and usability improvements. Example: OpenAI's GPT models.
  4. Evaluation: Metrics like Perplexity, Accuracy, BLEU, ROUGE, F1. Iterate: Evaluate → Fine-tune → Optimize.

Step 4: Deploy & Optimize

  • Quantization: Reduce model size (e.g., 16-bit → 8-bit precision).
  • Distillation: Use a smaller "student" model that mimics the larger model.
  • RAG: Combine LLMs with knowledge bases for factual accuracy.
  • Deploy on cloud platforms: AWS, GCP, Azure. Use libraries like HuggingFace Inference APIs or ONNX Runtime.

LLM Pretraining

Pretraining is the foundational step where a model learns statistical patterns, structures, and relationships from vast text data using self-supervised learning. The pretrained model does not yet follow instructions -- it is a foundation model requiring fine-tuning. Data sources: web scrapes, books, Wikipedia, Common Crawl, research articles.

Tokenization Methods

  • Byte Pair Encoding (BPE): Iteratively merges most frequent character pairs. "unhappiness" → ["un", "happiness"]. Handles rare words by breaking into subwords.
  • WordPiece: Similar to BPE but uses likelihood-based merging. Used by BERT.
  • SentencePiece: Language-neutral approach that works on raw text.

Evaluation: Perplexity

Perplexity = 2^H(p) Where H(p) = entropy of the model's probability distribution over the next token.

Lower perplexity = better prediction of sequences = better model. It measures how "surprised" the model is by real text.

Computational Challenges

  • Hardware: High-performance GPUs (NVIDIA A100) or TPU Pods.
  • Gradient Accumulation: Train with larger effective batch sizes on limited memory.
  • Mixed Precision Training: Use FP16 to reduce memory.
  • Parallelism: Data parallelism (split data across GPUs) or Model parallelism (split model layers across GPUs).
SEC 3.7

Fine-Tuning Introduction

Fine-tuning is the meticulous adaptation of pretrained LLMs to perform specialized tasks using domain-specific datasets. It builds upon the foundational knowledge captured during pretraining and uses comparatively smaller, targeted datasets.

Why Fine-tune?

  • Adaptability: Customize LLMs for specific tasks (legal doc summarization, medical diagnosis support).
  • Efficiency: Much cheaper than training from scratch -- leverages pretrained knowledge.
  • Performance: Task-specific accuracy, relevance, and coherence far beyond what prompt engineering alone can achieve.
  • Context limitations: Prompt engineering hits context window limits; fine-tuning embeds knowledge in weights.

RAG vs Fine-tuning

RAG

Retrieves external knowledge at inference time. Model weights unchanged. Good for current/domain-specific knowledge that changes often. No training required. Can be inaccurate if retrieval fails.

Fine-tuning

Permanently adjusts model weights. Requires labeled data and compute. Good when you need the model to behave differently (style, format, reasoning). More reliable once trained but static knowledge.

Fine-Tuning Workflow

  1. Select pretrained model aligned with task (BERT for classification, GPT for generation, T5 for seq2seq)
  2. Curate dataset: High-quality labeled data relevant to the domain
  3. Define objectives: Loss functions (cross-entropy), evaluation metrics (accuracy, F1)
  4. Fine-tune: Using HuggingFace Transformers, PyTorch, or TensorFlow
  5. Validate & test: On held-out data to check for overfitting
  6. Deploy & monitor: Continuous refinement based on real-world performance

  • Ethical Risks: Training data may contain biases; model may produce harmful outputs.

Quantization (with Numericals)

Quantization reduces the number of bits used to represent model weights and activations (e.g., FP32 → INT8), reducing memory usage, computational cost, and power consumption during inference.

Why Quantize? GPU RAM Requirements

1 parameter in FP32 = 4 bytes 1 Billion parameters = 4 GB (just for storing weights) Training requires additional 4-20x memory for gradients, optimizer states, activations

A 7B parameter model needs ~28 GB just for weights in FP32. Training needs ~80-120 GB+ of GPU RAM. Quantization to INT8 reduces weight storage to ~7 GB.

Benefits of Quantization

  • Reduced model size: Up to 75% reduction (FP32 → INT8).
  • Faster inference: Lower precision = fewer compute cycles. 2x faster on mobile.
  • Lower power consumption: Critical for edge devices, IoT, mobile.
  • Faster deployment: Smaller models transfer and load faster.

When to Use Which Format

FP32: When accuracy is critical (research, model training). FP16: Mixed precision training on GPUs. BFloat16: TPUs/GPUs for large models where overflow is a concern. INT8: Inference on edge devices or resource-constrained environments.

SEC 3.9

LoRA & QLoRA (with Numericals)

The Problem: Full Fine-Tuning is Expensive

Modern LLMs have billions of parameters. Updating all of them for every new task requires massive GPU memory and time. For a 7B model: full fine-tuning in FP16 needs ~28 GB just for weights, plus optimizer states and gradients pushing it to 80+ GB.

LoRA (Low-Rank Adaptation)

LoRA modifies the model by adding two small trainable matrices (A and B) alongside the original frozen weights. Instead of updating the full weight matrix W (which has d x k parameters), it learns a low-rank update through the product B x A, which has far fewer parameters.
LoRA Architecture Original: Input → [W] → Output
With LoRA: Input → [W (frozen)] + [B × A (trainable)] → Output
Where W is d×k, B is d×r, A is r×k, and r << d, k
LoRA Formula
W' = W + (α/r) × B × A Where: W = original frozen weight matrix (d × k) A = trainable matrix (r × k), initialized randomly B = trainable matrix (d × r), initialized to zero r = rank (much smaller than d and k, typically 4-64) α = scaling factor (controls magnitude of update) α/r = effective scaling that keeps updates stable
Parameter Savings
Full fine-tuning: d × k parameters LoRA: r × (d + k) parameters Example: d = 1024, k = 1024, r = 8 Full: 1024 × 1024 = 1,048,576 params LoRA: 8 × (1024 + 1024) = 16,384 params Savings: 98.4% fewer trainable parameters!
Training & Inference

Training: Only A and B are updated via backpropagation. W stays frozen. This means much less memory for optimizer states and gradients.

Inference: Merge W and ΔW (= α/r × BA) into a single matrix. No extra computation at inference time compared to the original model.

QLoRA (Quantized LoRA)

QLoRA combines 4-bit quantization with LoRA. The base model is stored in 4-bit NF4 format to drastically reduce memory, while LoRA adapters are added in FP16. This allows fine-tuning a 13B model on a 24GB GPU.
Key Techniques in QLoRA
  1. NF4 (NormalFloat4) Quantization: A special 4-bit format designed for neural network weights. It stores values in a distribution-aware way that preserves important weight values better than regular INT4. Reduces accuracy loss.
  2. Double Quantization: Even the quantization scaling factors use memory. Double quantization compresses these scaling constants too, further reducing memory overhead.
  3. Paged Optimizer: When GPU memory is exceeded during training, data is automatically moved between CPU and GPU (like virtual memory). Prevents out-of-memory crashes.

  1. Load pretrained model
  2. Quantize to 4-bit (NF4) using bitsandbytes library

  5. Save adapters separately or merge into base model for deployment
SEC 3.10

Soft Prompts & Prompt Tuning

Soft Prompts are continuous, learnable vectors that are prepended to input embeddings to guide LLM outputs. Unlike text-based hard prompts ("Classify this review as positive or negative"), soft prompts exist in the model's embedding space as trainable real-valued vectors. The base model weights remain completely frozen.

Mathematical Formulation

Traditional input embeddings: [E(x1), E(x2), ..., E(xn)] With soft prompts (P = learnable vectors): Augmented input: [p1, p2, ..., pm, E(x1), E(x2), ..., E(xn)] Optimization: argmin_P L(y | [P; E(x)]) Only P is updated. All model weights stay frozen. Typical prompt length: 20-100 tokens Each vector has same dimensionality as model embeddings (e.g., 768 for BERT)

Evolution of Prompting

  1. Manual Prompting: Hand-crafted text instructions with trial-and-error.
  2. Few-Shot Prompting: Including examples within context window.
  3. Hard Prompt Templates: Structured templates with slots for inputs.
  4. Soft Prompting: Continuous, learnable vectors (current state of the art for parameter efficiency).

Prompt Tuning Variants

Basic Prompt Tuning (Lester et al., 2021)
Trainable vectors at input level only. Fewest parameters (<0.01%).

Prepends learnable vectors only to the input embeddings. Base model completely frozen. Very simple: a few thousand parameters. Performance gap with full fine-tuning shrinks as model size increases (at 10B+ params, nearly matches full fine-tuning).

Prefix Tuning (Li & Liang, 2021)
Trainable parameters at EVERY Transformer layer (keys + values in attention).

Goes beyond basic prompt tuning by adding a "prefix" of trainable parameters to both keys and values in the attention mechanism of each Transformer layer. More parameters (hundreds of thousands) but much better for generation tasks like summarization and dialogue where deeper control of the model is needed.

P-Tuning (Liu et al., 2021)
Uses an MLP to generate prompt embeddings instead of directly optimizing them.

Key innovation: a small Multi-Layer Perceptron transforms initial embeddings into the final prompt vectors. Uses "pseudo-tokens" that don't correspond to actual vocabulary. Can incorporate discrete tokens as "anchors". The MLP provides better optimization properties, making training more stable. Best for classification and inference tasks.

P-Tuning v2 (Liu et al., 2022)
Deep prompts at every layer, directly optimized (no MLP). Works well even for smaller models.

Combines the multi-layer approach of Prefix Tuning with direct optimization. Inserts prompts at every Transformer layer. Achieves results comparable to full fine-tuning across various model scales and tasks. Unlike basic prompt tuning, it works well even for models under 10B parameters.

Prompt Pool
Maintains a collection of different prompts, dynamically selects per input.

Instead of a single fixed prompt, maintains a "pool" of prompts. An attention mechanism or router selects the appropriate prompt(s) for each input. Enables multi-task capability with a single model. Example: in customer service, different prompts selected based on detected user intent or emotion.

Training Tips

  • Learning Rate: Higher than full fine-tuning: 1e-3 to 1e-2 (vs 1e-5 for full FT).
  • Prompt Length: Start with 20-50 tokens, adjust based on task complexity.
  • Initialization: From task-relevant word embeddings works better than random initialization.
  • Regularization: Prompt dropout or L2 regularization to prevent overfitting.
  • Optimizer: Adam with warmup and cosine decay schedule.

Advantages & Limitations

Advantages
  • Extreme parameter efficiency (<0.01%)
  • Single frozen model with multiple task prompts (KB vs GB storage)
  • No risk of catastrophic forgetting
  • Composable for multi-task
  • Scales well with model size
Limitations
  • Underperforms full FT on small models (<10B)
  • Training instability (sensitive to initialization)
  • Prompts don't transfer between different models
  • Hard to interpret (continuous vectors lack semantic meaning)
  • Theoretical foundations still developing

Mixture of Experts (MoE)

Mixture of Experts (MoE) is a neural network architecture that dynamically selects different expert subnetworks for different inputs. Instead of a single large model for all computations, MoE activates only a subset of experts per input, making it more efficient and scalable.

Why MoE?

Model scale is one of the most important factors for quality. Given a fixed compute budget, training a larger model for fewer steps is better than a smaller model for more steps. MoE enables pre-training with far less compute by activating only a fraction of parameters per input. A MoE model achieves the same quality as its dense counterpart much faster during pre-training.

Two Main Elements

  1. Sparse MoE Layers: Replace dense feed-forward network (FFN) layers in the Transformer. Each MoE layer has N experts (e.g., 8), where each expert is typically an FFN. Only the FFN layers are replaced; attention layers and other parameters are shared.
  2. Gate Network (Router): Determines which tokens go to which expert. Uses softmax to assign probabilities. Top-K experts (e.g., K=2) are selected per token. The router is learned during pre-training.
MoE ArchitectureInput Token → Gating Network (Softmax) → Select Top-K Experts
→ Expert 1 (FFN) | Expert 2 (FFN) | ... | Expert N (FFN)
→ Weighted sum of selected expert outputs → Next Layer

Key Components

ExpertsSpecialized subnetworksIndependent FFNs that learn different patterns. Can be hierarchical (MoE within MoE).
Gating NetworkRoute tokens to expertsHard gating (select few) or soft gating (continuous weights). Uses softmax over expert scores.
Expert WeightingCombine expert outputsWeighted sum based on gating scores for final prediction.

Mixtral 8x7B (Key Exam Example)

  • By Mistral AI. 8 expert models, each with 7B parameters.
  • Only 2 experts active per token, making it computationally efficient.
  • Total parameters: ~47B (not 56B) because only FFN layers are per-expert; attention layers and other weights are shared across all experts.
  • Inference FLOPs: like a ~12B dense model (computes 2x7B multiplications but with shared layers).
  • Outperforms LLaMA 2 70B and competes with GPT-4 Turbo in some benchmarks.
Key Mixtral fact: 8x7B = 56B total, but actual unique params = ~47B (shared non-FFN layers). Only 2 experts active per token. Inference cost = ~12B model. ALL parameters must be in RAM even though only 2 experts are active (high memory, fast inference).

Why Sparsity Works

Conditional computation: only run parts of the network per input. Scale model size without proportionally increasing compute. Challenge: batch sizes are effectively reduced because tokens get routed to different experts, leading to uneven utilization.

Challenges

  • Training: Efficient for pretraining but historically struggles with fine-tuning (overfitting).
  • Memory: All parameters must be in RAM (47B for Mixtral 8x7B), even though only 2 experts are used per token.
  • Load Balancing: Some experts may be overused, others underused. Load Balancing Loss helps mitigate this.
  • Complexity: Managing multiple experts and the routing network adds system complexity.
SEC 3.12

Data Resolution & Precision

Data Resolution: The granularity and detail in training data. Higher resolution captures fine patterns but costs more compute. Data Precision: The accuracy of data values, ensuring minimal information loss during representation.

Why Resolution Matters

High Resolution Text

"The restaurant had a cozy atmosphere and the food was absolutely delicious, especially the pasta."

Rich context, clear sentiment, specific details the model can learn from.

Low Resolution Text

"Restaurant good pasta cozy food."

Lost context, ambiguous sentiment, model can't distinguish positive from neutral.

Numerical resolution: Per-second stock data captures micro-trends that per-day data misses. Image resolution: High-res lets models detect facial expressions and fine details; low-res causes blurring.

Impact of Low Precision

  • Text: "Let's eat, Grandma!" (invitation) vs "Let's eat Grandma!" (missing comma changes meaning entirely). Low precision data may lose such distinctions.
  • Numerical: Stock prices rounded from $100.12 to $100 hide fluctuations between $99.50 and $100.49.
  • Consequences: Reduced model performance, increased bias, diminished generalization to real-world data.

Strategies for Improvement

  • Preprocessing: Cleaning (remove noise, duplicates), Normalization (standardize ranges using MinMaxScaler/StandardScaler), Tokenization (BPE/WordPiece for handling rare words).
  • Data Augmentation: Paraphrasing, synonym replacement, back-translation (English → French → English) for text. Gaussian noise or interpolation for numerical data.
  • Precision Handling: Use float32/float64 for critical numerical data. Ensure proper spelling/grammar in text. Curate datasets from trusted sources.

Stable Diffusion

Stable Diffusion is a latent diffusion model that generates AI images from text. Instead of working in the high-dimensional pixel space (a 512x512 RGB image = 786,432 dimensions), it operates in a compressed latent space that is 48x smaller, making it dramatically faster than pixel-space diffusion models like DALL-E or Imagen.

Forward Diffusion

Forward diffusion gradually adds Gaussian noise to a training image over T steps until it becomes pure random noise. Think of it like a drop of ink falling into water: the ink diffuses until you can't tell where it started. Mathematically, at each step t, a small amount of noise is added according to a schedule, pushing the image from the data distribution toward a standard Gaussian distribution.

Forward Diffusion VisualizationClean Image (cat) → [+noise] → Slightly Noisy → [+noise] → More Noisy → ... → [+noise] → Pure Random Noise

Reverse Diffusion

The key insight: if we can reverse the diffusion process, we can start from random noise and recover a clean image. A noise predictor (U-Net) is trained to estimate the noise added at each step. During generation, we start from pure noise, ask the predictor to estimate the noise, subtract it, and repeat. Each step slightly denoises the image.

The reverse diffusion drifts toward the learned data distribution. If trained on cats and dogs, it produces images that are clearly one or the other -- never something in between.

Training Process

  1. Pick a training image (e.g., a photo of a cat)
  2. Generate a random noise image
  3. Corrupt the training image by adding this noise up to a certain number of steps
  4. Teach the noise predictor to estimate how much total noise was added. This is done by tuning its weights against the known correct answer (since we generated the noise, we know exactly what was added).

After training, the noise predictor can estimate noise in any noisy image. To generate: start with random noise, predict noise, subtract it, repeat T times.

Variational Autoencoder (VAE)

The VAE is what makes Stable Diffusion a latent diffusion model. It has two parts:

  • Encoder: Compresses a 512x512x3 pixel image into a 4x64x64 latent representation (48x smaller).
  • Decoder: Reconstructs the image from latent space back to pixel space.

All forward and reverse diffusion happens in latent space, not pixel space. This is why it's so much faster. During training, latent noise is added to the latent representation. During generation, denoising happens in latent space, and the VAE decoder produces the final image.

VAE files in Stable Diffusion specifically fine-tune the decoder to paint finer details (especially eyes and faces). The original VAE loses some fine details during compression; the fine-tuned decoder compensates.

Latent Space PipelineImage (512x512x3) → VAE Encoder → Latent (4x64x64) → [Diffusion happens here] → VAE Decoder → Output Image (512x512x3)

Text Conditioning Pipeline

This is how text prompts steer image generation:

  1. CLIP Tokenizer: Converts each word to a numerical token. Limit: 75 tokens per prompt. Spaces matter: "dream beach" produces different tokens than "dreambeach". Unknown words are split: "dreambeach" → "dream" + "beach".
  2. Embedding: Each token maps to a unique 768-value vector. These embeddings are fixed, learned during CLIP training. The same embedding is used in tools like AUTOMATIC1111.
  3. Text Transformer: Processes embeddings into a form the noise predictor can use. Acts as a universal adapter -- can handle text, class labels, images, depth maps as conditioning inputs.
  4. Cross-Attention in U-Net: The text transformer output is consumed by the noise predictor via cross-attention mechanisms at multiple points throughout the U-Net. This is where the prompt meets the image. Example: "A man with blue eyes" causes "blue" and "eyes" to be paired together, steering specific image regions to render blue eyes.
Cross-attention is the MOST important mechanism in Stable Diffusion. LoRA modifies cross-attention weights to change styles. Hypernetwork hijacks cross-attention to insert styles. The fact that modifying this single module alone can fine-tune the entire model shows its importance.

Text-to-Image: Step by Step

  1. Generate random latent tensor (controlled by seed -- same seed = same starting noise = reproducible results)
  2. U-Net predicts noise from latent tensor + text prompt conditioning (outputs a 4x64x64 noise estimate)
  3. Subtract predicted noise from latent image to get a slightly cleaner latent
  4. Repeat steps 2-3 for N sampling steps (e.g., 20 times). The noise schedule dictates how much noise is subtracted at each step -- larger amounts initially, finer adjustments later.
  5. VAE Decoder converts the final latent image to pixel space -- this is the output image

Image-to-Image

  1. Input image is encoded to latent space by VAE encoder
  2. Noise is added to the latent image. Denoising strength controls how much: 0 = no noise (output identical to input), 1 = maximum noise (output completely regenerated)
  3. U-Net predicts noise conditioned on BOTH text prompt and noisy latent
  4. Subtract noise, repeat for N steps
  5. VAE decoder produces final image

Inpainting is a special case: noise is added only to masked regions, so only those parts are regenerated while the rest stays intact.

Classifier-Free Guidance (CFG)

Classifier Guidance: Uses image labels + a separate classifier to steer diffusion. Higher guidance scale = more unambiguous images matching the label, but less diversity. Requires training a separate classifier.

Classifier-Free Guidance: Achieves the same effect without a separate classifier by putting the conditioning directly into the U-Net (via the text transformer). The CFG scale controls how strongly the text prompt steers the diffusion process:

  • CFG = 0: prompt is completely ignored (unconditioned generation)
  • CFG = 1: standard conditioning
  • CFG = 7-12: typical range for good results
  • CFG very high: oversaturated, artifact-prone images

SEC 4.2

CLIP (Contrastive Language-Image Pre-training)

CLIP (by OpenAI) is a multimodal model that learns to associate images with text using contrastive learning. Trained on 400 million image-text pairs from the internet. It can perform zero-shot image classification using only natural language descriptions.

Contrastive Learning: Core Concept

The model learns by comparing: Positive pairs (an image with its correct caption) should have high similarity. Negative pairs (an image with an unrelated caption) should have low similarity. The objective: maximize similarity for matching pairs, minimize for non-matching.

Contrastive Learning MatrixIn a batch of N image-text pairs, there are N correct pairs (diagonal) and N²-N incorrect pairs (off-diagonal). CLIP maximizes the diagonal and minimizes the off-diagonal similarities.

Architecture

  • Image Encoder: Vision Transformer (ViT) or ResNet. Converts image patches into a sequence, processes through self-attention layers to produce a visual embedding vector.
  • Text Encoder: Transformer (similar to GPT). Tokenizes text, processes through self-attention to produce a text embedding vector.
  • Projection to Shared Space: Both embeddings are projected into a shared multimodal embedding space using linear layers and normalized. This allows direct comparison via cosine similarity.

InfoNCE Loss (Detailed)

L_i = -log( exp(sim(xi, yi) / τ) / Σ_j exp(sim(xi, yj) / τ) ) Where: xi = image embedding for the i-th image yi = text embedding for the i-th caption (correct match) yj = text embeddings for ALL captions in the batch (j = 1 to N) sim(x, y) = cosine similarity = (x · y) / (||x|| × ||y||) τ (tau) = temperature parameter (hyperparameter) exp(·) = exponentiation function log = logarithm (converts multiplication to addition, penalizes small probabilities heavily) N = batch size

What each part does:

  • Numerator exp(sim(xi, yi)/τ): Similarity of the correct image-text pair.
  • Denominator Σ exp(sim(xi, yj)/τ): Sum of similarities of the image with ALL text in the batch (correct + incorrect).
  • Temperature τ: Controls sharpness of the distribution. Smaller τ = more confident, sharper predictions. Larger τ = softer, more uncertain predictions.
  • The loss is minimized when the correct pair has much higher similarity than all incorrect pairs (probability close to 1, log close to 0).
Know every term in InfoNCE: xi = image embedding, yi = correct text embedding, sim = cosine similarity, τ = temperature, exp = exponentiation, N = batch. The numerator is the correct pair; denominator sums all pairs. Loss pushes correct pair similarity up, incorrect down.

How CLIP Works: Zero-Shot Classification

  1. User provides class labels as text templates (e.g., "a photo of a cat", "a photo of a dog", "a photo of a car")
  2. Input image passes through the image encoder to produce an image embedding
  3. Each text label passes through the text encoder to produce text embeddings
  4. Cosine similarity is computed between the image embedding and each text embedding
  5. The class label with the highest similarity score is the predicted class

This is "zero-shot" because CLIP was never specifically trained on these classes -- it generalizes from its broad pre-training on 400M image-text pairs.

Training Process

  1. Batch Sampling: Sample a batch of N image-text pairs.
  2. Feature Extraction: Image and text encoders generate embeddings.
  3. Contrastive Learning: Optimize using InfoNCE loss.
  4. Optimization: AdamW optimizer.
  5. Normalization: Embeddings are L2-normalized for stable cosine similarity computation.

Evaluation Metrics

  • Zero-Shot Classification Accuracy: Classify unseen images using only text descriptions.
  • Recall@k: How often the correct text/image is within the top-k predictions.
  • Mean Average Precision (mAP): How well relevant items are ranked in retrieval tasks.
  • Cosine Similarity: How closely aligned image and text embeddings are.

Applications

Zero-shot image classification, image search and retrieval, cross-modal understanding, guiding generative AI (Stable Diffusion uses CLIP), multimodal reasoning tasks.

SEC 4.3

BLIP (Bootstrapped Language-Image Pre-training)

BLIP (by Salesforce) is a unified vision-language model that can both understand and generate language about images. Key innovation: it handles noisy, uncurated web data through a bootstrapped pretraining strategy that self-corrects bad labels. Trained on 129 million image-text pairs.

CLIP vs BLIP: Key Difference

CLIP only aligns images and text in a shared embedding space (understanding). BLIP can also generate text (captioning, VQA answers) and self-improve its training data quality. BLIP addresses CLIP's limitation of relying on perfectly labeled datasets.

Bootstrapped Pretraining: How BLIP Learns from Noisy Data

Web-scraped data is noisy -- many image-text pairs are mismatched. BLIP handles this through iterative self-improvement:

  1. Initial Learning with Noisy Data: BLIP trains on both correct and incorrect pairs. It makes rough predictions but isn't very accurate yet.
  2. Caption Generation (Self-Improvement): BLIP's captioning module generates its own descriptions for images. For a mismatched pair (cat image labeled "dog playing in park"), it generates a better caption: "A cat sitting on a wooden table." This acts as a pseudo-label.
  3. Contrastive Learning & Filtering: BLIP compares generated captions with originals using contrastive loss. It learns to trust correct pairs and reduce the influence of noisy/misleading data.
  4. Iterative Refinement: After filtering, BLIP updates its model and repeats. Each iteration improves at identifying and correcting mislabeled data.

Architecture (Three Parts)

  • A. Image Encoder (ViT): Extracts visual features. Represents image as a sequence of patches, similar to words in a sentence.
  • B. Text Encoder (BERT-like): Processes text data to extract textual embeddings.
  • C. Multimodal Transformer: Fuses image and text embeddings using cross-attention. Learns the interaction between modalities.

BLIP's architecture integrates three components with shared parameters for efficiency:

  2. Image-grounded Text Encoder: Models vision-language interactions with cross-attention layers. Used for ITM.
  3. Image-grounded Text Decoder: Uses causal self-attention for text generation tasks. Used for ITG.

Benefits of shared parameters: efficiency in pre-training and multi-task learning adaptability.

  1. Input Processing: Image fed into ViT encoder; text tokenized and fed into BERT-like encoder
  2. Feature Extraction: Image and text features extracted as embeddings
  3. Multimodal Fusion: Multimodal transformer combines embeddings using cross-attention
  4. Task-Specific Head: Output passed through appropriate head (caption generation, QA, or retrieval)
  5. Output: For captioning: generates natural language text. For retrieval: ranks images/text by similarity scores.

Evaluation Metrics

  • Captioning: BLEU (how closely generated text matches ground truth), CIDEr (how well caption describes image).
  • VQA: Accuracy (how often predicted answer matches correct answer).
  • Retrieval: Recall@k (how often correct image/text is in top-k results).

SEC 4.4

Multimodal LLMs & Fusion Strategies

Multimodal LLMs are AI systems that process multiple types of input -- text, images, audio, video -- providing more comprehensive understanding than text-only models. A modality is a specific type of data or input.

How Multimodal Models Work

  1. Data Encoding: Each modality converted to numerical representations using specialized encoders (CNNs/ViTs for images, Transformers for text, mel-spectrograms for audio).
  2. Fusion: Encoded data combined using concatenation, attention mechanisms, or cross-modal learning.
  3. Prediction: Model generates output based on fused representation (text generation, classification, retrieval).

Fusion Strategies (Critical for Exams)

1. Early Fusion (Feature-Level)
Combine raw/low-level features BEFORE feeding into the model for joint processing.

Input features (pixel values from images + word embeddings from text) are concatenated or transformed into a shared feature space early on. Best for tasks requiring tight integration: image captioning, video understanding.

Models: CLIP, ImageBind, BLIP, ViLT

2. Late Fusion (Decision-Level)
Process each modality INDEPENDENTLY, then combine outputs at the decision level.

Each modality has its own model/pathway. Outputs (classification scores or embeddings) are merged using weighted averaging or voting. Best when each modality provides independent insights: sentiment analysis, biometric authentication.

Models: AVEC, MMIM, VQA 1.0, HUME

3. Hybrid Fusion (Intermediate)
Integrate at MULTIPLE levels during model processing. Combines aspects of both early and late.

Features are fused at intermediate layers, allowing both low-level and high-level interactions between modalities. Best for complex tasks: video question answering, autonomous driving.

Models: Flamingo, GPT-4V, VisualBERT, MMF

LLAVA Architecture

  • Vision Encoder: ViT or similar, converts images to feature maps.
  • Language Model Backbone: GPT or LLaMA, processes text and integrates image embeddings via cross-attention.
  • Cross-Modality Attention: Aligns text and image embeddings at various layers.
  • Fine-Tuning Module: Adapts for task-specific requirements.

SEC 4.5

Whisper

Whisper (OpenAI, 2022) is a general-purpose ASR (Automatic Speech Recognition) model. It is a generative model using an encoder-decoder Transformer that converts speech to text, supports 90+ languages, can translate, and detects input language -- all within a single model.

Architecture Pipeline

Raw Audio → Resample to 16kHz → Split into ~30s segments → Log-Mel Spectrogram → Encoder (extract acoustic features) → Decoder (generate text token by token) → Text Output

Audio Processing & Encoder

Input audio is converted to a log-Mel spectrogram -- a numerical representation of sound features. The encoder processes this using attention mechanisms to extract acoustic features and speech patterns, producing a compact representation of the audio.

Decoder & Multitask

The decoder generates text sequentially, one token at a time, using the encoded audio representation. Whisper handles multiple tasks through special tokens that guide output:

  • Transcription token → transcribe speech to text in original language
  • Translation token → translate speech to English text
  • Language token → detect and identify the spoken language

No separate models needed -- one model, multiple tasks, controlled by tokens.

Training

Trained on ~680,000 hours of diverse real-world audio (podcasts, interviews, lectures) using weakly supervised learning (data may include noise or imperfect labels). This large, diverse training data is the key reason for Whisper's robustness to noise, accents, and varied conditions.

AI Agent: A system that can make decisions, take actions, and adapt based on results to achieve a goal. Uses an LLM as its reasoning engine. An agent is not just answering -- it is thinking (reasoning), deciding (planning), acting (executing), and learning from results (observing & reacting).

SEC 4.7

Agentic Design Patterns

Design patterns are reusable solutions to common system-design problems. In Agentic AI, they describe proven ways to organize reasoning, planning, tool use, memory, validation, and coordination. Without clear patterns, agentic workflows become unstable, expensive, hard to debug, and inconsistent.

Why patterns are needed: (1) Improve reliability via structure, (2) Reduce hallucination via retrieval/verification/reflection, (3) Improve task performance by separating planning/execution/evaluation, (4) Make systems scalable, (5) Make systems monitorable.

1. Planning Pattern
Agent creates a sequence of steps BEFORE executing. Best for complex, multi-stage, constrained tasks.

Example: "Plan a 3-day budget trip to New York" → (1) Set budget to $400, (2) Find bus/train options, (3) Search hostels under $60/night, (4) Group nearby attractions, (5) Add food estimate, (6) Produce final schedule.

Where to add: In the planner layer, before tool execution or action generation.

2. ReAct Pattern (Reason + Act)
Agent alternates between reasoning and action. Adapts dynamically after each observation.

Example: "Find cheapest flight NYC to Seattle" → Reason: need prices → Act: call flight API → Observe: $280, $310, $355 → Reason: $280 is cheapest but check baggage fees → Act: call details API → Observe: $70 baggage fee → Reason: $310 may be cheaper overall → Act: compare totals → Final: recommend $310 flight.

Where to add: In the main agent loop where reasoning and tool use happen repeatedly.

3. RAG Pattern
Retrieve external information before generating. Reduces hallucination, increases factual grounding.

Example: "What is the leave policy for maternity leave?" → Search company HR docs → Find maternity section → Use only retrieved content → Generate grounded answer.

4. Tool-Use Pattern
Agent calls external tools (search, calculators, code interpreters, APIs, databases).

Example: "25% growth on $84,500 monthly revenue, plot for 6 months" → Use calculator for exact math → Use Python for chart → Return result with graph.

5. Reflection / Self-Critique Pattern
Agent evaluates its own output and improves before returning. Quality over single-pass speed.

Example: Write email draft → Self-critique: tone? justification? proposed deadline? → Improve: "Hi" → "Dear Manager, I am writing to request..."

6. Router Pattern
Direct each request to the correct specialist handler.

Example: "Summarize this chest X-ray report" → Router classifies as medical → Routes to healthcare agent → Specialist answers.

7. Orchestrator-Worker Pattern
One orchestrator assigns sub-tasks to multiple workers, gathers and merges results.

Example: "Analyze customer reviews" → Orchestrator assigns: Worker 1 (sentiment analysis), Worker 2 (complaint themes), Worker 3 (praised features), Worker 4 (executive summary) → Orchestrator combines into final report: "68% positive, top complaints: battery/delivery, most liked: camera/screen".

Know all 7 patterns with examples: Planning (trip planning), ReAct (flight search), RAG (company docs), Tool-Use (calculations), Reflection (email drafting), Router (medical routing), Orchestrator-Worker (review analysis). Know where each is added in the system.

SEC 4.9

LLM Evaluation

Evaluation Dimensions

Factual Accuracy, Relevance, Coherence, Fluency, Safety, Robustness, Fairness, Groundedness.

Precision (Foundation for BLEU)

Precision = Number of correct predicted words / Total predicted words Example: Target: "Adobe makes Photoshop" Predicted: "Adobe makes ColdFusion" Correct words: "Adobe", "makes" (2 out of 3) Precision = 2/3 = 0.667

Problem with basic precision: "Adobe makes makes" would get 3/3 = 100% (repeating "makes" counts twice).

Clipped Precision: Clips each word count to the maximum number of times it appears in the reference. "makes" appears once in reference, so clipped count = 1, not 2. Clipped precision = 2/3.

BLEU Score

BLEU (Bilingual Evaluation Understudy) measures how similar candidate text is to reference text. Values 0-1; ≥0.6 considered high. Used for machine translation and text generation evaluation.
BLEU Computation Steps
  1. Compute clipped precision for n-grams: 1-gram (unigram), 2-gram (bigram), ..., n-gram. Example for 1-gram: count matching unigrams in candidate vs reference, clip by max reference count.
  2. Geometric average of all n-gram precisions: exp((1/N) x Σ log(precision_n)). This penalizes if any n-gram precision is very low.
  3. Brevity Penalty (BP): Penalizes candidates that are too short (prevents gaming by outputting only high-confidence words).
BP = exp(1 - r/c) if c < r BP = 1 if c >= r Where: c = candidate length, r = reference length BLEU = BP × exp( (1/N) × Σ_n log(precision_n) )

BERTScore

Uses BERT embeddings to compute token-level similarity between reference and candidate. Unlike BLEU (exact word matching), BERTScore captures semantic meaning. "Happy" and "joyful" would get high similarity in BERTScore but zero in BLEU.

SEC 4.11

Data behind LLMs

Why Data is Crucial

  • Learning Patterns: LLMs learn grammar, syntax, semantics from data. More diverse data = better generalization. Also learns contextual understanding: "She couldn't bear the weight of the decision" = emotional burden, not physical.
  • Capturing Knowledge: Absorbs factual and conceptual knowledge (history, science, trends). Also picks up common sense reasoning and cultural norms.
  • Data Quality: High-quality data = accurate responses. Training on unreliable sources (biased news, conspiracy sites) taints outputs with misinformation.
  • Data Diversity: Avoids bias and overfitting. Handles multiple languages and dialects. Ensures inclusive representation of all demographics.

Types of Training Data

  • Text Data (core): Books, articles, websites, conversations, code.
  • Multilingual Data: Multiple languages for cross-lingual capabilities.
  • Domain-Specific Data: Healthcare, finance, legal -- for specialized models.
  • Structured Data: Tables, graphs for factual knowledge retrieval.

Data Sources

Public: Common Crawl, Wikipedia, Project Gutenberg. Licensed: Publishers, news outlets. User-Generated: Social media, forums (with privacy considerations). Private/Corporate: For customized enterprise models.

Preprocessing

Cleaning (remove noise, spam, offensive content), Normalization (standardize formats), Tokenization (BPE, WordPiece, SentencePiece), Bias detection and mitigation.

Data Augmentation

Artificially increase diversity: paraphrasing, back-translation, synonym replacement, synthetic text generation. Improves robustness and reduces bias impact.

Data Efficiency Techniques

  • Transfer Learning: Fine-tune pre-trained models on domain data (don't start from scratch).
  • Few-Shot Learning: Train models to perform with minimal labeled data.

Challenges

  • Quality: Ensuring accuracy and reliability is critical.
  • Bias: Training data biases propagate to model outputs.
  • Privacy: Must comply with GDPR, CCPA. Personal data handling requires care.
  • Scarcity: Some domains/languages lack sufficient high-quality data.
  • Scale: Training on trillions of tokens requires petabytes of storage and massive compute.

Future Directions

Multimodal data (images + text) for more robust models, privacy-preserving techniques (federated learning), open and ethical data sharing initiatives, techniques for low-resource languages.
