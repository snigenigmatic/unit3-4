§ 3.1

GRASP — Introduction & Responsibility

GRASP stands for General Responsibility Assignment Software Patterns — a set of guidelines for assigning responsibilities to classes and objects in object-oriented design. First published by Craig Larman with 9 fundamental principles. Often paired with SOLID.

What is a Responsibility?

A responsibility is a contract or obligation that a class, module, or component must fulfil. It can be accomplished by a single object or by a group of objects collaborating via messages.

Two Types of Responsibility
TypeMeaningExamples
DoingDoing something itselfCreating an object, performing a calculation, initiating action in others, controlling/coordinating activities
KnowingKnowing informationKnowing private encapsulated data, knowing related objects, knowing things it can derive or calculate
Responsibility-Driven Design (RDD)
  • Design done in terms of responsibilities, roles, and collaborations.
  • RDD view: abstraction of what objects do.
  • UML view: a contract or obligation of a classifier.
  • Translation of responsibilities into classes/methods depends on the granularity of the responsibility.

The Nine GRASP Principles

#PrincipleCore Question
1CreatorWho creates a new instance of X?
2Information ExpertWho has the info to fulfil the responsibility?
3Low CouplingHow to keep dependencies minimal?
4ControllerWho handles a system event from the UI?
5High CohesionHow to keep each class focused?
6PolymorphismHow to vary behaviour by type?
7IndirectionHow to avoid direct coupling between two elements?
8Protected VariationsHow to shield from variation in other elements?
9Pure FabricationWhat if domain-based assignment gives bad design?
The slides explicitly cover principles 1–6 in detail. The remaining three (Indirection, Protected Variations, Pure Fabrication) are listed in the agenda but not expanded — examiners may still ask for the full list of 9.
§ 3.2

Creator

Problem: Who should create a new instance of some class?
Solution: Assign the responsibility of creating instances of class X to class B if one or more apply.
When B should create X
  • B contains or aggregates X
  • B records X
  • B closely uses X
  • B has the initialising data for X

Principle: "Container" objects create "contained" objects.

Worked Examples

  • POS: a Sale contains SalesLineItem objects → Sale is the Creator.
  • Monopoly: the Board creates Square objects since the Board aggregates all squares.
Java — OrderService as Creator
public class OrderService {
    private final OrderRepository orderRepository;

    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    public void createOrder(Customer customer, List<Product> products) {
        Order order = new Order(customer, products);     // OrderService is Creator
        orderRepository.saveOrder(order);
    }
}
Note: When creation logic itself is complex, prefer the Factory pattern over plain Creator.
§ 3.3

Information Expert

Problem: What is the basic principle for assigning responsibilities?
Solution: Assign a responsibility to the class that has the information needed to fulfil it.

"Objects do things related to the information they have." If the information is spread across several classes, the objects collaborate via messages.

POS — calculating the grand total
ClassResponsibility
SaleKnows sale total
SalesLineItemKnows line-item subtotal
ProductDescriptionKnows product price
Java — Shopping cart total
public class ShoppingCart {
    private List<Item> items = new ArrayList<>();

    public void addItem(Item item) { items.add(item); }

    public double calculateTotalPrice() {
        double totalPrice = 0;
        for (Item item : items) totalPrice += item.getPrice();
        return totalPrice;                // ShoppingCart is the Info Expert
    }
}

public class Item {
    private String name;
    private double price;
    public double getPrice() { return price; }
}

The ShoppingCart is the info expert for the total; Item is the info expert for its own price.

§ 3.4

Low Coupling

Coupling = degree of dependency between classes/modules.
Problem: How to support low dependency, low change impact, and increased reuse?
Solution: Assign responsibility so coupling stays low.
Tight Coupling

High dependency. Changes in one class force changes in another. Happens when a class instantiates another concretely and directly calls its methods/fields.

Loose Coupling

Low dependency. Classes interact through well-defined interfaces. Achieved via interfaces, abstract classes, or dependency injection.

Two elements are coupled if…
  • One has an aggregation/composition with the other, OR
  • One implements/extends the other
Coupling tells you "X depends on Y" if…
  • TypeX has an attribute of TypeY
  • TypeX calls services of a TypeY object
  • TypeX has a method referencing an instance of TypeY
  • TypeX is a subclass of TypeY
  • TypeY is an interface and TypeX implements it
Subclassing implies high coupling. Inheritance is the strongest form of coupling — prefer composition where possible.

Inversion of Control / Dependency Injection

Dependencies are injected into objects by a third party (Spring DI, Java EE CDI) rather than created internally. The key benefit is loose coupling.

Loose-coupling Java example
interface Vehicle { void move(); }

class Car implements Vehicle {
    public void move() { System.out.println("Car is moving"); }
}

class Bike implements Vehicle {
    public void move() { System.out.println("Bike is moving"); }
}

class Traveler {
    private Vehicle v;
    public void setV(Vehicle v) { this.v = v; }
    public void startJourney() { v.move(); }
}

// Main
Traveler traveler = new Traveler();
traveler.setV(new Car());  traveler.startJourney();   // Car injected
traveler.setV(new Bike()); traveler.startJourney();   // Bike injected
POS Example — Payment association
  • Tight: Register creates Payment and sends it to Sale → Register now coupled to Payment.
  • Loose: Register asks Sale to create the Payment → Register no longer needs to know about Payment.
§ 3.5

High Cohesion

Cohesion = how strongly related and focused the responsibilities of an element are.
Problem: How to keep complexity manageable and support low coupling?
Solution: Assign responsibilities so cohesion remains high.
Problems caused by low cohesion

A class that does too many unrelated things is:

  • Hard to understand
  • Hard to reuse
  • Hard to maintain
  • Brittle — easily affected by change
High cohesion and low coupling go hand in hand — a class with a single clear purpose tends to depend on fewer things.
§ 3.6

Controller

Problem: Who should handle a system event? What object receives and coordinates a system operation?
Solution: Assign responsibility to an object that represents either the overall system (Façade Controller) or a single use-case scenario (Use-Case Controller).

Controller deals with delegating requests from the UI layer to domain-layer objects. Helps minimise dependency between GUI components and system operation classes. When a UI event arrives, the Controller decides which domain object first receives the message — it delegates work and coordinates activities, but does not do the work itself.

Two types of Controller
TypeWhat it represents
Façade ControllerThe whole system, device, or subsystem
Use-Case ControllerA single use case; one controller per use case allows state to be maintained (e.g., events must happen in order)
Bloated Controller (anti-pattern): a single class that receives all system events, does the work itself instead of delegating, and hoards state that should live in domain objects.
Java implementation steps
  1. Identify system boundaries (UI, external systems, modules).
  2. Define Controller classes — one per related set of actions.
  3. Separate concerns — coordinate, don't implement business logic.
  4. Handle user inputs via controller methods/event handlers.
  5. Coordinate interactions between components.
  6. Promote loose coupling (interact via interfaces/abstractions).
  7. Handle exceptions gracefully.
Benefits
  • Reusable controller
  • Maintains use-case state
  • Controls sequence of activities
§ 3.7

Polymorphism

Problem: How to handle alternatives based on type? How to swap one server component for another without affecting the client?
Solution: When behaviours vary by type, assign responsibility using polymorphic operations — give the same name to similar/related services on the types where behaviour varies.

Four Types of Polymorphism

TypeAlso calledDescription / Java
Ad-hocOverloadingSame function name acts differently for different types. + adds ints, concatenates strings.
ParametricEarly bindingSame code works with different types. Implemented via Generics.
InclusionSubtyping / RuntimeUse derived classes through a base reference. Dynamic method dispatch — invoked method decided at runtime by the actual object.
CoercionCastingCast object/primitive into another type. Implicit widening; explicit narrowing.
Coercion example
int x = 10;
double y = x;             // implicit widening: int → double

double d = 9.7;
int i = (int) d;          // explicit narrowing cast
Shape example

getArea() varies by shape → assign responsibility to subclasses Circle, Triangle. Sending a message to a Shape reference invokes the right subclass method at runtime.

§ 3.8

Indirection · Protected Variations · Pure Fabrication

These three complete the GRASP nine. The slides list them but don't expand. Quick definitions to keep on hand:

Indirection

To avoid direct coupling between two elements, assign the responsibility of mediating between them to an intermediate object. Example: the Adapter pattern uses an indirection class.

Protected Variations

Identify points of predicted variation/instability and wrap them in a stable interface so that variation on the implementation side does not ripple into callers. (Parent of patterns like Polymorphism, Indirection, Adapter.)

Pure Fabrication

When assigning a responsibility based on the domain model would produce bad cohesion/coupling, assign it instead to a made-up class that does not represent a domain concept. Common example: a TableDataGateway or a persistence helper class.

GRASP MCQ traps from the slides — GRASP stands for General Responsibility Assignment Software Patterns, has nine principles, and Controller's main job is delegating UI requests to domain objects.
§ 3.9

SOLID — Interface Segregation Principle (ISP)

ISP: Make fine-grained interfaces that are client-specific. Clients should not be forced to implement interfaces they do not use.
Key rules
  • Break up fat interfaces with too many methods.
  • Many client-specific interfaces are better than one general-purpose interface.
  • Classes should implement multiple small interfaces as needed.
  • Failure to comply → implementations depend on methods they don't need but must define.

Violation Example

interface IUser {
    boolean login(String name, String password);
    boolean register(String name, String password, String email);
    void logError(String error);
    boolean sendmail(String emailContent);
}

ISP-Compliant Refactor

interface IUser {
    boolean login(String name, String password);
    boolean register(String name, String password, String email);
}
interface ILogger { void logError(String error); }
interface IMail   { boolean sendmail(String emailContent); }

Bookstore Example

Violates ISP — the interface:

public interface BookAction {
    void seeReviews();
    void searchSecondhand();
    void listenSample();
}

Violates ISP — both classes forced to implement everything:

class HardcoverUI implements BookAction {
    public void seeReviews()       { /* ... */ }
    public void searchSecondhand() { /* ... */ }
    public void listenSample()     { /* ... unused */ }
}

class AudiobookUI implements BookAction {
    public void seeReviews()       { /* ... */ }
    public void searchSecondhand() { /* ... unused */ }
    public void listenSample()     { /* ... */ }
}

ISP-compliant:

public interface BookAction { void seeReviews(); }

public interface HardcoverAction extends BookAction { void searchSecondhand(); }
public interface AudioAction     extends BookAction { void listenSample(); }

class HardcoverUI implements HardcoverAction {
    public void seeReviews()       { /* ... */ }
    public void searchSecondhand() { /* ... */ }
}

class AudiobookUI implements AudioAction {
    public void seeReviews()  { /* ... */ }
    public void listenSample(){ /* ... */ }
}

Case Study — Xerox

Robert C. Martin first formulated ISP while consulting for Xerox. Xerox had a new printer system (print, staple, fax, …). Problem: one large Job class held every responsibility. Modifications and deployment became complex. Solution: segregate the Job class into multiple role interfaces based on client need.

ISP MCQ keys — "Clients should not be forced to implement interfaces they do not use" / Implementing classes should implement only necessary methods / ISP suggests fine-grained, client-specific interfaces / Not following ISP → dependencies on unused methods.
§ 3.10

SOLID — Dependency Inversion Principle (DIP)

DIP: Program to the interface, not the implementation.
The two laws of DIP
  • High-level modules should not depend on low-level modules. Both should depend on abstractions.
  • Abstractions should not depend on details. Details should depend on abstractions.

Goal: avoid tightly coupled code. Decouple high- and low-level classes — their interaction is an abstract interaction. Per Robert C. Martin, DIP is a specific combination of the Open-Closed and Liskov Substitution principles.

Violation — Bookstore Shelf

class Book {
    void seeReviews() { /* ... */ }
    void readSample() { /* ... */ }
}
class Shelf {
    Book book;
    void addBook(Book book)     { /* ... */ }
    void customizeShelf()       { /* ... */ }
}

Adding DVDs forces the Shelf class to be modified → violates Open-Closed too.

DIP-Compliant Fix

public interface Product {
    void seeReviews();
    void getSample();
}

class Book implements Product {
    public void seeReviews() { /* ... */ }
    public void getSample()  { /* ... */ }
}
class DVD implements Product {
    public void seeReviews() { /* ... */ }
    public void getSample()  { /* ... */ }
}
class Shelf {
    Product p;
    void addProduct(Product product) { /* ... */ }
    void customizeShelf()             { /* ... */ }
}
  • Follows LSP: Product can be substituted by Book or DVD without breaking the program.
  • Follows DIP: the high-level Shelf depends on the abstraction Product, not the concretions.
Layered Architecture

Presentation layer → Business layer (Employee, save button click) → DataAccess layer (save data in DB). Each layer depends on interfaces exposed by the layer below, not concrete types.

DIP MCQ keys — High-level modules depend on abstractions / Abstractions should not depend on details / Primary goal is avoiding tightly coupled code / Interaction between high- and low-level modules should be abstract.
§ 3.11

Introduction to Design Patterns & the GoF

The Gang of Four (GoF) Book

Design Patterns: Elements of Reusable Object-Oriented Software, Addison-Wesley, 1994.
Authors: Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides.
The book defined 23 patterns in three categories.
CategoryFocus
CreationalThe process of object creation
StructuralStatic composition and structure of classes/objects
BehavioralDynamic interaction among classes/objects

What is a Design Pattern?

  • A reusable solution to a commonly occurring problem in software design.
  • Shows relationships and interactions between classes/objects without specifying final application classes.
  • A template, not a finished solution — it cannot be directly transferred into code.
  • Language-independent.

Four Essential Elements of a Pattern

  1. Pattern Name — a handle describing the problem, solutions, and consequences in a word or two.
  2. The Problem — describes when to apply the pattern (context, class/object structures, conditions).
  3. The Solution — describes the elements, their relationships, responsibilities, and collaborations.
  4. Consequences — results and trade-offs of applying the pattern.
Other elements of a pattern description
  • Applicability — situations where the pattern applies.
  • Structure — graphical representation (OMT/UML).
  • Participants — the classes/objects taking part.
  • Collaborations — how participants interact.
  • Implementation — pitfalls, hints, language-specific issues.
  • Related Patterns — closely related patterns and their differences.

Architecture vs Design Patterns

Architecture Patterns

Describe the structure of an entire system (e.g., MVC, Layered, Microservices).

Design Patterns

Solve recurring problems in smaller modules/classes within a system.

Why Learn Patterns?

  • Common design vocabulary — engineers can discuss abstractions independently of implementation.
  • Capture design expertise for reuse and avoiding mistakes.
  • Better documentation — patterns are documented well once.
  • Composition over inheritance: "Favor object composition over class inheritance." — GoF 1995:20

Pros and Cons

Pros
  • Consistency across designs
  • Common vocabulary
  • Faster time to solution
  • Reuse through composition
Cons
  • Some patterns have negative consequences (object proliferation, performance hits, extra layers)
  • Consequences are subjective to scenario
  • Subject to misinterpretation
  • Overuse → Anti-Patterns

How to Select a Design Pattern

  • Consider how patterns solve design problems (granularity, interfaces, object roles).
  • Scan intent sections.
  • Study how patterns interrelate.
  • Study patterns of similar purpose for similarities/differences.
  • Examine causes of redesign.
  • Consider what should be variable in your design — encapsulate what varies.
Intro MCQs — Design patterns represent best practices used by experienced developers / GoF book defined 23 patterns / Creational = object creation / Structural = static composition / Behavioral = dynamic interaction / Disadvantage: extra complexity or layers.
§ 3.12

Creational Patterns — Overview

Creational patterns provide various object creation mechanisms → increase flexibility and reuse. The basic new operator scatters creation across the codebase; over time, classes become tightly coupled. Creational patterns control object creation.
Recurring themes
  • Encapsulate knowledge about which concrete classes the system uses.
  • Hide how instances are created and put together.
Singleton
Exactly one instance of a class with a global access point.
Factory Method
Define an interface for creating an object; let subclasses decide the class.
Abstract Factory
Create families of related objects without specifying concrete classes.
Builder
Build a complex object step-by-step; same process produces different representations.
Prototype
Create new objects by cloning a prototypical instance.
Principle: Composition over inheritance — "Favor object composition over class inheritance." — GoF 1995:20
§ 3.13

Singleton

Intent: Ensure a class has only one instance and provide a global point of access to it.

Motivation

  • Part of the creational family.
  • Ensures only one instance exists in the JVM.
  • Provides a unique global access point.
  • The singleton is initialised only when first requested (in lazy variants).
  • Real-life: shared cloud storage — a single shared storage object instead of one per user.
Why Singleton
  1. Ensures a class has only one instance.
  2. Gives a global access point to that instance.
  3. Lazy variants: instance initialised only when first requested.
Problems it solves (at the cost of SRP)
  • Controls access to a shared resource (e.g., database, file).
  • Protects the single instance from being overwritten (unlike a plain global variable).
Solution skeleton (all implementations share this)
  1. Make the default constructor private → prevents outside new.
  2. Create a static method acting as a constructor — it calls the private constructor and caches the object in a static field. Every later call returns the cached object.
Uses

Logging, drivers, caching, thread pools. Used in java.lang.Runtime, java.awt.Desktop. Common inside Abstract Factory, Builder, Prototype, Façade.

Four Implementation Methods

Method 1 — Lazy Instantiation (NOT thread-safe)
class Singleton {
    private static Singleton obj;
    private Singleton() {}                       // private ctor
    public static Singleton getInstance() {
        if (obj == null) obj = new Singleton();  // lazy
        return obj;
    }
}

Simple, but two threads can both see obj == null and create two instances.

Method 2 — Synchronised getInstance()
class Singleton {
    private static Singleton obj;
    private Singleton() {}
    public static synchronized Singleton getInstance() {
        if (obj == null) obj = new Singleton();
        return obj;
    }
}

Thread-safe but slow — every call pays the synchronisation cost.

Method 3 — Eager Instantiation
class Singleton {
    private static Singleton obj = new Singleton();   // created at class load
    private Singleton() {}
    public static Singleton getInstance() { return obj; }
}

Thread-safe because the JVM runs the static initialiser once. Use only when the singleton is lightweight and used throughout the program.

Method 4 (Best) — Double-Checked Locking
class Singleton {
    private static volatile Singleton obj = null;
    private Singleton() {}
    public static Singleton getInstance() {
        if (obj == null) {
            synchronized (Singleton.class) {
                if (obj == null)               // second check inside lock
                    obj = new Singleton();
            }
        }
        return obj;
    }
}
The volatile keyword is essential — it ensures multiple threads see the obj variable correctly during initialisation. Synchronises only when the instance is still null → much less overhead.

Implementation Trade-offs

MethodThread-safe?Lazy?CostNotes
Classic lazy❌✔LowTwo threads can both create instances
Synchronised getInstance✔✔High on every callSimple, slow
Eager✔ (via JVM class loading)❌LowLightweight, always-used singletons
Double-Checked Locking✔✔Low after first initBest general-purpose choice

Participants & Applicability

  • Singleton — defines an Instance operation (class method/static) that lets clients access its unique instance. May be responsible for creating its own instance.
  • Client — accesses the instance only via Singleton.getInstance().
Use Singleton when…
  • There must be exactly one instance accessible from a well-known point.
  • The sole instance should be extensible by subclassing and clients should use the extended instance without code changes.

Consequences

Benefits
  1. Controlled access to the sole instance
  2. Reduced namespace pollution (vs globals)
  3. Permits refinement of operations (subclassing)
  4. Permits variable number of instances
  5. More flexible than class operations (statics can't be overridden polymorphically)
Drawbacks
  1. Violates SRP (controls instantiation + global access)
  2. Can mask bad design — components know too much about each other
  3. Special treatment in multi-threaded environments
  4. Hard to unit-test — frameworks use inheritance for mocks; singletons resist
Relations with other patterns
  • Façade is often transformed into a Singleton (one façade usually suffices).
  • Flyweight resembles Singleton if you reduce all shared state to one object — but Flyweight allows many instances with different intrinsic states, and Flyweights are immutable while Singletons can be mutable.
  • Abstract Factory, Builder, Prototype can all be implemented as Singletons.
Singleton MCQ keys — Primary purpose: restrict to one instance, provide global access / Keyword in DCL: volatile / Private constructor prevents outside creation / Major drawback: difficult unit testing.
§ 3.14

Factory Method (a.k.a. Virtual Constructor)

Intent: Provide an interface for creating objects in a superclass, but allow subclasses to alter the type of objects that will be created.
Per GoF — "defines an interface for creating an object, but lets subclasses decide which class to instantiate. The Factory method lets a class defer instantiation to subclasses."

Motivation

  • Create objects without specifying the exact class.
  • Object creation often requires complex processes not fitting inside the composing object.
  • Avoid duplication of creation code; avoid giving the composing object too much knowledge.
  • The factory method defines a separate method for creating objects; subclasses override to specify the derived type.

Applicability — use Factory when:

  • A class can't anticipate the class of objects it must create.
  • A class wants its subclasses to specify the objects it creates.
  • Classes delegate responsibility to one of several helper subclasses and you want to localise the knowledge of which helper is used.

Participants

  • Product — defines the interface of the objects the factory method creates.
  • ConcreteProduct — implements the Product interface.
  • Creator — declares the factory method; may provide a default returning a default ConcreteProduct.
  • ConcreteCreator — overrides the factory method to return a specific ConcreteProduct.
Collaboration

The Creator relies on its subclasses to define the factory method so it returns an instance of the appropriate ConcreteProduct.

Car Rental Example

Two customer types (company contract, no contract), each requesting grade A or B cars.

Without Factory

Client's main is cluttered with selection logic mixed with preparation steps (clean / mechanic / fuel).

With Factory

A Factory super-class has a create() method which calls the abstract retrieveCar() (subclass-specific selection logic), then runs shared preparation steps.

Consequences

  • Provides hooks for subclasses — more flexible than direct creation.
  • Connects parallel class hierarchies — when one class delegates responsibilities to another, factory methods can be called from clients.

Issues

  1. Two major varieties:
    • Abstract Creator (no default impl) → subclass must override.
    • Concrete Creator (provides default impl).
  2. Parameterised factory methods — one factory method creates multiple kinds of products based on a parameter; all share the Product interface.
  3. Language-specific variants — e.g., Smalltalk methods returning a class.
Factory MCQ keys — Main purpose: interface for object creation, subclasses decide the concrete class / Also known as Virtual Constructor / Client accesses objects via a common interface / Factory hides creation logic / Improves flexibility by delegating creation to subclasses / Returns objects of the Product interface.
§ 3.15

Builder

Intent: Separate the construction of a complex object from its representation so that the same construction process can create different representations.

Introduction

  • For when a sequence of complex operations is needed to produce an object instance.
  • Delegate construction to a specialised class aware of the required logic.
  • Enables Single-Responsibility classes — separates creation from business logic.
  • Can lead to reusability, smaller code base, better SOLID compliance.

Analogy — a car factory: a boss tells workers/robots to build each part; workers build each part and add them to the car.

Why Builder — Problems Solved

  • How can the same construction process create different representations of a complex object?
  • How can a class that creates a complex object be simplified?

Solution: encapsulate creating and assembling parts of a complex object in a separate Builder; let a class delegate creation to a Builder.

Motivation — RTF Reader

  • An RTF reader needs to convert RTF to many text formats (ASCII, text widget, TeX).
  • Number of possible conversions is open-ended → easy to add new conversion without modifying the reader.
  • Configure the RTFReader with a TextConverter object.
  • Each converter = a Builder; the reader = the Director.

Participants

  • Builder — abstract interface for creating parts of a Product.
  • ConcreteBuilder — constructs and assembles parts; defines and tracks the representation; provides the product-retrieval method (e.g., getBurger, GetASCIIText).
  • Director — constructs an object using the Builder interface.
  • Product — the complex object under construction.
Collaboration
  1. Client creates the Director and configures it with a Builder.
  2. Director notifies Builder to build each part of the product.
  3. Builder handles Director's requests and adds parts.
  4. Client retrieves the product from the Builder.

Burger-Restaurant Example

Goal: build varieties of burgers (cheeseburger, veg-burger).

Without Builder — construction in the main context
public class Main {
    public static void main(String[] args) {
        Burger cheeseBurger = new Burger();
        cheeseBurger.setBun("White Bread");
        cheeseBurger.setMeat("Beef");
        cheeseBurger.setSalad("Iceberg");
        cheeseBurger.setCheese("American Cheese");
        cheeseBurger.setSauce("Secret Sauce");
        cheeseBurger.print();
    }
}

Every variant requires the same boilerplate; client code drowns in setter calls and there's no way to enforce a build order.

Without Builder

main sets bun, meat, salad, cheese, sauce manually every time.

With Builder

Chef (Director) is given a BurgerBuilder subclass; Chef invokes setBuilder() then build(); client only interacts with Chef.

Applicability — use Builder when:

  • The algorithm for creating the complex object should be independent of the parts and how they're assembled.
  • The construction process must allow different representations.
  • You want to avoid a telescoping constructor (a constructor with many optional parameters, overloaded in multiple short versions).
  • You want different representations where construction differs only in details.

Consequences

  • Lets you vary the product's internal representation by swapping Builders.
  • Isolates code for construction and representation.
  • Gives finer-grain control over construction.

How to Implement

  1. Clearly define common construction steps for all product representations.
  2. Declare those steps in the base Builder interface.
  3. Create a ConcreteBuilder for each representation.
  4. Consider a Director class encapsulating various construction sequences.
  5. Client code creates both Builder and Director; passes Builder to Director.
  6. Result comes from the Director if products share an interface; else the client fetches from the Builder.
Pros and Cons
Pros
  • Construct objects step-by-step, defer/recurse steps
  • Reuse the same construction code for various representations
  • SRP — isolate complex construction from business logic
Cons
  • More classes → more complexity
Relations with other patterns
  • Many designs start with Factory Method (simpler) and evolve toward Abstract Factory, Prototype, or Builder (more flexible, more complex).
  • Builder constructs complex objects step by step. Abstract Factory creates families of related objects and returns the product immediately; Builder allows extra construction steps before fetching.
  • Useful for Composite trees — construction can be recursive.
  • Can combine with Bridge — Director = abstraction, different Builders = implementations.
  • Abstract Factory, Builder, Prototype can all be Singletons.
Builder MCQ keys — Director controls the building process / Builder interface creates parts of a product / ConcreteBuilder implements building steps / Product is the complex object under construction / Benefit: finer control over construction / Helps avoid telescoping constructors.
§ 3.16

Prototype

Intent: Prototype lets you copy existing objects without making your code dependent on their classes.

Motivation

  • Specify the kind of objects to create using a prototypical instance; create new objects by copying this prototype.
  • Use Prototype when a client needs objects that are alike or differ only in state, and new is expensive/complicated.
  • Makes new instances by copying existing instances — in Java via clone() or de-serialisation for deep copies.
  • Client code can make new instances without knowing the specific class.
Usage in Java
  • Available via the Cloneable interface.
  • Any class implementing Cloneable can use Object.clone().
  • Identification: presence of clone() / copy() methods.

Geometric Objects Example

  • All shape classes follow the same interface providing clone().
  • A subclass may call the parent's cloning method before copying its own fields to the resulting object.

Prototype Registry

  • A centralised store of pre-defined prototype objects.
  • Clients ask the registry (by name/parameters) for a new object.
  • The registry finds the appropriate prototype, clones it, and returns a copy.

Cell Division Example

The mitotic division of a cell into two identical cells is a prototype demonstrating self-cloning.

Checklist

  1. Add clone() to the existing product hierarchy.
  2. Design a registry caching prototype objects — encapsulated in a Factory or in the product base class.
  3. Design a factory method that (optionally) accepts arguments, finds the right prototype, calls clone(), and returns the result.
  4. Client replaces every new call with the factory method.

Applicability — use Prototype when:

  • System should be independent of how products are created, composed, or represented.
  • Classes to instantiate are specified at run-time (e.g., dynamic loading).
  • To avoid building a class hierarchy of factories that parallels the products.
  • Instances have only a few combinations of state → install a few prototypes and clone them rather than manually instantiating each time.
Collaboration

A client asks a prototype to clone itself.

Consequences

  • Adding/removing products at run-time.
  • Specify new objects by varying values or structure.
  • Reduced subclassing.
  • Configuring applications with classes dynamically.

Issues to Consider

  • Using a prototype manager — keep a registry of available prototypes when the set is dynamic. Clients fetch from the registry before cloning.
  • Implementing Clone correctly — particularly tricky with circular references.
  • Initialising clones — some clients want to set state after cloning. Since parameter counts vary across classes, passing params in clone() would break a uniform cloning interface.
Prototype MCQ keys — Creates objects by copying existing objects / Main intent: copy without depending on classes / Useful when creation is expensive or complex / Registry stores and retrieves prototype objects / Client requests an object to clone itself / Benefit: reduced subclassing.

§ 4.1

Structural Patterns — Overview

Structural design patterns are concerned with how classes and objects can be composed to form larger structures. They simplify structure by identifying relationships, focus on inheritance and composition, and keep larger structures flexible and efficient.
  • Structural class patterns use inheritance.
  • Structural object patterns use composition.
The four structural patterns in the syllabus
Adapter
Convert one interface into another a client expects (a.k.a. Wrapper).
Facade
Unified simple interface to a complex subsystem.
Proxy
Surrogate / placeholder controlling access to a real object.
Flyweight
Share fine-grained objects efficiently via intrinsic / extrinsic state split.

Other GoF structural patterns (not in syllabus): Bridge, Composite, Decorator.

§ 4.2

Adapter (a.k.a. Wrapper)

Intent: Adapter lets classes work together that couldn't otherwise because of incompatible interfaces. "Convert the interface of a class into another interface that a client wants."

Motivation

  • Adapts between classes/objects — acts as a bridge.
  • Real-world: power-supply adapters, camera memory-card adapters.
  • Sometimes a reusable toolkit class isn't usable because its interface doesn't match the domain interface the application needs.
Advantages
  • Lets two previously incompatible objects interact.
  • Enables reusability of existing functionality.

Participants

  1. Target — defines the domain-specific interface that the Client uses.
  2. Adapter — adapts the interface of Adaptee to the Target.
  3. Adaptee — the existing interface that needs adapting.
  4. Client — collaborates with objects conforming to Target.

The Adapter class is a wrapper that implements the desired Target interface and internally modifies the specific request available from the Adaptee class.

Example 1 — MediaPlayer

We have MediaPlayer (plays mp3) and AdvancedMediaPlayer (plays vlc, mp4). Create a MediaAdapter implementing MediaPlayer and using AdvancedMediaPlayer objects internally. AudioPlayer now uses MediaAdapter to play vlc/mp4 without knowing the concrete class.

Example 2 — Bird / ToyDuck

  • Bird has fly() and makeSound(). ToyDuck has squeak().
  • A BirdAdapter implements ToyDuck wraps a Bird and maps squeak() to Bird.makeSound().
  • This is an Object Adapter — the adapter holds an instance of the adaptee (composition/delegation).
  • Switch implementations with one line: ToyDuck toyDuck = new BirdAdapter(sparrow);

Example 3 — MPH ↔ km/h

A US app returns top speed in MPH; a UK client needs km/h. Build an adapter that converts the values.

Applicability — use Adapter when:

  • You want to use an existing class but its interface doesn't match what you need.
  • You want a reusable class that cooperates with unrelated or unforeseen classes with incompatible interfaces.
  • (Object Adapter only) You need to use several existing subclasses but it's impractical to adapt each interface by subclassing.

Software examples: wrappers for 3rd-party libraries — a middle layer between application and library, decoupling them.

Class Adapter vs Object Adapter

Class Adapter
  • Uses multiple inheritance
  • Only in languages supporting it (NOT Java/C#/PHP directly)
  • Subclasses both Target and Adaptee
  • Adapts a specific Adaptee class — its subclasses can't be adapted
  • No delegation code needed
  • Single object; no extra pointer indirection
  • Adapter can override Adaptee behaviour (it is a subclass)
Object Adapter
  • Uses composition / delegation
  • Single Adapter works with the Adaptee and all subclasses
  • Hard to override Adaptee behaviour
  • Main disadvantage: must write delegation code for every method
  • If a new method is added to the adaptee, Adapter must be extended

Issues

  • How much should the Adapter do? If Target and Adaptee are similar, delegate; if dissimilar, convert data structures and implement missing operations.
  • Two-way adapters for transparency — an adapted object no longer conforms to the Adaptee interface, so it can't be used where an Adaptee is expected. Two-way adapters provide that transparency — useful when two clients view the object differently.
Adapter MCQ keys — Primary purpose: convert incompatible interfaces into a compatible one / Allow incompatible classes to collaborate / Target defines the interface expected by the client / Adaptee = existing interface that needs adapting.
§ 4.3

Facade

Intent: Provide a unified interface to a set of interfaces in a subsystem. Façade defines a higher-level interface that makes the subsystem easier to use.

What is a Façade?

  • In architecture: the principal front of a building ("the house has a half-timbered façade").
  • Figurative: a deceptive outward appearance.

Facade in Software

  • An object providing a simplified interface to a larger body of code, a library, or any complex set of classes.
  • Example: fopen — a simple interface to open/read system commands.
  • A structural design pattern wrapping a complicated subsystem with a simpler interface.
Trade-off: if the Façade is the only access point, "power users" may lose flexibility.

Why Facade?

  • Structuring a system into subsystems reduces complexity.
  • Minimises communication and dependencies between subsystems — introduce a façade exposing a single simplified entry point.

Applicability — use Facade when:

  • Providing a simple interface to a complex subsystem (a default view good enough for most clients).
  • Decoupling a subsystem from clients and other subsystems → promotes subsystem independence and portability.
  • Defining an entry point to each subsystem level.
  • Simplifying inter-subsystem dependencies — they communicate only through façades.

Advantages

  • Shields clients from subsystem components → fewer objects for clients → easier to use.
  • Promotes weak coupling between the subsystem and its clients.
  • Eliminates complex/circular dependencies when client and subsystem are developed independently.
  • Limits recompilation needed for small changes in an important subsystem.

Known Uses

  • ET++ framework — built-in browsing tools implemented in a separate subsystem with a ProgrammingEnvironment façade class with operations like InspectObject, InspectClass.
  • Choices OS — uses façades to compose many frameworks. Key abstractions (processes, storage, address spaces) each have a façade supporting porting to different hardware platforms.

Implementation

  • Reduce client-subsystem coupling by creating a "Façade Abstract Class" and concrete subclasses implementing the subsystem. Clients talk to the abstract façade. Alternative: configure a façade object with different subsystem objects.
  • Public vs Private interfaces: classes and subsystems both have public/private interfaces. Façade belongs to the public interface — the private interface is only for subsystem extenders.

Examples

  • Booking a package: a TravelFacade simplifies booking flight + hotel + cab through a single call.
  • Home theater: a HomeTheaterFacade provides watchMovie() / endMovie() calls coordinating amplifier, tuner, DVD player, projector, lights, screen, popcorn popper.
Facade MCQ keys — Primary intent: unified interface to a set of interfaces / Used to minimise communication/dependencies between subsystems / Key advantage: reduces objects clients interact with / Promotes weak coupling between subsystem and client.
§ 4.4

Proxy

Core idea: A proxy object provides a surrogate or placeholder for another object to control access to it. The client doesn't directly talk to the original — it calls the proxy, which calls the real subject.

Intent

  • Heavily used for lazy loading — don't create a full object until actually needed.
  • A class represents the functionality of another class.

Three Main Variations

VariationPurpose
Remote ProxyLocal representative for an object in a different address space.
Virtual ProxyCreates expensive objects on demand.
Protection ProxyControls access rights to the original object.

An additional one often discussed: Cache Proxy — caches request results (data that doesn't change often or is expensive to create).

Participants

  • Subject — interface exposing the functionality available to clients.
  • Real Subject — class implementing Subject; concrete logic hidden behind the proxy.
  • Proxy — hides the real object by extending the same interface; clients talk to the real object via the proxy.

Examples

  • Check / bank draft as a proxy for funds in an account — used in place of cash; controls access to cash in the issuer's account.
  • Corporate network proxy — all network requests go through a proxy that checks allowed websites/data; if suspicious, blocks the request.

When to use — Applicability

  • Cache Proxy — caches relevant data; Proxy returns cached results without hitting the real object.
  • Remote Proxy — local object represents one in another location; handles wrapping/unwrapping and remote calls.
  • Protection Proxy — security around the Real Subject, enforcing access control.
  • Virtual Proxy — lazy initialisation for a complex/expensive object only needed some of the time.

Advantages and Disadvantages

Advantages
  • Works even if the real service isn't ready/available
  • Follows the Open/Closed Principle — new proxies introduced without changing service or clients
Disadvantages
  • Adds an extra layer between client and Real Subject → more complexity
  • Response time may be delayed
  • Extra request forwarding introduced

Usage Examples

  1. Image viewer that lists + displays high-res photos — don't load actual photos until the user selects one.
  2. Document editor with embedded graphics — don't load all pictures on open since not all are visible.
  3. Protective proxy — office proxy server allows only valid sites.
  4. Adds thread-safety to an existing class without modifying it.

Worked Example — Lazy Image Loading

  • Interface accessed by the client is implemented by both ProxyImage and RealImage.
  • RealImage lives on another system, loading image info from the DB.
  • ProxyImage on our system represents the RealImage, avoiding repeated heavy loads.

YouTube Downloader Example (lazy + caching)

  • A 3rd-party YouTubeDownloader keeps re-downloading the same video.
  • A CachedYouTubeDownloader proxy implements the same interface, delegates to the real downloader, caches downloaded files, and returns cached files on repeat requests.
Proxy MCQ keys — Primary role: act as a placeholder/surrogate / Key use: lazy loading / Subject defines the interface used by clients / Virtual Proxy creates expensive objects only when needed / Protection Proxy controls access rights / Common scenario: controlling internet access in networks.
§ 4.5

Flyweight

Intent: Use sharing to support a large number of fine-grained objects efficiently. Flyweight lets you fit more objects into available RAM by sharing common parts of state instead of keeping all data in each object.

Motivation

  • Application uses many objects.
  • A flyweight is a shared object used in multiple contexts simultaneously.
  • Acts as an individual object in each context.
  • Key concept: intrinsic vs extrinsic state.

Problem

Placing the same information in many different places:

  1. Mistakes can be made in copying information.
  2. If data changes, every occurrence must be located and changed → difficult maintenance.
  3. Documents become large if the same info is repeated.

Issues: RAM overhead per instance, CPU overhead managing memory.

Solution — Intrinsic vs Extrinsic State

Intrinsic StateExtrinsic State
Shareable?✔ Yes✗ No (stateless)
Depends on context?✗ No✔ Yes
StorageInside flyweight (stored once for many objects)Outside flyweight; computed on the spot or passed in
ExampleName, postal abbreviation, time zoneAccess to region

Flyweight objects contain only intrinsic state; extrinsic state is passed in by the client.

Example — OO Document Editor

  • Represent each character as an object → flexibility; characters and embedded elements treated uniformly for drawing/formatting.
  • One flyweight per letter of the alphabet:
    • Intrinsic: character code (stored in the flyweight).
    • Extrinsic: coordinate position in the document; typographic style (font, color) — determined by text-layout algorithms and formatting commands at each location.

Participants

  • Flyweight (e.g., Glyph) — interface through which flyweights receive and act on extrinsic state.
  • ConcreteFlyweight (e.g., CharacterGlyph) — implements Flyweight; adds storage for intrinsic state. Must be sharable; stored state must be intrinsic.
  • UnsharedConcreteFlyweight (e.g., Row, Column) — not all subclasses need to be shared. The interface enables sharing; it doesn't enforce it.
  • FlyweightFactory (e.g., GlyphFactory) — creates and manages flyweight objects; ensures proper sharing. When a client asks, factory supplies an existing instance or creates one.
  • Client — keeps a reference to flyweight(s); computes or stores the extrinsic state.

Applicability — use Flyweight when:

  • Application uses a large number of objects.
  • Storage costs are high due to sheer quantity.
  • Most object state can be made extrinsic.
  • Many groups can be replaced by relatively few shared objects once extrinsic state is removed.
  • Application doesn't depend on object identity (since flyweights are shared, identity tests will return true for conceptually distinct objects).

Consequences

Disadvantage

Run-time cost of transferring/finding/computing extrinsic state — especially if it was previously intrinsic.

Advantage

Storage savings depend on:

  • Reduction in total instances from sharing
  • Amount of intrinsic state per object
  • Whether extrinsic state is computed or stored

Implementation Notes

  • Removing extrinsic state — applicability is largely determined by how easily you can identify and externalise it. If extrinsic kinds are as varied as the objects, no sharing benefit.
  • Managing shared objects — clients shouldn't instantiate directly. FlyweightFactory lets clients look up flyweights, often via an associative store.

More Examples

  • Pen with refills — pen is a flyweight with refill as an extrinsic attribute; body, pointer are intrinsic. Modules asking for a red pen share the same instance.
  • Java String constants — string pool is a flyweight. Strings with the same content share the same reference.
  • Browsers — images used in multiple places are loaded once and reused from cache. URL = intrinsic; position/width/height = extrinsic.
Flyweight MCQ keys — Primary goal: reduce memory by sharing objects / Intrinsic state: context-independent and sharable / Extrinsic state: context-dependent and not shared / FlyweightFactory creates and manages shared flyweight objects / Flyweight objects must be sharable / Example: String constant pool.

§ 4.7

Chain of Responsibility

Intent: Avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle the request. Chain receiving objects and pass the request along until one handles it.

Core Idea

Establishes a chain within a system: a message is either handled at the level where it is first received, or directed to an object that can handle it. Used to manage algorithms, relationships, and responsibilities between objects.

Motivation

  • Promotes loose coupling between sender and receiver.
  • More than one object gets an opportunity to handle the request.
  • Receiving objects are chained; the request passes along until one handles it.
  • The set of potential handlers and the chain order can be decided dynamically at runtime by the client.

Participants

  • Handler — usually an interface; the entry point that receives and dispatches to the chain. Holds reference only to the first handler.
  • ConcreteHandler — handles the requests it is responsible for; can access its successor; if it can't handle, forwards to the successor.
  • Client — initiates the request to a ConcreteHandler on the chain.

Applicability — use when:

  • More than one object may handle a request, and the handler isn't known a priori.
  • You want to issue a request to one of several objects without specifying the receiver explicitly.
  • The set of objects that can handle a request should be specified dynamically.

Scenario 1 — Enterprise Email Triage

Enterprise gets 4 types of email: fan mail, complaints, feature requests, spam. AI detectors exist for each type. Build a chain that routes incoming mail to the right detector.

Scenario 2 — Transaction Approval (full implementation)

Rule: Transactions < ₹1,00,000 — Manager / < ₹10,00,000 — Vice President / < ₹25,00,000 — CEO.
Abstract Handler
abstract class TransactionProcessor {
    private TransactionProcessor transactionProcessor;

    abstract protected Double getApprovalLimit();
    abstract protected String getDesignation();

    public void setSuccessor(TransactionProcessor tp) { this.transactionProcessor = tp; }

    public void approve(Transaction transaction) {
        if (transaction.getAmount() <= 0.0) {
            System.out.println("Invalid Amount. Amount should be > 0"); return;
        }
        if (transaction.getAmount() <= getApprovalLimit()) {
            System.out.println("Transaction for amount " + transaction.getAmount()
                + " approved by " + getDesignation());
        } else {
            if (transactionProcessor == null) {
                System.out.println("Invalid Amount. Amount should not exceed 25lacs!");
                return;
            }
            transactionProcessor.approve(transaction);     // forward to successor
        }
    }
}
Concrete Handlers
class Manager extends TransactionProcessor {
    protected Double getApprovalLimit() { return 100000.0; }
    protected String getDesignation()   { return "manager"; }
}
class VicePresident extends TransactionProcessor {
    protected Double getApprovalLimit() { return 1000000.0; }
    protected String getDesignation()   { return "Vice President"; }
}
class CEO extends TransactionProcessor {
    protected Double getApprovalLimit() { return 2500000.0; }
    protected String getDesignation()   { return "CEO"; }
}
Request Object
class Transaction {
    private Double amount;
    private String purpose;
    Transaction(Double amount, String purpose) { this.amount = amount; this.purpose = purpose; }
    public Double getAmount() { return amount; }
    public String getPurpose() { return purpose; }
}
Wiring the Chain
public class ChainOfResponsibility {
    public static void main(String[] args) {
        Manager manager = new Manager();
        VicePresident vicePresident = new VicePresident();
        CEO ceo = new CEO();

        manager.setSuccessor(vicePresident);
        vicePresident.setSuccessor(ceo);

        manager.approve(new Transaction(2600000.0, "general")); // rejected (exceeds 25 lac)
        manager.approve(new Transaction(50000.0,   "general")); // manager
        manager.approve(new Transaction(120000.0,  "general")); // VP
        manager.approve(new Transaction(1500000.0, "general")); // CEO
        manager.approve(new Transaction(0.0,       "general")); // invalid
    }
}

Consequences

  • Reduced Coupling — objects free from knowing which object handles the request.
  • Added Flexibility in assigning responsibilities — chain can change at runtime; specialised handlers via subclassing.
  • Receipt not guaranteed — request could fall off the chain or be dropped with a bad chain.

Benefits, Uses, Drawbacks

Benefits
  • Decouples sender and receivers
  • Object doesn't need to know chain structure or direct member references
  • Add/remove responsibilities dynamically by changing members or order
Uses
  • Event handling in Windows systems (mouse clicks, keyboard events)
  • Handling chains of Exceptions in Java
Drawbacks: Execution isn't guaranteed — may fall off the end (can be advantage or disadvantage). Hard to observe runtime characteristics and debug.
CoR MCQ keys — Main idea: requests handled by multiple objects sequentially in a chain / Passed along a chain until handled / Handler receives requests and dispatches / ConcreteHandler handles its requests and forwards others / Used when multiple objects may handle a request / Drawback: execution not guaranteed.
§ 4.8

Command

Intent: Encapsulate a request in an object. Allows parameterisation of clients with different requests, and saving requests in a queue.

Core Idea

  • Command pattern is a data-driven behavioral pattern.
  • A request is wrapped as an object (command) and passed to an invoker.
  • The invoker looks up the right object to handle the Command and passes it.
  • The Command object contains everything needed to perform the action: method name, owner, arguments.

Motivation

  • Sometimes you need to issue requests to objects without knowing the operation or the receiver.
  • Example: UI toolkits have buttons and menus that carry out a request on user input. The toolkit can't know what should happen on which object — only applications using it know.
  • Command lets toolkit objects make requests of unspecified application objects by turning the request itself into an object.

Participants

  • Command — interface with an execute() operation.
  • ConcreteCommand — binds a Receiver to an action; implements execute() by invoking the corresponding operation on the Receiver.
  • Client (Application) — creates a ConcreteCommand and sets its Receiver.
  • Invoker — asks the Command to carry out the request.
  • Receiver — knows how to perform the operation. Can be any class.
Collaborations
  1. Client creates a ConcreteCommand and specifies its Receiver.
  2. Invoker stores the ConcreteCommand.
  3. Invoker calls execute(). For undoable commands, ConcreteCommand stores state for undoing prior to executing.
  4. ConcreteCommand invokes operations on the Receiver.

Scenario 1 — Video Game Joystick

Use the Command pattern to control a player's movement via a joystick — joystick inputs invoke commands that move the player.

Consequences

  • Decouples the object that invokes the operation from the one that knows how to perform it.
  • Commands are first-class objects — manipulable and extendable like any object.
  • Easy to add new Commands without changing existing classes.
  • Commands can be composed into a composite command.
Limitations: Must create a concrete command class for each request. Can end up with a large number of classes if not managed.

Applicability — use when:

  • Parameterise objects by the actions they perform (a way to implement callbacks).
  • Specify, queue, and execute requests at different times (partitioning complex requests).
  • Need to support undo (single or multi-level).
  • Need a change log for recovery.
  • Transaction-based applications (extendable).
  • Applications that learn the receiver dynamically.

§ 4.9

Interpreter

Intent: Define a way to interpret a domain-specific language or grammar. Define the grammar of a language and provide a way to evaluate sentences in it.

Intent / Capabilities

  1. Interpret and execute expressions in the language.
  2. Break complex expressions into smaller components.
  3. Create parsers and compilers for the defined language.
  4. Build scripting and domain-specific languages.
  5. Provide flexibility/extensibility for defining and interpreting expressions.
  6. Support rules governing the behaviour of the language.
  7. Facilitate creation of abstract syntax trees (AST) to represent expressions.
  8. Allow interpreters that handle different types of expressions and operators.

Motivation

  1. Define a domain-specific language — tailored to a particular problem; makes code expressive and concise.
  2. Dynamic interpretation — interpret/execute expressions at runtime; useful where static evaluation is hard.
  3. Separate concerns — defining the language and interpreting it are separate; changes are easier.
  4. Build parsers/compilers for the defined language.
  5. Facilitate communication between components via a common language.

Participants

  • AbstractExpression — declares an interpret() operation.
  • TerminalExpression (e.g., ThousandExpression, HundredExpression, TenExpression, OneExpression) — implements interpret() for terminal symbols. One instance per terminal symbol in a sentence.
  • NonterminalExpression — one class per grammar rule R ::= R1 R2 ... Rn. Maintains AbstractExpression variables for each symbol. Implements interpret() recursively.
  • Context — contains information global to the interpreter.
  • Client (InterpreterApp) — builds or receives an AST for a sentence; invokes interpret().

Example

Expression 2 + 3:

  • Model with ConstantExpression for operands and AddExpression for addition. All implement a common Expression interface.
  • AddExpression(ConstantExpression(2), ConstantExpression(3))

Expression (2 + 6) + (1 + 9):

AddExpression(
    AddExpression(ConstantExpression(2), ConstantExpression(6)),
    AddExpression(ConstantExpression(1), ConstantExpression(9))
)

Applicability

Useful when you need to interpret a domain-specific language or grammar — compilers, interpreters for scripting languages, rule-based systems.

Applications

  • Regular expressions — parse and execute regexes for complex matching.
  • SQL queries — parse and execute queries.
  • Mathematical expressions — parse and evaluate.
  • Programming languages — interpreters execute scripts/runtime code/on-the-fly compilation.

Advantages and Disadvantages

Advantages
  • Flexibility — implement your own grammar
  • Extensibility — add new expressions easily
  • Separation of Concerns — parsing separated from execution
Disadvantages
  • Complexity — adds complexity for complex grammars
  • Performance — slower than compiling or using a parser generator
  • Maintenance — hard when language changes frequently
Consequence: Interpreter is powerful for flexible/extensible languages but trades off complexity, performance, and maintenance. Weigh these before adopting.
Interpreter MCQ keys — Primary purpose: interpret a DSL or grammar / Useful for DSLs because it lets developers define a language specific to a problem domain / Context contains information global to the interpreter / Client builds an AST and invokes interpret() / Typical application: Regular expressions / Key advantage: flexibility for complex grammars.
§ 4.10

Iterator

Intent: Provide a way to access the elements of an aggregate object sequentially without exposing the underlying representation. Gives a uniform way to access elements of a list, set, or map without knowing the collection's implementation.

Core Idea

  • Standardised way to access and manipulate collections simply, efficiently, and flexibly.
  • Useful for algorithms needing sequential access — sorting, searching, filtering. Iterator-based algorithms work on any collection regardless of its implementation.

Motivation — Five Key Concepts

  • Encapsulation — encapsulates traversal logic separately from the collection. Algorithms and clients don't need to know internal representation.
  • Separation of Concerns — separates managing the collection from traversal. Code is modular, reusable, maintainable.
  • Abstraction — provides a standard interface for traversing. Multiple kinds of collections share the same traversal interface.
  • Lazy Evaluation — objects in the collection can be computed or retrieved only when needed (useful for large datasets or streams).
  • Multiple Traversals — multiple iterators over the same collection can coexist, each with its own state. Pause and resume iteration.

Participants

  • Iterator (AbstractIterator) — interface for accessing and traversing elements.
  • ConcreteIterator — implements the iterator interface; tracks current position in the traversal.
  • Aggregate (AbstractCollection) — interface for creating an Iterator object.
  • ConcreteAggregate (Collection) — implements the creation interface; returns an instance of the proper ConcreteIterator.

How to Implement

  1. Declare the iterator interface — at minimum a method to fetch the next element; add conveniences (previous, current position, hasNext).
  2. Declare the collection interface; describe a method for fetching iterators. Return type equals the iterator interface.
  3. Implement concrete iterator classes for each collection. Iterator linked with a collection instance via constructor.
  4. Implement the collection interface in your collection classes. Collection passes itself to the iterator's constructor.
  5. Replace all collection traversal code in the client with uses of iterators. Client fetches a new iterator whenever it needs to traverse.

Consequences

  1. Improved code flexibility — iterate in different ways without changing the collection's structure.
  2. Better encapsulation — collection internals hidden from the client.
  3. Simplified client code — standardised access reduces complexity.
  4. Performance overhead — extra method calls and object creations; negligible for small collections.

Applicability — use when you need:

  • Accessing elements in a collection (standard interface for traversal).
  • Simplifying complex data structures — traverse trees and graphs.
  • Implementing lazy evaluation — elements computed only when needed.
  • Supporting multiple traversals — without resetting state.
  • Providing a standard interface to algorithms working with different collection types.

Advantages and Disadvantages

Advantages
  • Single Responsibility — extracting bulky traversal cleans up clients and collections
  • Open/Closed — new collections and iterators pluggable without breaking existing code
  • Iterate the same collection in parallel (each iterator has its own state)
  • Can delay iteration and continue when needed
Disadvantages
  • Overkill for simple collections
  • May be less efficient than going through specialised collections directly

§ 4.11

Anti-Patterns — Definitions

Pattern: Good ideas. AntiPattern: Bad ideas. Refactoring: Better ideas.

What is an Anti-Pattern?

A literary form describing a commonly occurring solution to a problem that generates decidedly negative consequences.

May result from…
  • A manager/developer not knowing any better
  • Having insufficient knowledge/experience for the problem
  • Applying a perfectly good pattern in the wrong context
Also described as…
  • "Negative solutions" — solutions presenting more problems than they address.
  • A natural extension of design patterns.
  • Bridge the gap between architectural concepts and real-world implementations.
  • Provide knowledge to prevent and recover from common mistakes.
  • Present detailed plans for reversing causes and implementing productive solutions.

Anti-Patterns vs Design Patterns

Design Pattern

A general repeatable solution to a commonly occurring problem.

Anti-Pattern

A recognised poor solution to the problem, plus a refactored solution.

Why Study Anti-Patterns?

  • Easily identifiable templates for common problems + a path of action to fix them.
  • Real-world experience in recognising recurring problems with detailed remedies.
  • A common vocabulary for identifying problems and discussing solutions.
  • Provide stress release as shared misery.
  • Ensure common problems are not repeated within an organisation.

Software Design-Level Model (SDLM)

Architectural scale goes from widest to narrowest:

  • Global level — design issues globally applicable across systems; coordination across organisations participating in cross-organisational communication and information sharing.
  • Enterprise level — coordination and communication across a single organisation; may be distributed across locations and heterogeneous systems.
  • System level — communications and coordination across applications and sets of applications.

§ 4.14

Project Management — Analysis Paralysis

In modern engineering, more than half the job is human communication. Management anti-patterns identify scenarios where these issues are destructive to software processes. Managers play a vital role in (1) software process management, (2) resource management, (3) external relationship management.

Symptoms and Consequences

  • Multiple project restarts and model rework due to personnel changes or shifts in direction.
  • Design/implementation issues continually reintroduced in the analysis phase.
  • Cost of analysis exceeds expectation with no predictable end point.
  • Analysis phase no longer involves user interaction; analysis becomes speculative.
  • Complexity of analysis models produces intricate implementations — hard to develop, document, test.
  • GoF-level design decisions are being made in the analysis phase.

Typical Causes

  • Management assumes a waterfall progression, but in reality systems are built incrementally.
  • Management trusts its ability to analyse/decompose more than to design/implement.
  • Insistence on completing all analysis before design begins.
  • Analysis-phase goals aren't well defined.
  • Planning/leadership lapses when moving past analysis.
  • Management unwilling to declare any part of the domain sufficiently described.
  • Project vision/focus on deliverable is diffused; analysis goes beyond meaningful value.

Refactored Solution — Incremental Development

Key to OO development is incremental development. Waterfall assumes a-priori problem knowledge; incremental processes assume problem+solution details emerge during development.

Internal Increment

Builds software essential to implementation infrastructure. E.g., a third-tier database + data-access layer. Builds common infrastructure used by multiple use cases and minimises rework.

External Increment

User-visible functionality.

§ 4.15

Software Architecture — Vendor Lock-In

A software project adopts a product technology and becomes completely dependent on the vendor's implementation. When upgrades happen, software changes and interoperability problems occur, requiring continuous maintenance to keep the system running.

Architecture AntiPatterns focus on system-level and enterprise-level structure of applications and components. Although software architecture as an engineering discipline is relatively immature, its overarching importance has been demonstrated repeatedly.

Symptoms and Consequences

  • Commercial product upgrades drive the application-software maintenance cycle.
  • Promised features are delayed or never delivered → causes failure to deliver application updates.
  • The product deviates significantly from the advertised open-systems standard.
  • If a product upgrade is missed entirely, a repurchase and reintegration is often required.

Typical Causes

  • Product varies from open-systems standards because there's no effective conformance process.
  • Product selected based entirely on marketing/sales info, not technical inspection.
  • No technical approach for isolating application software from direct product dependency.
  • Application programming requires in-depth product knowledge.
  • Product's complexity/generality greatly exceeds the application's needs.

Refactored Solution — Isolation Layer

Separate software packages and technology with an isolation layer. Applicable when:

  • Isolation of application software from lower-level infrastructure (middleware, OS, security).
  • Changes to infrastructure anticipated within the life cycle.
  • A more convenient programming interface is needed.
  • Consistent handling of the infrastructure is needed across many systems.
  • Multiple infrastructures must be supported (serially or concurrently).
§ 4.16

Software Development — The Blob (God Class)

Procedural-style design produces one object holding the lion's share of responsibilities, while other objects only hold data or execute simple processes. Solution: refactor to distribute responsibilities more uniformly and isolate change.

Good software structure is essential for extension and maintenance. Development is chaotic, so implemented structure tends to stray from planned structure (architecture, analysis, design).

Background

Found in designs where one class monopolises processing, and other classes primarily encapsulate data. Class diagram: a single complex controller class surrounded by simple data classes. The majority of responsibilities are allocated to a single class.

Symptoms and Consequences

  • A single class with a large number of attributes/operations — 60+ usually indicates a Blob.
  • A disparate collection of unrelated attributes/operations in a single class → lack of cohesion.
  • The Blob class is typically too complex for reuse and testing; reusing for a subset introduces excessive complexity.
  • May be expensive to load into memory, wasting resources for simple operations.

Typical Causes

  • Lack of an object-oriented architecture.
  • Lack of (any) architecture.
  • Lack of architecture enforcement.
  • Too limited intervention.
  • Specified disaster — architecture explicitly specifies a blob.

Refactored Solution (Steps)

  1. Categorise related attributes and operations according to contracts — cohesive groups directly related to a common focus, behaviour, or function within the system.
  2. Look for "natural homes" for these contract-based collections of functionality and migrate them there. E.g., catalog-related operations move from LIBRARY to CATALOG.
  3. Remove all "far-coupled" / redundant indirect associations. E.g., ITEM was far-coupled to LIBRARY, but each item really belongs to a CATALOG, which belongs to a LIBRARY. Migrate ITEM to CATALOG.
  4. Where appropriate, migrate associates to derived classes → a common base class.
  5. Remove transient associations, replacing them with type specifiers to attribute and operation arguments.
