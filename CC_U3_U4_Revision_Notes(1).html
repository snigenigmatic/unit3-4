3.1

Storage — Introduction & Types

Data: A collection of raw facts. Can be Structured (rows & columns, stored in DBMS) or Unstructured (no fixed schema, stored in object stores).

Data is exploding — predicted 572 Zettabytes (1021 bytes) by 2030 and 50,000 ZB by 2050.

Characteristics of Storage Systems

Cost · Speed/Performance · Reliability · Availability · Scalability · Management

Three Fundamental Storage Formats

1. File Storage
  • Data organised as a hierarchy of files in folders.
  • Requires a known path to locate data; limited metadata.
  • Examples: NAS (Network Attached Storage), DAS (Direct Attached Storage).
  • Scaling: typically scale-out (horizontal) because of per-system limits.
2. Block Storage
  • Data chunked into fixed-size, evenly-sized blocks, each with a unique identifier.
  • Storage software reassembles blocks on request.
  • Typically used with SAN environments.
  • Fast, reliable; limited metadata handling — metadata must be managed at the application/DB layer.
  • Best for large transactions and big databases.
3. Object Storage
  • Flat structure; data stored as objects in a single repository (not in folders).
  • Each object = data + unique identifier (for distributed retrieval) + rich metadata.
  • Metadata can include age, privacy/security settings, and application-specific info.
  • Retrieval uses metadata + identifier → better load distribution + richer search.

3.3

RAID & Disk Latency

RAID (Redundant Array of Independent Disks): A data storage virtualisation technology combining multiple physical disks into logical units for redundancy and/or performance.

Data is distributed across drives via "RAID levels" (e.g. RAID 0, 1, 4, 5, 6, 10) depending on required redundancy and performance trade-off.

Magnetic Disk Latency

Reading or writing a block requires three steps:

  1. Seek Time: Disk controller moves the head assembly to the correct cylinder.
  2. Rotational Latency: Waiting for the first sector of the block to rotate under the head.
  3. Transfer Time: All sectors and gaps of the block pass under the head while data is read/written.
Disk Latency = Seek Time + Rotational Latency + Transfer Time
Disk latency formula is a commonly tested numerical/conceptual question.
3.4

Cloud Storage

Cloud Storage: A data storage service model in which data is maintained, managed, and backed up remotely and made available to users over a network (typically the Internet).

The OS analogy: just as an OS translates rich file operations (rename, append, compress…) to fixed-size block reads/writes, a web service translates rich cloud operations (shopping carts, user profiles…) to simple key/value store reads/writes.

What Constitutes Cloud Storage (8 Key Properties)

  1. Dramatic reduction in TCO — Cuts cost >10× vs block/file storage.
  2. Unlimited Scalability — Built on distributed technology; seamlessly add/remove nodes.
  3. Elasticity — Storage virtualisation decouples pool from physical hardware → elastic unified pool.
  4. On-Demand — Pay-as-you-go; pay only for data stored and accessed.
  5. Universal Access — No SAN dependency; flexible number of users and access locations.
  6. Multitenancy — Centralised management, higher utilisation, lower cost.
  7. Data Durability & Availability — Commodity hardware + software layer provides HA even with partial failures.
  8. Usability & Disaster Recovery

Cloud Storage Pros & Cons

Pros
  • Lower cost
  • Easier management
  • Enables sharing and access from anywhere
Cons
  • Loss of control
  • No guarantees of data availability
  • Provider failures

Cloud Storage Architecture (Layered)

  • User Access Layer: Authorised user logs in from anywhere via a public API.
  • Data Service Layer: Handles user demands — storage, space leasing, public resources, multi-user sharing, backup.
  • Data Management Layer: Unified management interface — user management, security, replica management, strategy management.
  • Data Storage Layer: Massive pool of organised storage.

Examples: Amazon S3, Microsoft Azure Storage, OpenStack Swift, EMC Atmos, EMC ECS, Hitachi Content Platform, OceanStore.

3.5

Storage Virtualisation Enablers

Storage virtualisation can be implemented in hardware or software, and at the server, storage device, or network level.

1. File Systems

  • Manages groups of data (files) and their names.
  • Without a file system, a storage medium is one undifferentiated blob.
  • Local File Systems: ext4, NTFS — manage a single machine's storage.
  • Network File Systems (NFS): Makes local files/directories available over LAN; daemon process nfsd on server; client mounts exported directories → transparent access.

2. Logical Volume Manager (LVM)

  • Independent layer between File System and disk drives.
  • Physical disks → Physical Volumes (PV) → grouped into Volume Groups (VG) → split into Logical Volumes (LV).
  • File systems created on logical volumes; can dynamically grow/shrink.

3. Thin Provisioning (Virtual Provisioning)

Thin Provisioning: Capacity-on-demand from a shared storage pool. Logical units presented to hosts have more capacity than physically allocated. Physical storage allocated only when the host actually requires it.
  • Host sees (and uses) a larger virtual pool than actually exists.
  • Provisioning decisions not bound by currently available physical storage.
Thin vs Thick provisioning is a classic exam question. Remember: physical allocation happens lazily (on demand) in thin provisioning.
3.6

Storage Virtualisation — File Level & Block Level

Storage Virtualisation: Abstracting physical storage subsystems from applications; presenting them as logical entities. Hides complexity of storage subsystems, nature of access, network, and physical device changes.

Core capability: aggregates multiple storage devices into pools OR subdivides a resource into many addressable entities.

Category 1: File-Level Virtualisation

  • Provides a standard file-serving protocol interface (NFS / CIFS) to the application.
  • Eliminates dependency between data at file level and physical storage location.
  • Manages files, directories, file systems across multiple servers → single logical file system to the user.
  • Benefits: optimise storage usage, server consolidation, non-disruptive file migrations.
Distributed File Systems (DFS)

File system distributed across multiple servers. Provides location transparency, directory replication, and fault tolerance.

Two metadata management strategies:

a. Centralised Metadata (Lustre)
  • Dedicated metadata server for all metadata operations.
  • Lock-based synchronisation on every read/write.
  • Metadata server can become a bottleneck at high metadata op rates.
  • Scales well for large file workloads.
  • Lustre: Massively parallel, scalable for Linux. Petabyte-scale. Components: Object Storage Servers (OSS) + Object Storage Targets (OST), single Metadata Target (MDT) + Metadata Server (MDS), Lustre Clients (POSIX interface).
  • Lustre flow: client looks up filename on MDS → MDS returns layout → client uses LOV to map offset to OSTs → parallel read/write directly to OSTs.
b. Distributed Metadata (GlusterFS)
  • Metadata distributed across all nodes.
  • Greater complexity; no single metadata bottleneck.
  • GlusterFS: Open-source, no centralised metadata server. Modular, stackable user-space design. Aggregates storage bricks over Infiniband RDMA or TCP/IP.
  • Uses translators (filter programs) for features like replication (AFR translator = Automatic File Replication), load balancing (I/O Scheduler translators).
  • Storage brick = server with DAS or SAN + local FS (ext3/ext4).

Category 2: Block-Level Virtualisation

Virtualises multiple physical disks as a single logical disk. Block addresses can span multiple physical sub-systems.

a. Host-Based (LVM)

Uses LVM to create a storage pool from multiple disks. Allows dynamic shrink/increase of physical volumes.

b. Storage-Device Level

RAID arrays create Logical Units (LUNs) spanning disks in RAID groups. Low latency; host-agnostic. Can virtualise third-party external storage.

c. Network-Based (most common scalable form)

Virtualisation in the network connecting hosts and storage (Fibre Channel SAN).

Switch-Based
  • Virtualisation occurs in an intelligent switch in the fabric.
  • Works with a metadata manager in the network.
  • Example: HP StorageWorks SAN Virtualisation Services Platform.
Appliance-Based
  • I/O flows through an appliance controlling the virtualisation layer.
  • Example: IBM SAN Volume Controller.
  • In-band: All I/O requests and data pass through the virtualisation appliance.
  • Out-of-band: Appliance handles only metadata/control path; data flows directly (agents on each host).

Object Storage — Amazon S3 & OpenStack Swift

Object Storage: Stores data as objects (logical view) by virtualising physical implementation in a flat namespace, eliminating name collisions using REST HTTP APIs.
  • Operations: GET · PUT · DELETE · UPDATE via RESTful APIs.
  • Resources identified by URIs; messages are self-descriptive and stateless (XML).
  • Objects contain additional descriptive properties enabling better indexing.
  • Uses commodity hardware; scale-out distributed systems.
Critical Tasks of an Object Storage System
  • Data placement
  • Automating management tasks: durability and availability (multiple copies / erasure coding, periodic health checks, self-healing, data migration).
Durability Techniques
  • Multiple copies of the object.
  • Erasure Coding (EC): Data broken into fragments, expanded with redundant pieces, stored across different locations. More space-efficient than pure replication.

Amazon S3

Highly reliable, available, scalable, fast cloud storage. REST API with HTTP operations (GET, PUT, DELETE, HEAD).

Data Organising: Buckets, Objects, Keys
  • Objects: Up to 5 TB each; unlimited objects per bucket.
  • Buckets: Logical containers for objects; up to 100 per account.
  • Keys: Path-like names (e.g. resume.doc); used as path in HTTP URL.
  • URL example: http://s3.amazonaws.com/johndoe/resume.doc
  • Objects replicated across multiple geographic locations for resilience to multiple failures.
Security
  • Access Control: Permissions set via AWS Management Console.
  • Audit Logs: S3 logs complete bucket access — AWS account, time, IP, operations.
Data Protection
  • Replication: Default — multiple storage devices; survives 2 replica failures. Reduced Redundancy Storage (RRS) — survives 1 replica failure.
  • Regions: Select geographic region at bucket creation for performance/legal/availability reasons.
  • Versioning: Full history of all objects from time of enablement. Restore to prior version; undo deletes.
  • Multi-part Uploads: Parallelise large object uploads; retry only failed parts.

OpenStack Swift

  • Partition: Core of replication; storage locations for account DBs, container DBs, objects. Distributed across all disks. Adding a node triggers reassignment of partitions.
  • Account: A user in the storage system. Enables multi-user access.
  • Container: Namespace to group objects (conceptually like directories).
  • Object: Actual data stored on disk.
  • Ring: Maps partition space to physical locations on disk. Uses hash for lookup (like an encyclopedia but with hashes instead of letters).

DynamoDB — NoSQL Key-Value & Document DB

DynamoDB: A NoSQL fully managed cloud-based document and key-value database by AWS.

Key Characteristics

  • Schema-less flavour — only key structure and local secondary indexes must be defined upfront.
  • Only item-level consistency (analogous to row-level in RDBMS).
  • No JOIN support between tables — join-like functionality must be in application code.
  • Supports secondary indexes for flexible querying beyond primary key.

Core Components

  • Table: Collection of items.
  • Item: Collection of attributes (analogous to a row).
  • Attribute: Key-value pair.

Primary Key Types

Partition Key (Simple)

Single attribute; value hashed to determine storage partition.

Partition Key + Sort Key (Composite)

Two attributes. Items with same partition key stored together, sorted by sort key value.

Secondary Indexes

Allow querying with alternate keys. Can create one or more per table.

Partitions & Data Distribution

Data stored in partitions backed by SSDs, automatically replicated across multiple Availability Zones within an AWS Region. Partition management handled entirely by DynamoDB.

3.9

Partitioning & Consistent Hashing

Partitioning: Intentionally breaking a large volume of data into smaller partitions, each placed on a single node or distributed across nodes, so that queries/IO can be executed across multiple nodes in parallel.
  • Each record belongs to exactly one partition.
  • Usually combined with replication for fault tolerance.
  • Goal: spread data and query load evenly across nodes.
  • Skew: When some partitions have more data than others.
  • Hot Spot: A partition with disproportionately high load.

4 Approaches to Partitioning

1. Vertical Partitioning (Column Partitioning)

Different columns stored on different data stores/nodes. No two critical columns stored together → improved performance. Example: relational DB (block/file) + object storage.

2. Workload-Driven Partitioning

Partitions formed based on analysed data access patterns from the application. Improves scalability of transactions.

3. Partitioning by Random Assignment

Simplest approach; distributes data evenly. Disadvantage: To read a specific item, must query all nodes in parallel.

4. Horizontal Partitioning (Static)

Rows of data distributed across nodes using one of these techniques:

4.1 Partitioning by Key Range
  • Assigns a continuous range of keys to each partition (min to max).
  • Ranges can be non-uniform (data distribution isn't even).
  • Keys within partition can be kept sorted.
  • Advantage: Efficient range scans.
  • Disadvantage: Can create hot spots (e.g. all writes to timestamp-keyed data go to "today's" partition).
  • Mitigation: prefix the key with another attribute (e.g. sensor name + timestamp).
4.2 Schema-Based Partitioning

Schema designed to minimise distributed transactions. Related rows kept in the same partition.

4.3 Graph Partitioning

Workload-based static partition. Partitions formed by analysing data access patterns. No dynamic re-partitioning (unless using dynamic strategies).

4.4 Partitioning by Hash of Key
  • Hash function applied to partition key → determines which partition stores the record.
  • Evenly distributes rows; avoids hot spots.
  • Disadvantage: Loses ability to do efficient range queries.
  • Workaround: Concatenated/Composite Index — e.g. (user_id, update_timestamp) as primary key → all updates by user stored together sorted by time.

Distributed Hashing

Hash table split across multiple servers to overcome memory limits of a single system. Distribution key: server = hash(key) mod N

Usually paired with a caching layer. Cache miss → compute hash mod N → find server → retrieve data → repopulate cache.

Rehashing Problem

Problem: When a server is added or removed, most keys change their hash mod N → nearly all keys need to be moved. This causes massive cache miss storms.

Consistent Hashing

Consistent Hashing: A distributed hashing scheme that operates independently of the number of servers by assigning servers and keys a position on an abstract hash ring. This allows servers and objects to scale without affecting the overall system.
  • Hash output range mapped to a circle (0° to 360°).
  • Keys hashed → placed on circle.
  • Servers hashed by name → placed on same circle.
  • Each key belongs to the server whose position is closest in the clockwise direction.
  • Adding a server: Only keys between the new server and its predecessor need to move → minimal disruption.
  • Removing a server: Only keys on that server are reassigned to the next clockwise server.
Consistent hashing is heavily tested. Know: the hash ring concept, key assignment rule, and why it minimises key movement on node add/remove.

Partitioning with Secondary Indexes

Secondary indexes don't map neatly to partitions. Two strategies:

Document-Based (Local Index)
  • Each partition maintains its own secondary index covering only its documents.
  • Write: only touch one partition.
  • Read: must query ALL partitions → scatter/gather → expensive reads.
  • Prone to tail latency amplification.
Term-Based (Global Index)
  • Global index covering all partitions; itself partitioned differently from primary key.
  • Read: efficient — query only the partition containing the term.
  • Write: slower/complex — a write may affect multiple index partitions.
  • Index updates often asynchronous → brief inconsistency possible after write.
3.10

Rebalancing Partitions & Request Routing

Rebalancing: Moving load from one node to another in a cluster when nodes are added, removed, or fail.
Requirements of Rebalancing
  • Load (data, reads, writes) should be shared fairly after rebalancing.
  • Database must continue accepting reads and writes during rebalancing.
  • Move no more data than necessary.

Strategies for Rebalancing

1. Hash mod N (Bad!)

When N changes, most keys move to different servers → very expensive. Not recommended.

2. Fixed Number of Partitions
  • Create many more partitions than nodes; assign several partitions per node.
  • Adding a node: steal a few partitions from existing nodes.
  • Only entire partitions move; key-to-partition assignment unchanged.
  • Old assignment used during transfer (no downtime).
  • Challenge: hard to choose the right number if dataset size varies greatly.
3. Dynamic Partitioning
  • For key-range partitioning: if fixed boundaries are wrong, all data could pile in one partition.
  • When a partition grows beyond a configured size → split into two (~50/50).
  • When a partition shrinks below a threshold → merge with adjacent partition.
  • Number of partitions adapts to total data volume.
  • Initial empty DB starts with one partition (all writes hit one node until first split).
  • Works for both key-range and hash-partitioned data.
4. Partitioning Proportional to Nodes
  • Fixed number of partitions per node.
  • When nodes increase → partitions become smaller.
  • New node randomly chooses existing partitions to split; takes ownership of one half of each.
  • Randomisation may create unfair splits but averages out across many partitions.
  • Requires hash-based partitioning.

Request Routing (Service Discovery)

When partitions are rebalanced, clients need to know which node to connect to.

ApproachDescription
1. Any NodeClient contacts any node (round-robin LB). Node handles directly if it owns the partition, else forwards & relays reply.
2. Routing TierAll requests go to a partition-aware routing tier (load balancer) that forwards to the right node.
3. Client-awareClient knows the partition-to-node mapping and connects directly.
4. ZooKeeperNodes register in ZooKeeper. Routing tier/client subscribes to ZooKeeper for authoritative partition-to-node mapping. ZooKeeper notifies on changes.

Users: HBase, SolrCloud, Kafka (ZooKeeper) · Cassandra, Riak (Gossip protocol).

3.11

Replication — Leader-Based, Multi-Leader & Leaderless

Replication: Keeping a copy of the same data on multiple machines connected via a network.

Reasons: reduce latency (geographic closeness) · increase availability (fault tolerance) · increase read throughput (scale out readers).

Each node storing a copy = replica. Challenge: every write must reach every replica.

1. Leader-Based (Single-Leader) Replication

Also known as: leader-follower, master-slave.

  • Leader: Only replica accepting writes. Writes new data to local storage first.
  • Followers (read replicas / slaves): Receive data change via replication log; apply writes in same order as leader. Accept only reads.
Synchronous Replication
  • Leader waits for follower ACK before reporting success to client.
  • Guarantees follower has up-to-date data.
  • Disadvantage: write blocked if follower unavailable.
Asynchronous Replication
  • Leader sends to follower without waiting for response.
  • Better write availability.
  • Disadvantage: follower may lag → stale reads.
Replication Log Implementation Methods
  1. Statement-Based: Leader logs every write SQL statement and sends it. Risk: non-deterministic functions (NOW(), RAND()) produce different results on replicas.
  2. Write-Ahead Log (WAL) Shipping: Append-only byte log sent to followers. Followers build exact same data structures. Tightly coupled to storage engine version.
  3. Change Data Capture (CDC) / Logical Log Replication: Sequence of row-level change records. Engine-independent format; easier to parse for external apps.
  4. Trigger-Based: Custom app code automatically executes on data change; logs change externally. Most flexible but highest overhead.
Handling Failures
  • Follower Failure (Catch-up Recovery): Follower keeps local log of changes received. After restart, requests all changes that occurred during downtime from leader → applies → catches up.
  • Leader Failure (Failover):
    1. Determine leader has failed (often via timeout).
    2. Choose new leader (typically replica with most up-to-date data).
    3. Reconfigure system: clients send writes to new leader; followers follow new leader.
Replication Lag
  • Asynchronous followers may serve stale data → eventual consistency.
  • Eventual consistency: If no new writes, all replicas eventually converge to same value.
  • Replication lag = delay between write on leader and reflection on follower.
Solutions for Replication Lag
  • Read critical data from leader only.
  • Monitor lag; don't query followers with significant lag.
  • Client remembers timestamp of most recent write → read from replicas that reflect at least that timestamp.
  • Monotonic reads: User always reads from the same replica.
  • Consistent prefix reads: Writes seen in the order they occurred.

2. Multi-Leader Replication

Multi-Leader (master/master, active/active): Multiple nodes can accept writes. Each leader acts as follower to other leaders.
Use Cases
  • Multi-datacenter: One leader per datacenter; regular leader-follower within; cross-datacenter leader-to-leader replication.
  • Clients with offline operation: Every device has a local DB acting as leader.
  • Collaborative editing.
Challenge: Write Conflicts
  • Conflict Avoidance: Route all writes for a record through the same leader.
  • Converging to Consistent State: Give each write a unique ID (timestamp, UUID, hash); pick the highest ID as winner (LWW — Last Write Wins).
  • Custom Conflict Resolution Logic: App code runs on write or on read.
Single-Leader vs Multi-Leader Comparison
AspectSingle-LeaderMulti-Leader
PerformanceEvery write must go to datacenter with leader → high latencyEvery write processed locally → inter-datacenter delay hidden from users
Datacenter outageFailover needed — promote follower in another DCEach DC continues independently; replication catches up on recovery
Network problemsVery sensitive to inter-DC link issues (synchronous writes)Tolerates temporary network interruptions (async replication)
Multi-Leader Topologies
  • Circular: Each node forwards to one other.
  • Star: Root node forwards to all others.
  • All-to-all: Messages travel multiple paths → no single point of failure (most common).

3. Leaderless Replication

Leaderless Replication: All nodes are peers; every node accepts reads and writes. A write is successful when acknowledged by a quorum of at least k out of n replicas.
Writes

Client broadcasts write to all replicas. Write considered successful when k out of n ACK.

Reads

Client contacts all replicas; read successful when k out of n agree on a value. Version numbers used to identify newest value.

Read quorum (r) + Write quorum (w) > n → guarantees at least one node in read set has latest write.

Common config: n = 3 or 5, w = r = (n+1)/2
Pitfalls & Fixes
  • Node Down During Write: Missed write → stale reads when it comes back.
    • Read Repair: Client detects stale value during parallel read; writes newest value back.
    • Anti-Entropy Process: Background process constantly checks for differences and syncs.
  • Concurrent Writes (Conflict Detection):
    • Events can arrive in different orders at different nodes due to variable network delays.
    • Last Write Wins (LWW): Most "recent" value wins (risk: clock skew).
    • Happens-Before / Version Vectors: Server maintains version number per key; client must read before write; client merges conflicting values.
Multi-Datacenter with Leaderless

Each write sent to all replicas in all datacenters; client waits only for quorum from local datacenter; cross-datacenter writes are async.

3.12

Consistency Models

Consistency: Refers to the consistency of values in different copies of the same data item in a replicated distributed system.

A Consistency Model is a contract between a distributed data store and processes about rules for data access correctness.

Eventual Consistency

  • If you stop writing, eventually all reads return the same value.
  • Weak guarantee — no bound on when replicas converge.
  • Provides improved performance at the cost of strong guarantees.

Sequential Consistency (Lamport)

  • All processes see the same order of all memory access operations on shared memory.
  • The exact interleaving doesn't matter, but must be consistent across all processes.
  • Not deterministic — multiple executions may produce different orderings.
  • Conceptually: one global memory + a switch connecting an arbitrary processor at any time.

Causal Consistency

  • Two events are causally related if one can influence the other.
  • All processes see causally-related operations in the same correct order.
  • Non-causally-related operations may be seen in different orders by different processes.
  • Relaxes sequential consistency requirements → better concurrency.

PRAM Consistency (Pipelined RAM)

  • All writes by a single process are seen by all other processes in order (as if in a pipeline).
  • Writes by different processes may be seen in different orders.

Strict Consistency (Linearizability / Strong Consistency)

Linearizability: A read always returns the value of the most recent write, regardless of location of processes. Like sequential consistency but execution order must match the actual clock order of issue.
  • System appears as if there is only one copy of the data, all operations atomic.
  • Also called: atomic consistency, immediate consistency, external consistency.
  • Recency guarantee: a read sees the latest written value.
  • After one client's read returns a new value, ALL subsequent reads (on any client) must also return new value.
  • Add CAS (Compare-and-Set): cas(x, v_old, v_new) → ok/error — atomically sets x to v_new if current value equals v_old.

3.13

CAP Theorem

CAP Theorem: It is impossible to simultaneously guarantee all three of Consistency, Availability, and Partition Tolerance in a distributed system with data replication. We can strongly support only two of the three.

The Three Properties

PropertyDefinition
C ConsistencyEvery node returns the same, most recently updated value of a successful write at any logical time. All reads return the latest write.
A AvailabilityEvery read/write request to a working node receives a valid response (not an error). System operational 100% of the time.
P Partition ToleranceSystem continues operating despite network partitions (temporary breaks/failures). Data sufficiently replicated to keep system up through outages.

CAP Trade-offs (The "Pick Two")

ChoiceWhat you getWhat you sacrificeExample
CPConsistency + Partition ToleranceAvailability (may reject requests if can't reach majority)MongoDB, HBase, ZooKeeper
APAvailability + Partition ToleranceConsistency (may return stale data)Cassandra, CouchDB, DynamoDB
CAConsistency + AvailabilityPartition Tolerance (not viable in real distributed systems)Single-node RDBMS (PostgreSQL)
Important: Since network partitions (P) are unavoidable in cloud environments, systems effectively choose between CP and AP.

Nuances & Criticisms

  • CAP does not specify an upper bound on response time → latency matters in practice.
  • The choice is between C and A only when a partition occurs. When no partition, both can be satisfied.
  • CAP suggests a binary decision; in reality it is a continuum (different degrees of consistency).
  • Cassandra supports AP but provides eventual consistency — allows writes to any node, reconciles inconsistencies ASAP.

CAP & Cloud/Microservices

Understanding CAP helps choose the right database for microservices:

  • Need fast iteration + horizontal scaling + can tolerate eventual consistency → Cassandra / CouchDB (AP)
  • Need strong data consistency (e-Commerce, payments) → PostgreSQL (CA/CP)
Be able to classify given databases into CP/AP/CA and explain the trade-off in context. Also know real-world examples from the slides.
3.14

Distributed Transactions & Two-Phase Commit (2PC)

Transaction: An operation composed of discrete steps — all must complete for commit, else the system reverts to its pre-transaction state.

ACID Properties

PropertyMeaning
AtomicAll-or-nothing. Others don't see intermediate results.
ConsistentSystem invariants hold before and after transaction (e.g. total money in accounts unchanged after transfer).
Isolated (Serialisable)Concurrent transactions produce same final result as some serial execution.
DurableOnce committed, results are permanent. No post-commit failure reverts them.

Nested Transactions

Top-level transaction may create sub-transactions. Sub-transactions may commit (results durable) but parent transaction may abort. Solution: Private Workspace — each sub-transaction gets a private copy; on commit, private copy displaces parent's copy.

Distributed Transactions

Distributed Transaction: A set of operations on data performed across two or more data repositories across different systems. Either all complete successfully, or none do.
Architecture
  • Multiple data repositories on different nodes connected by a network.
  • Transaction Manager (per node): Maintains recovery log; participates in concurrent execution; handles sub-transaction prepare/commit/abort.
  • Transaction Coordinator (across nodes): Starts and distributes sub-transactions; coordinates termination (commit at all sites or abort at all sites).

Two-Phase Commit Protocol (2PC)

2PC: Ensures atomicity across sites — a transaction either commits at ALL sites or aborts at ALL sites.
Phase 1 — Prepare (Voting)
  1. Coordinator writes prepare T to its log.
  2. Coordinator sends prepare T message to all participant nodes.
  3. Each node prepares its component of T.
  4. If node is ready to commit → enters pre-committed state, writes Ready T to local log (flushes to disk), sends Ready T to coordinator. Holds locks.
  5. If node cannot commit → writes Don't Commit T, sends Don't commit T to coordinator.
Phase 2 — Commit / Abort
  1. If coordinator received Ready T from ALL → writes Commit T to log, sends commit T to all sites.
  2. If coordinator received Don't commit T from any → writes Abort T to log, sends abort T to all sites.
  3. On receiving commit T: site commits, releases locks, writes Commit T to local log, sends Done.
  4. On receiving abort T: site rolls back all changes, releases locks, writes Abort T, sends Done.
  5. Coordinator receives all Done messages → clears all state.
Draw and explain the 2PC message flow. Know what happens if the coordinator fails during Phase 2 (participants are blocked waiting).

P2P vs Master-Slave Distributed System Models

Cloud infrastructure is structured as distributed clustered systems. Two main architectural styles:

Peer-to-Peer (P2P)
  • Every node acts as both client and server.
  • No hierarchy; no single point of failure.
  • Self-organising; distributed control — no node has global view.
  • Scalable, resilient, highly decentralised.
  • Limitations: difficult to manage; harder to secure; limited ability to guarantee consistency/linearizability.
  • Real-world example: Amazon Aurora Multi-Master (multiple DB instances with read/write capacity).
Client-Server / Master-Slave
  • Asymmetric roles: servers offer services, clients consume.
  • Hierarchy present → vulnerable to SPOF.
  • Master acts as central coordinator → easy decision making.
  • Cannot scale out indefinitely (master is performance bottleneck).
  • Real-world example: MapReduce / Hadoop — single master assigns work, monitors jobs, redistributes on worker failure.

Cloud Controller: Control Path vs Data Path

Earlier units focused on technologies, programming models, deployment models, application architectures, and data paths. Once requests start reaching the physical resources, those requests have to be managed: that is the control path. Unit 4 is about the control path - how distributed clusters in cloud data centers are organised, how they tolerate unreliable communication and faults, how they reach consensus, elect leaders, schedule resources, lock shared state, and coordinate via systems like ZooKeeper.

Eucalyptus (Cloud Controller Example)

Eucalyptus is one concrete instance of a cloud control plane.

  • Open-source suite for private/hybrid cloud, AWS EC2/S3 compatible.
  • UFS (User Facing Services): Implements AWS-compatible web service APIs; accepts requests from CLI clients or GUIs.
  • Cloud Controller (CLC): High-level resource tracking and management; only one CLC can exist per Eucalyptus cloud.
  • Cluster Controller (CC): Manages node controllers; deploys and manages VM instances.
  • Node Controller (NC): Runs on the host machine that hosts VMs; interacts with both the OS and hypervisors to maintain the lifecycle of instances on each node.
  • Storage Controller (SC): Interfaces with various storage systems.
Cloud Controller Module: The Key Interface

The CLC is the key interface for users and administrators. It queries the different nodes, makes broad decisions about the virtualization setup, and executes those decisions through cluster controllers and node controllers.

Concretely, the Cloud Controller is a set of web services that provides:

  1. System-wide arbitration of resource allocation.
  2. Governing of persistent user and system data.
  3. Authentication and protocol translation for user-visible interfaces.

The CLC maintains a System Resource State populated with data from the CC and NCs.

  1. A user request arrives at the CLC.
  2. The SRS is consulted to make an admission control decision based on service-level expectation.
  3. To create a VM, resources are first reserved in the SRS and the request is sent downstream to CC and NC.
  4. Once the request is satisfied, the resource is committed in the SRS. On failure, the reservation is rolled back.

Unreliable Communication & Fault Types

What Can Go Wrong When You Send a Request

  1. Request lost (cable unplugged).
  2. Request queued, delivered later (network/recipient overloaded).
  3. Remote node failed (crash or powered down).
  4. Remote node temporarily stopped responding (will resume later).
  5. Remote node processed request but response lost on network.
  6. Remote node processed request but response delayed.
Key insight: You cannot distinguish cases 1–6 without additional mechanisms. This non-determinism + partial failure is what makes distributed systems hard.

Reliability

Probability that a system meets performance standards and yields correct output for a specific time under expected load/data volume. High reliability is critical for cloud-hosted applications (failure = loss of business or lives).

Fault Types

Fault TypeDescription
TransientAppears once, then disappears.
IntermittentOccurs, vanishes, reappears with no real pattern. (Worst form of fault.)
PermanentOnce it occurs, only replacement/repair of faulty component fixes it.
MTBF = Total Uptime / Number of Breakdowns

Fault Model Classes

ClassBehaviour
Crash-stopNode stops responding permanently (gone forever).
Crash-recoveryNode may crash and restart; stable storage preserved; in-memory state lost.
Byzantine (arbitrary)Node exhibits arbitrary/malicious behaviour.

Timeouts & Unbounded Delays

  • Asynchronous networks have no upper bound on packet delivery time.
  • No single "correct" timeout value — must be determined experimentally.
  • Measure distribution of round-trip times; use observed response time to auto-adjust timeouts (consider jitter).

Network Congestion & Queueing

  • Packet may wait in a queue on a busy network link (network congestion).
  • If switch queue fills → packet drop → need retransmission.
  • In multi-tenant cloud: shared network links → variable delays based on other tenants' workloads (noisy neighbour effect).
  • Batch workloads (MapReduce) can easily saturate network links.

Failure Detection: Heartbeats & Probing

  • Heartbeats: Application periodically sends signal. If not received for N consecutive intervals → declare failed.
  • Probing: Monitoring service periodically sends lightweight request. No response → declare failed.
  • GFS example: Master communicates with each chunkserver via heartbeat messages. On failure: master begins replicating its chunks elsewhere. Heartbeat messages also used to send additional configuration.
4.3

Fault Tolerance & System Availability

Fault Tolerance: A system's ability to continue operating uninterrupted despite the failure of one or more components.

Availability Metrics

  • Uptime: Percentage of time the system is running and functional (service provider's view). E.g. 99.999% = "five nines".
  • Downtime: Duration the system is unavailable due to unplanned failures or planned maintenance.
  • Single Point of Failure (SPOF): Any failure that brings down the entire system. Rule: design with no SPOF through redundancy.
Cloud Provider Availability Measurement
  • Microsoft: Time service is up / total time.
  • Gmail: % of successful interactive user requests; consecutive minutes of downtime.
  • AWS: Error rate — average % of requests resulting in errors in 5-minute intervals.

Two Types of Fault Tolerance

  • Fail-Safe: Safety state preserved but performance may drop (e.g. traffic lights go red in all directions on failure).
  • Graceful Degradation: Application continues in reduced capacity mode.

Approaches for Building Fault Tolerance

  1. Redundancy: Eliminate SPOF — redundant power, processors, memory, disks, software components.
  2. Reliability: Dependability of components; analysed via failure logs, frequency of failures.
  3. Availability: Measured by MTBF and MTTR.
    MTTF = average time to first failure | MTTR = average time to repair
  4. Recoverability: Ability to overcome momentary failure with no end-user impact. Techniques: retries of disk/network I/O, ACK mechanisms (TCP), Error Control Coding (ECC — parity, Hamming code, CRC), Checkpointing/Rollback.

Strategies for Dealing with Partial Failures

  • Async communication: Avoid long chains of synchronous HTTP calls; use event-driven architecture.
  • Retries with Exponential Backoff: Retry with increasing wait intervals; avoids overwhelming a temporarily overloaded service.
  • Timeouts: Clients should never block indefinitely; use connection/request timeouts.
  • Circuit Breaker: Track failed request count. If error rate exceeds threshold → "trip" breaker → fail fast for configured timeout → try again → if successful, close breaker.
  • Fallbacks: Return cached data or default value when a request fails.
  • Bulkhead Isolation: Separate process/thread pools per function so failures in one don't cascade.
  • Queue: Persist requests until dependency is available.
  • Cache: Serve from cache during downstream service failure.

Failover Architectures

Active-Active (Symmetric)
  • Multiple servers run identical services simultaneously.
  • On failure, surviving server(s) absorb the load.
  • Also called Continuous Availability.
  • Better utilisation; no idle standby.
Active-Passive (Asymmetric)
  • Primary server active; dedicated redundant standby does nothing normally.
  • On failure: standby takes over; primary restarted and relocated.
  • Failback: process of restoring to primary after recovery.
  • Client must reconnect to new active node.
VMware FT (Fault Tolerance)
  • Uses a Backup and Primary VM configuration.
  • Leverages virtual lock-step — both VMs execute all operations.
  • Only Primary VM sends outputs to clients.
  • On Primary VM failure → Backup VM takes over.
  • After Primary VM recovers → acts as new Backup VM.

Consensus & Paxos Algorithm

Consensus: All non-faulty processes in a distributed system collectively agreeing on the same output value.

A replicated state machine starts with the same initial value. For each transition, processes decide on the next value. Consensus ensures all computers reach the same state.

Consensus Problem — Formal Definition

N nodes, each with input xₚ (0 or 1) and output yₚ (unchangeable once set).

PropertyMeaning
AgreementAll non-faulty nodes decide on the same output value.
TerminationAll non-faulty nodes eventually decide on some value.
ValidityAny decided value is a value that was proposed.
IntegrityA node decides at most once.
Non-TrivialityAt least one initial state leads to all-0s or all-1s outcome.

Generic Consensus Algorithm Steps

  1. Elect: Processes elect a single leader to propose the next valid output value.
  2. Vote: Non-faulty processes validate the proposed value and propose it as next valid value.
  3. Decide: If a threshold of identical votes is received → decide. Otherwise, restart.

Problems Equivalent to Consensus

  • Reliable Multicast: All servers receive same updates in same order despite failures.
  • Membership Failure Detection: All processes know which members are alive/failed.
  • Leader Election: Elect a single leader known to all group members.
  • Mutual Exclusion: Ensure mutually exclusive access to critical shared resources.

Consensus in Synchronous vs Asynchronous Systems

Synchronous

Messages received within a bounded time; clock drifts bounded. Consensus IS possible — nodes take turns proposing; skip any that don't respond within max time limit. (E.g. Cray supercomputer, multi-core machine.)

Asynchronous

No bounds on execution time, clock drift, or message delay (e.g. the Internet). Consensus is IMPOSSIBLE to solve deterministically. Fischer-Lynch-Paterson (FLP) impossibility. Probabilistic solutions are used in practice.

Paxos Consensus Algorithm

Family of distributed algorithms for consensus. Used by Google, Amazon.

Three Roles in Paxos
  • Proposers (Leaders/Coordinators): Propose values for agreement.
  • Acceptors: Listen to requests from proposers; respond with values.
  • Learners: Learn the final decided values.

Notes: Nodes can take multiple roles. Nodes must know the number of majority acceptors (quorum). A Paxos run aims at a single consensus.

Paxos Algorithm Steps
  1. Proposer sends prepare(n, v) to all acceptors (n = proposal number, v = proposed value).
  2. Each acceptor compares n to highest proposal number it has responded to (nv). If n > nv → responds with ack(nv, v') where v' is value of highest-numbered accepted proposal.
  3. Proposer waits for ACK from majority of acceptors. Sends accept(n, max(v, v' from all acks)) to all acceptors.
  4. On receiving accept(n, v), acceptor accepts v unless it already responded to a higher-numbered prepare. If majority accept → v is the decision value.
Trace a Paxos execution with two proposers. Know the prepare/accept message cycle and why the proposer takes the value of the highest accepted previous proposal.

Leader Election — Ring & Bully Algorithms

Leader Election: Designating a single node as the organiser/coordinator of tasks in a distributed system, especially when the current leader fails.

Election System Model

  • Any process can call an election.
  • Multiple processes can call an election simultaneously → must yield ONE leader.
  • Result must not depend on which process initiated.
  • Liveness: Every node will eventually enter elected or not-elected state.
  • Safety: Only one node enters the elected state.
  • Winner: non-faulty process with best attribute value (highest CPU, most disk, highest priority, etc.).

Ring Election Algorithm

Nodes organised in a logical ring. All messages sent clockwise.

  1. Process pᵢ discovering coordinator failure initiates an election message containing its own id:attr.
  2. Any receiving process pᵢ compares the attr in message with its own:
    • Arrived attr > own attr → forward the message unchanged.
    • Arrived attr < own attr AND pᵢ hasn't forwarded yet → overwrite with own id:attr, forward.
    • Arrived id:attr == own id:attr → pᵢ has highest attr → declares itself new coordinator; sends elected message.
  3. On receiving elected message: set local elected = id in message, forward unless you are the new coordinator.
Worst-case message complexity: 3(N-1) · Turnaround time: 3(N-1)
Modified Ring Election (handles failures during election)
  • Message circulated, bypassing failed processes.
  • Each process appends its id:attr to the message (instead of overwriting).
  • Initiator receives full list → elects process with best attribute.
  • Sends coordinator message; each process adds id and records coordinator locally.
  • If would-be coordinator's id not in final list → algorithm repeats.

Bully Algorithm

Bully Algorithm: Every process knows all other processes and their IDs. The process with the next-highest ID (after the failed leader) that detects the failure elects itself — it "bullies" lower ID processes.
Rules
  • If a process knows its ID is highest → elects itself; sends Coordinator to all lower ID processes. Done.
  • Otherwise → sends Election message to all processes with higher ID.
    • If no answer within timeout → declares itself leader; sends Coordinator to all lower IDs.
    • If answer received → wait for Coordinator message. If none after second timeout → restart election.
  • Process receiving Election message → replies with disagree/OK → starts own election protocol.
Timeouts
  • First timeout (waiting for first response): ≈ 2T + processing time where T = one-way transmission time.
  • Second timeout (waiting for Coordinator after disagree): use worst-case turnaround time.
Bully Algorithm Complexity
  • Worst case (lowest ID process detects failure): O(N²) messages; turnaround ≈ 5 message transmission times.
  • Best case (second-highest ID detects failure): N-2 coordinator messages; 1 transmission time turnaround.
Practice: 7 processes, P7 fails, P4 detects first. Count election messages and response messages. Answer: 6 election messages (3 by P4, 2 by P5, 1 by P6) + 3 response messages.

Resource Management & Scheduling

Scheduling: Deciding how to allocate system resources (CPU, memory, storage, network) between users and tasks. Involves policies (principles guiding decisions) and mechanisms (means to implement policies).

Cloud Resource Management Challenges

  • Multi-objective optimisation — policy decisions are difficult.
  • Impossible to have accurate global state information.
  • System failures and attacks make the system unpredictable.
  • Large fluctuating loads challenge cloud elasticity claims.
  • Resource management strategies differ for IaaS, PaaS, SaaS.

Cloud Resource Management (CRM) Policies

  1. Admission Control: Prevent system from accepting workload violating high-level policies.
  2. Capacity Allocation: Allocate resources based on need — different fixed/variable sizes.
  3. Optimisation & Load Balancing: Distribute workload evenly among servers.
  4. Energy Optimisation: Minimise energy consumption via component/operating mode choice.
  5. QoS Guarantees: Satisfy conditions specified in SLA.
  6. Locale-Based: Location-based legal compliance, performance.
  7. Cost: Based on perceived value or auctions.

Mechanisms for CRM Policy Implementation

  • Control Theory: Feedback-based (utilisation, latencies, scheduling time) to control resource allocation → guarantee stability and predict transient behaviour.
  • Utility-Based: Performance model + correlation of user-level performance with cost; uses SLA rewards/penalties.
  • Machine Learning: Uses historical data to predict resource requirements.
  • Market-Oriented / Economic: Combinatorial auctions; users pay for resources.

Cloud Scheduling

  • Server shared among VMs → VMs share among applications → applications share among threads.
  • Scheduler must be: efficient, fair, starvation-free.
Scheduling Objectives
  • Batch systems: maximise throughput, minimise turnaround time.
  • Real-time systems: meet deadlines, be predictable.
Common Algorithms (Best-Effort)

Workload Partition Rules

  • Optimal Partitioning Rule (OPR): Partitions workload so all tasks complete at the same time (earliest possible completion).
  • Equal Partitioning Rule (EPR): Assigns equal workload to all worker nodes.

Distributed Locking

Distributed Lock: A mutual exclusion lock with global presence. Ensures only one client (across different machines) holds a lock at any given moment.

Distributed systems rely on quorum decisions because a single node cannot trust its own judgment alone.

Why Use Distributed Locks?

  1. Efficiency: Prevent multiple nodes performing the same expensive work (e.g. triggering a timer twice).
  2. Correctness: Prevent concurrent processes from corrupting shared data, causing data loss or inconsistency.

Features of Distributed Locks

  • Mutual Exclusion: Only one client globally holds a lock at a given moment.
  • Deadlock-Free: Lease-based mechanism — if client acquires lock and crashes, lock is auto-released after expiry period.
  • Consistency: Lock must remain the same after failovers (replica takeover).

Types of Locks

Optimistic Lock

Don't block — continue anyway, check if someone else modified the record before committing. If modified → abort and retry. Good for low-conflict scenarios.

Pessimistic Lock

Take exclusive lock BEFORE operating → no one else can modify. Release after operation. Good for high-conflict scenarios.

Problem: GC Pause + Expired Lock

Issue: A client acquires lock. Garbage Collector pauses the client process. Lock lease expires. Lock is granted to another client. First client resumes and makes unsafe changes — stale lock!

Solution: Fencing Tokens

  • Lock server returns a monotonically increasing fencing token each time a lock is granted.
  • Client includes fencing token in every write request to storage.
  • Storage service rejects writes with a token lower than the highest token already seen.
  • Ensures stale lock holders cannot modify resources even after their lease expires.

Distributed Lock Manager (DLM) Implementations

  • Google Chubby: Lock service for loosely coupled distributed systems.
  • Apache ZooKeeper: Open-source distributed lock service (ephemeral sequential znodes).
  • Redis (Redlock Algorithm): Advanced key-value store with distributed lock support.

Apache ZooKeeper

ZooKeeper: An open-source, high-performance coordination service for distributed applications. Provides centralised coordination including naming, configuration management, locks & synchronisation, and group services via a shared hierarchical namespace (znodes).

Why ZooKeeper?

  • Eliminates need to build synchronisation services from scratch.
  • Provides reliability and availability via ensemble (cluster) of nodes — no SPOF.
  • Ensures atomicity — transactions either succeed or fail completely.
  • Maintains standard hierarchical namespace (like files and directories).

ZooKeeper Objectives

Simple · Robust · High Availability · High Throughput · Low Latency · Wait-free (slow client doesn't block fast client) · Tuned for read-dominant workloads.

Architecture Components

ComponentRole
Server (ZooKeeper node)Stores data tree in memory; sends ACK when client connects. Leader elected at startup.
LeaderHandles all write requests; provides info to clients; performs automatic recovery.
FollowerHandles read requests locally; forwards writes to leader.
ClientOne of the nodes in distributed app; sends regular heartbeats to server. Auto-redirects to another server if connected server unavailable.
Ensemble/ClusterGroup of ZooKeeper servers for HA.

Data Model — Znodes

  • Hierarchical namespace like a file system (slash-separated, absolute paths).
  • Each znode stores: data (configurations, status, counters, location info) + optional children.
  • Designed for coordination data only — small amounts; NOT big datasets.
  • Entire ZooKeeper tree kept in main memory for performance.
  • Only transaction logs and snapshots stored persistently.
Types of Znodes
TypeDescription
PersistentPermanent; must be explicitly deleted by client; survives session termination.
EphemeralTemporary; deleted automatically when creating session ends. Cannot have children. Used to detect client termination.
Sequence NodesAppend monotonically increasing counter to path. Applies to both persistent and ephemeral nodes.

ZooKeeper Operations

  • create · delete · exists · get data · set data · get children · sync

Reads, Writes, and Watches

  • Reads: Served locally by the connected ZooKeeper server.
  • Writes: Forwarded to leader → goes through majority consensus (Zab protocol) → response.
  • Watches: Set on a znode to monitor changes. When the znode changes, client receives a watch event notification.

Implementation

  • Request Processor
  • Atomic Broadcast (Zab Protocol): Ensures all servers see all transactions in the same order.
  • Replicated Database: In-memory + persistent logs/snapshots.
  • Each update assigned a unique zxid (ZooKeeper Transaction ID).
  • Stat structure on each znode: version numbers for data changes, ACL changes, timestamps.
  • setData and delete require client to supply correct version → optimistic concurrency control.

ZooKeeper Recipes (Use Cases)

  • Naming Service: Map names to info (which servers/services are up, look up status).
  • Distributed Locks: Clients create ephemeral sequential znode under /Cluster/_locknode. Least ID holds the lock; others watch and acquire when lock released.
  • Data Synchronisation: Producer-consumer queues, barriers.
  • Cluster/Config Management: Centralised config; new nodes pick up latest config on join.
  • Leader Election: Built-in support via ephemeral znodes.
  • Message Queue: Znode designated as queue; producers add children znodes; consumers get & delete child znodes.

Advantages & Disadvantages

Advantages
  • Simple distributed coordination.
  • Synchronisation (mutual exclusion, co-operation).
  • Ordered messages; serialisation.
  • Reliability; atomicity of data transfers.
Disadvantages
  • No built-in service discovery / health checks (Consul has these).
  • Ephemeral node lifecycle tied to TCP; process may die without closing TCP.
  • Complex to set up and operate.

ZooKeeper users: Yahoo!, Zynga, Rackspace, LinkedIn, Netflix. Projects: Apache YARN/MapReduce, HBase, Kafka, Storm, Neo4j.

Reverse Proxies, Forward Proxies & Nginx

Reverse Proxy

Server sitting in front of web servers, forwarding client requests to them. Resources returned appear to originate from the proxy. Owned/managed by the web service.

Flow: Client → Proxy (checks validity, checks cache) → Content server (if cache miss) → Proxy caches → Client.

Forward Proxy

Server sitting in front of a group of client machines. Clients make requests; proxy communicates with web servers on their behalf.

Uses: Bypass state/institutional restrictions; content filtering; identity protection.

Benefits of Reverse Proxy

  • Security: Backend servers hidden from internet; DDoS protection (block/blacklist IPs, limit connections); zero-trust architecture support.
  • Scalability & Flexibility: Load balancing across multiple servers; no single point of failure; backend infrastructure can change without client knowledge.
  • Web Acceleration: Faster response via caching, compression, SSL offload.
  • Caching: Static content (images, videos) cached at proxy → reduced load on backend.
  • Identity Branding: Conceal external service URLs (e.g. payment processor) from users.

Nginx

Nginx: Open-source web server with capabilities for reverse proxying, caching, load balancing, and media streaming. Can also function as a proxy for email (IMAP, POP3, SMTP) and a reverse proxy/load balancer for HTTP/2, TCP, UDP.
Nginx Architecture
  • Follows master-slave architecture with event-driven, asynchronous, non-blocking model.
  • Master: Allocates jobs to workers; doesn't wait for worker responses; routes responses back to client.

  • Load balancing: evenly distributes requests to upstream servers; redundancy on server failure.

  • TLS/SSL: secure HTTPS communication.

Nginx can handle >10,000 simultaneous connections with low CPU/memory under heavy load.

Multitenancy & Multitenant Databases

Multitenancy: An architecture in which a single instance of a software application serves multiple customers (tenants). A tenant is a group of users sharing common access with specific privileges to the software instance.

Benefits of Multitenancy

  • Better resource use: One machine shared among multiple tenants → maximise utilisation.
  • Lower costs: Shared infrastructure → cloud vendor can serve many at lower cost.

4 Levels of Multitenancy

  1. Ad Hoc/Custom Instances: Each customer has their own custom software version. Lowest level; highest management complexity.
  2. Configurable Instances: All customers share same software version; customisation via configuration (logos, workflows). Significant management savings.
  3. Configurable Multi-Tenant Efficient Instances: Single running instance shared among all customers. Additional efficiency.
  4. Scalable, Configurable, Multi-Tenant Efficient Instances: Level 3 + hosted on a cluster → scales from small to very large customer bases.

Authentication in Multi-Tenant Systems

Centralised Authentication

All authentication via centralised user database. Cloud admin gives tenant admin rights to manage user accounts. Users authenticated against centralised DB.

Decentralised Authentication

Each tenant maintains own user database. Tenant deploys a federation service interfacing between tenant's auth framework and cloud auth service.

Access Control

  • Roles: Set of permissions included in role (e.g. storage administrator role = permission to define storage devices).
  • Business Rules: Fine-grained policies dependent on operation parameters (e.g. limit transfer amount, allow only during business hours). Implemented via policy engines (Drools Expert, Drools Guvnor).
  • ACL (Access Control List): Each object has a list of permissions per role.
  • Capability-Based: Like a house key — if you hold a reference to an object, you have access.

Resource Sharing Methods

Storage — Dedicated Table Method

Each tenant stores data in a separate set of tables. Access restricted via SQL GRANT: GRANT SELECT,... ON FriendlyTable TO FriendlyGarage WITH GRANT OPTION.

Storage — Shared Table Method

All tenants' data in same table; rows prefixed with customer ID. Space-efficient; requires multiple SELECT statements + auxiliary metadata table.

Customisation Methods
  • Pre-allocated Columns: Space reserved for custom columns. Risk: wasted space if unused.
  • Name-Value Pairs: Extra column pointing to name-value pair table. Space-efficient; requires data reconstruction before joins.
  • XML Method: Final column is XML document; arbitrary structure per record.

Failure Detection, Checkpointing & Application Recovery

Two support techniques for high availability:

  1. Failure Detection: Cloud infrastructure detects failed instances; avoids routing requests to them.
  2. Application Recovery: Failed instances are restarted.

Failure Monitoring Techniques

Push Model (Heartbeats)

Process q sends heartbeat to monitoring process p every t seconds. If p doesn't receive heartbeat after timeout T → suspects q failed.

Pull Model (Probing)

Process p sends "are you alive?" to q every t seconds; q responds "I am alive". If q doesn't respond after timeout T → suspects failure.

Dual Scheme (Hybrid)

First phase: processes assumed to use push model (send liveness messages). After some time: p assumes processes that didn't send liveness use pull model. Combines efficiency of both.

Speed vs Accuracy Trade-off
  • Low threshold → detect failures faster → more false positives (momentary overload flagged as failure).
  • High threshold → fewer false positives → slower failure detection.
  • Generally, high threshold preferred because consequences of falsely declaring failure are severe.

Failure Redirection

After identifying failed instances, avoid routing new requests to them. Common mechanism: HTTP Redirection — server returns 3xx code + new URL pointing to a live server.

Application Recovery — Checkpoint/Restart

  • Cloud infrastructure periodically saves application state (checkpoint).
  • On failure: activate most recent checkpoint; resume from that state.
  • Distributed Checkpoint/Restart: All processes of distributed app instances checkpointed; all instances restarted from a common checkpoint if any instance fails.
  • Challenges: scalability limitations; correctness issues if inter-process communication data was in-transit at checkpoint time.

Cloud Security Requirements

Cloud Security: A set of control-based safeguards and technology protection designed to protect resources stored online from leakage, theft, or data loss. Protection encompasses cloud infrastructure, applications, and data.

Basic objectives: Confidentiality · Integrity · Availability. Additional: Cost-effectiveness · Reliability & Performance.

Physical Security Requirements

  • Central monitoring and control center with dedicated staff.
  • Monitoring for each possible physical threat (intrusion, floods, etc.).
  • Staff training in response to threat situations.
  • Manual or automated backup systems (e.g. pumps for flood containment).
  • Secure access to facility.

Virtual Security Best Practices

1. Cloud Time Service

All systems synchronised to same clock using NTP (Network Time Protocol). Critical for correlating events across geographically distributed systems. Protocol messages can be encrypted to protect against false reference sources.

2. Identity Management
  • Must scale to number of users in cloud system.
  • Federated identity management for heterogeneous cloud systems (single identity, single sign-on across multiple system types).
  • Must satisfy applicable legal and policy requirements (e.g. user deletion within specified period).
  • Maintain historical records for future investigation.
3. Access Management
  • Allow only authorised users access to cloud facilities.
  • No unrestricted access even for cloud management personnel.
  • Multi-factor authentication for sensitive operations.
  • Disallow shared accounts (e.g. 'admin').
  • Whitelist IP addresses for remote administrative actions.
4. Break-Glass Procedures

Alarm-triggered procedures that bypass normal security controls in emergencies (analogy: breaking glass to pull fire alarm). Must be executable only in emergencies under controlled conditions; always triggers an alarm.

5. Key Management

Cloud infrastructure needs secure facilities for generation, assignment, revocation, and archiving of cryptographic keys. Procedures for recovering from compromised keys.

6. Auditing

Captures all security-related events with: time, system, userid. Centrally maintained, secure log. Can be sanitised for sharing with cloud customers during log analysis.

7. Security Monitoring

Infrastructure for alerts on critical security events. Cloud-wide intrusion and anomaly detection system (on both network and host nodes).

8. Security Testing

All software tested in isolated test bed before deployment. Patches tested before production. Ongoing security testing to identify vulnerabilities. Remediation process for identified vulnerabilities. Some tests may be carried out by third parties.

Risk Management & Security Design Patterns

Risk Management: Process for evaluating risks, deciding how they are to be controlled, and monitoring their operation.

Risk Management Process

  1. Information Resource Categorisation: Evaluate each resource by Criticality (business impact of security failure) and Sensitivity (confidentiality). Determines required security level.
  2. Select Security Controls: Choose controls appropriate to criticality/sensitivity (e.g. IP whitelisting for admin access).
  3. Risk Assessment: Determine if selected controls provide adequate protection; augment if needed.
  4. Implement Security Controls: Administrative, technical, or physical controls implemented.
  5. Operational Monitoring: Continuously monitor effectiveness of controls.
  6. Periodic Review: Periodically review controls — new threats may emerge; operational changes (new software) may require redesign.
NIST Security Control Categories

Technical · Operational · Management. Each category has families (18 total). E.g. Audit and Accountability family → Response to Audit Processing Failures control.

System Impact Levels (FIPS 200)
  • Low-impact: Limited degradation in capability; primary functions still performed.
  • Moderate-impact: Significant degradation; system still performs primary functions.
  • High-impact: System incapable of performing some primary functions.

Security Design Patterns

1. Defense in Depth

Defenses layered so attacker must overcome multiple barriers. Example: remote admin access only through VPN + IP whitelist + one-time password. Any one layer broken → still two more.

2. Honeypots

Decoy computer system that appears attractive to attackers. While attacker attacks honeypot, security personnel observe and attempt to trap the attack. Can be deployed by both cloud provider and cloud customer. In cloud context: honeypot VM monitors suspicious VM break-in attempts.

3. Sandboxes

Software executed in a restricted environment inside the OS. Attacker who breaks into software does not get unrestricted OS access → limits damage. Also provides defense in depth. Example: browser sandboxes, container isolation.

4. Network Patterns
  • VM Isolation: Encrypt traffic between VMs sharing physical hardware; tighten security controls (port restrictions).
  • Subnet Isolation: Physically separate traffic for administrative, customer, and storage networks. Routing between networks via firewalls.
5. Common Management Database (CMDB)

Database containing inventory of all IT components, their current configuration and status. Ensures all administrative components have a single consistent view → simplifies security implementation and management.

Security Design Example — PaaS System

  • Two network entry points: control network (administration) and public access.
  • Control network: Whitelisted IP addresses + multi-factor auth.
  • Public network: Two redundant switches for availability.
  • Internal separation: Management network physically separated from public network.
  • DBMS security: Disallow unneeded ports; intrusion detection system on DBMS server; check ODBC connections.
  • Security Server: Performs auditing, monitoring, security operations center, security scanning.

Cloud Threats — DoS, DDoS & EDoS

Denial of Service (DoS)

DoS Attack: A cyber attack where a malicious actor renders a computer unavailable to intended users by interrupting its normal functioning through overwhelming/flooding with requests. Characterised by using a single computer.
DoS Attack Categories
Buffer Overflow Attack

Memory buffer overflow causes machine to consume all available disk space, memory, or CPU time. Results in sluggish behaviour, system crashes.

Flood Attack

Saturate targeted server with overwhelming number of packets. Requires attacker to have more bandwidth than target.

Indicators of DoS Attack
  • Typically slow network performance (long load times).
  • Inability to load a particular website.
  • Sudden loss of connectivity across devices on the same network.

Distributed DoS (DDoS)

DDoS Attack: A malicious attempt to disrupt normal traffic of a server/service/network by overwhelming the target using multiple compromised computer systems (bots/zombies forming a botnet).
  • Bots = internet-connected devices infected with malware; controlled remotely by attacker.
  • Botnet = group of bots; attacker sends remote instructions to each bot.
  • Each bot is a legitimate internet device → difficult to separate attack from normal traffic.
  • Analogy: an unexpected traffic jam preventing regular traffic from reaching its destination.

Economic Denial of Sustainability (EDoS)

EDoS Attack: Attacker deliberately exploits cloud auto-scaling to increase resource utilisation of a target cloud user, resulting in huge resource costs — a financial attack rather than a service disruption attack.

Cloud Security Defense Strategies Overview

  • Basic: Facility security (biometric, CCTV, motion detection, man traps), network security (firewalls, IDS, vulnerability assessment), platform security (SSL, password policies, trust certification).
  • VM Security: Program shepherding; VMware vSafe/vShield; Intel vPro; security compliance for hypervisors.
  • Defense via Virtualisation: VM isolation; live migration for DIDS; PKI domain trust; security attack isolation between VMs.
  • Privacy & Copyright: Dynamic web services; trust via SLAs; effective identity and access management; single sign-on; auditing; proactive enforcement.
Know the key differences between DoS (single attacker), DDoS (botnet/multiple machines), and EDoS (financial attack via auto-scaling exploitation). This is frequently tested.

Edge & Fog Computing

Edge Computing (Cao & Shi, 2018): Places storage and compute resources closer to where data is produced, on devices, gateways, cloudlets or micro data centers, so results return in milliseconds rather than travelling to a distant hyperscale cloud.

Why Edge Computing Now?

  • Explosion of IoT devices generating data at the network edge.
  • Success of rich cloud services since 2005 raised user expectations for latency.
  • Data deluge: ~500 ZB produced annually by 2019 (Cisco Global Cloud Index), but only ~10.4 ZB expected to reach data centers; ~45% of IoT-created data is processed at or near the edge.

Limitations of Cloud-Only for IoT

  • Higher end-to-end latency for real-time tasks.
  • Increased bandwidth costs for raw data upload.
  • Privacy and data-sovereignty concerns.

Five Drivers for Edge

1. Ultra-Low Latency

Cloud round-trips are often >= 50 ms, which is unacceptable for autonomous vehicles, telesurgery, and similar real-time applications. Moving compute to the edge drops latency into the single-digit millisecond range via:

  • Last-mile proximity: data only travels device, edge node, device, bypassing WAN hops.
  • Hop reduction: fewer routers and switches in the path mean fewer queuing and processing delays.
  • Jitter control: local processing yields more consistent response times.
  • Packet sizing: smaller, context-specific packets at the edge optimise transmission.
2. Bandwidth Optimization
  • Filter & aggregate: send only anomalies or summaries upstream.
  • Compress & dedupe: remove redundancy before transmission.
  • Prioritize traffic: mission-critical flows take precedence over bulk uploads.
3. Enhanced Privacy & Security

Sensitive data (medical records, biometrics) stays on local edge servers, simplifying GDPR and HIPAA compliance.

  • Local encryption & anonymization: tokenize or noise-inject data before any cloud transfer.
  • Zero-trust edge security: authentication and threat detection at every node.
  • Isolated processing enclaves: confidential computing for code and data in use.
4. Reliability & Resilience

Edge nodes keep working when the WAN link to the central cloud drops. Critical for industrial control (power grids, manufacturing), remote monitoring (oil rigs, maritime vessels), and emergency response (disaster zones with intermittent networks).

5. Cost Efficiency
  • Lower bandwidth bills: only meaningful data over expensive links.
  • Defer cloud scaling: small edge clusters absorb peak bursts locally.
  • Heterogeneous edge hardware (CPU, TPU, FPGA) sized to specific workloads.

EdgeOS(H): A Smart Home Edge OS Design

  • Communication Adapter: normalises Wi-Fi / BLE / ZigBee / Cellular into abstracted readings.
  • Event Hub (core): captures events, issues commands, orchestrates services, reads/writes the database.
  • Self-Learning Engine: builds a per-home model that feeds decisions back to the Event Hub.
  • API & Service Registry: lets third-party services integrate cleanly.
  • Name Management: assigns human-friendly IDs.

Other Edge Use Cases

  • Cloud offloading for mobile AR/VR and gaming.
  • Smart homes and connected healthcare.
  • Smart cities and intelligent transportation.
  • Collaborative edge analytics across devices.

Enabling Frameworks

  • Micro Data Centers at telco base stations.
  • Cloudlets for offloading compute-intensive tasks.
  • Fog computing for hierarchical resource pooling between edge and cloud.
Available Edge Platforms

OpenStack, EdgeX Foundry, Machine.io, IOTRACKS, Azure IoT Edge, Google Cloud IoT, AWS Greengrass.

OpenStack at the Edge

  • Cloud OS for data centers: pools compute, storage, and networking and exposes them as a single platform.
  • Self-service via the Horizon dashboard plus REST APIs / CLI.
  • Edge-ready: loosely-coupled microservice design lets you deploy trimmed-down control planes on edge sites while keeping full services in core data centers.
  • Runs both VMs (Nova/KVM and other hypervisors) and containers (Zun, Magnum, Kata Containers).
  • Uniform control plane: a single OpenStack deployment can span core and edge nodes with consistent APIs for automation and federation.

Azure IoT Edge Architecture

  • Cloud side: IoT Hub pushes deployment manifests and receives telemetry; Container Registry stores Docker images for all Edge modules.
  • Device runtime: Edge Agent deploys and updates containers; Edge Hub routes secure messages.
  • Edge modules are containers and can be Azure services, third-party code, or custom logic (e.g. API proxy, temperature sensor).
  • Nested topology: a top-layer gateway device can proxy image pulls and telemetry for lower-layer devices.
  • Data vs image flow: solid lines = telemetry to IoT Hub; dotted lines = container images pulled to the device.

Edge Computing Challenges

1. Resource Heterogeneity & Management
  • Heterogeneous edge pool: phones, MCUs, gateways, micro-servers all differ in CPU, memory, energy budget, and link speed.
  • Single-binary problem: runtimes, ISAs, and toolchains are node-specific; static placement wastes scarce capacity or drains batteries.
  • Remedy: Computing Stream - split an application into ordered function blocks; the scheduler relocates each block (device, gateway, cloud) using latency, energy, TCO, or bandwidth as cost signals. Code and state migrate together; needs live health and cost telemetry.
2. Security, Trust & Privacy
  • Edge data is sensitive: raw electricity or water use can reveal home occupancy.
  • Awareness gaps: many gateways ship with default credentials; surveys found ~49% of household Wi-Fi still unsecured.
  • Data ownership: keep computation at the edge so users control raw data; the cloud receives only authorised aggregates.
  • Lightweight protection tools needed: classic crypto stacks overwhelm tiny sensors; need resource-lean auth, encryption, and OTA patching.
  • Safeguards: on-device redaction (e.g. face masking), hardware roots-of-trust, explicit user consent before data leaves the premises.
3. Programming Models & Developer Tools
  • Developers must target heterogeneous, intermittently-connected nodes while keeping a cloud-grade experience.
  • Computing-stream model: treat the app as an ordered list of compute blocks tied to latency/energy SLAs; the compiler maps each block to the right node.
  • Data-abstraction challenge: sensors emit diverse, noisy formats (temperature vs camera). The gateway should normalise into a unified table (id, time, name, data). Too much filtering loses insight; too little explodes storage and bandwidth.
  • Tooling needs: cross-compilation for many ISAs, simulators to spoof edge sensors, safe migration and rollback when re-allocating functions and their state.

Container Security: Exploits, Attacks & Defenses

With cloud-native applications and microservices, containerization (Docker, Kubernetes) is the dominant deployment model. Unlike VMs, containers share the host OS kernel, which introduces unique security challenges. Containers are also ephemeral and highly dynamic, so traditional security mechanisms are often insufficient. Modern container environments need a layered approach covering everything from infrastructure to application code.

Container Security: Protecting containerized environments from threats like unauthorized access, container escapes, malware injection, and resource misuse, across image, runtime, network, and orchestration layers.

1. Overview & Foundations

The Container Security Paradigm
  • Evolution of virtualization: bundles code, data, and dependencies at the OS level, removing Guest OS overhead.
  • Efficiency vs isolation: shares a single Host Kernel (L4), improving efficiency but reducing isolation compared to VMs.
  • Shared kernel risk: a single vulnerability (E35) can compromise all containers on a host.
  • Overlapping dependencies: shared environments weaken inter-container isolation.
  • Ephemeral containers in cloud-native systems require advanced layered security.
Why Security Matters
  • Single point of failure: all containers share the Host Kernel; a single exploit (e.g. Dirty Pipe) compromises all workloads.
  • Microservice blast radius: in flat networks, a breach in a public web microservice enables lateral movement to internal databases.
  • Forensic auditing challenges: ephemerality means attack evidence disappears once the container is deleted.
  • Resource throttling: without strict cgroups, a "noisy neighbour" can cause Host DoS by consuming 100% CPU/RAM.
  • Supply chain trust: apps use many third-party layers; a single poisoned layer (E13) turns the whole app into a malware vector.
Mapping Exploits to Architectural Layers

The literature catalogues 47 technical exploits (E) across 5 physical and logical layers (L) and 11 attack vectors (A). Legend: L = Layers, A = Attack Vectors, E = Exploits.

LayerExamples of Exploits
L1 OrchestrationAPI Server, Scheduler, Load Balancer attacks.
L2 ApplicationApplication code vulnerabilities.
L3 Container EngineDocker daemon, runc, containerd flaws.
L4 Kernel SpaceNamespace, cgroup, syscall vulnerabilities.
L5 HardwareSpectre, Meltdown at the CPU/BIOS level.
Threat Landscape
  • Attack evolution: code injection, then container escapes (E22), then side-channel attacks.
  • 200+ container CVEs (2013-2024) expose gaps in the Docker Engine (L3).
  • Orchestration risks (A1): attackers target the Kubernetes Control Plane to gain cluster-admin access.
  • Malicious binaries (E1): legitimate tools (e.g. tar, runc) replaced with malicious versions during build/runtime.
Shared Responsibility Model
  • Cloud provider: physical infrastructure, data center, hardware-rooted trust (L5).
  • Infrastructure / platform team: hardens Host OS (L4), Container Engine (L3), Kubernetes API Server (L1).
  • Developer / DevSecOps: manages Application Code (L2), image scanning, Pod Security Context.
  • Zero-trust: treat every layer as a potential entry point; assume lower layers may already be breached.
  • Handoff risk: breaches often occur at responsibility overlaps (runtime patching, shared secrets).

2. Docker & Kubernetes Core Security

Kubernetes Security Context
  • runAsNonRoot: container cannot start if it tries to run as UID 0 (root).
  • allowPrivilegeEscalation: false: blocks gaining higher privileges than the parent process (defends against E32).
  • readOnlyRootFilesystem: mounts root as immutable; prevents attackers from adding backdoors post-breach.
  • Capabilities management: drop specific root powers (e.g. CAP_NET_RAW to prevent sniffing).
Kube-bench: CIS Benchmark Auditing
  • Go-based tool that checks clusters against the CIS Kubernetes Benchmark.
  • Validates Control Plane (API Server, Controller Manager, etcd) for secure encryption and authentication.
  • Audits Worker Nodes (Kubelet, Kube-proxy) for insecure configurations.
  • Outputs Pass/Fail with exact config fixes; used for continuous auditing to prevent configuration drift.
Docker Security Basics
  • Process isolation: Docker uses Linux kernel primitives (namespaces / cgroups), not full machine isolation.
  • Runtime components: security depends on containerd and runc, both targeted by Host Escape exploits (E22).
  • Daemon socket risk: /var/run/docker.sock is a root-level gateway; mounting it inside a container is critical.
  • Immutable infrastructure: containers are not patched in production, they are rebuilt and redeployed.
  • Rootless Docker: runs daemon and containers without root, reducing engine-level attack surface.
Namespaces & cgroups: The Foundation
Namespaces (Isolation)
  • Provide the "userspace fiction" by isolating PID, Network, Mount points, and IPC.
  • Define what a process can see.
  • Namespace bypass (E23): misconfigured shared host PID or Network breaks isolation.
cgroups (Resource Metering)
  • Manage allocation of CPU, RAM, Disk I/O to each container.
  • Define how much a process can use.
  • Primary defense against resource starvation (E34) in multi-tenant clusters.
Union File Systems: Layered Risks
  • Copy-on-Write (CoW): read-only image layers + thin writable top layer.
  • Secret persistence: deleted secrets in later layers still exist in image history.
  • Vulnerability inheritance (E23): flaws in base layers propagate to all child images.
  • Storage exploits: malformed layers can target storage drivers (e.g. Overlay2).
  • Optimization risk: bloated images with extra tools (curl, netcat) aid attackers post-exploit.
Daemon Risks
  • Root privileges: the Docker Daemon (dockerd) historically requires root access to manage iptables and namespaces.
  • Daemon disruption (E19): maliciously crafted images can crash the daemon, causing host-wide DoS.
  • Remote API exploitation: unprotected Docker REST APIs are the #1 target for crypto-jacking botnets.
  • MITM attacks: CLI-Daemon communication must use Mutual TLS (mTLS) to prevent credential theft.
  • Debug mode exposure: can leak sensitive environment variables and auth tokens into host logs.

3. Linux Security Features

Namespaces Isolation in Practice
  • User namespace mapping: maps container root (UID 0) to a high non-root UID on the host to neutralize escapes.
  • PID isolation: prevents containers from signalling (e.g. SIGKILL) host processes.
  • Mount isolation (E27): blocks symlink-based escapes to host files like /etc/shadow.
Control Groups (cgroups) in Practice
  • Memory hard limits: prevent a single container from exhausting host RAM and triggering the Linux OOM Killer.
  • CPU shares & pinning: ensure fair CPU distribution; prevent CPU hogging by malicious or runaway containers.
  • PID limiting: mitigates fork bomb attacks by restricting process creation in a container.
  • Resource misallocation (E34): lack of cgroup limits causes instability in multi-tenant environments.
  • Always define resources.limits in Kubernetes manifests to enforce cgroup controls.
Best Practice: Use Official Images
  • Trusted source utilization (M7): pull only from verified publishers to avoid image poisoning with embedded malware.
  • Vulnerability propagation (E13): unverified images may contain backdoors that bypass network defenses.
  • Distroless & slim images: minimal images (no shell, no package manager) reduce attack surface.
  • Tag pinning: avoid :latest; use fixed versions or SHAs for deterministic, scanned images.
  • Private mirroring: store images in a private, scanned registry to prevent pull poisoning.
Minimize Layers & Avoid Root
  • Multi-stage builds: prevent compilers and build tools from leaking into final image layers.
  • The USER directive: using USER nonroot is the primary defense against runtime escapes.
  • Permission hardening: assign minimal permissions (e.g. 550) and non-root ownership.
  • Runtime immutability: non-root users cannot install packages, blocking post-exploit malware downloads.
  • Exploit prevention (E18): non-root execution mitigates container escapes in the runtime engine (L3).

4. Image, Runtime & Network Security

Vulnerability Scanning
  • Static analysis (M5): tools like Trivy or Clair scan image layers for known CVEs (e.g. OpenSSL, Log4j).
  • Shift-left strategy: CI/CD integration blocks deployment of images with critical/high vulnerabilities.
  • Secret detection: scans flag hardcoded API keys, certificates, or .env files in image layers.
  • Application-level scans: include language dependencies (npm, pip, maven) to catch vulnerable libraries.
  • Continuous scanning: images must be rescanned regularly as new CVEs are discovered.
Image Signing & Trusted Registries
  • Image integrity (M6): use Cosign or Notary to sign images; Kubernetes runs only signed images.
  • Provenance verification: signing gives a verifiable chain of custody for all running code.
  • Trusted registries (M7): use private registries (Harbor, ACR) with RBAC and built-in scanning.
  • Image spoofing (E14): signing prevents typosquatting and malicious look-alike images.
  • Compliance: signatures act as proof that production code matches tested code.
Runtime Security: Seccomp Profiles
  • Syscall filtering: Seccomp restricts kernel syscalls (e.g. blocks reboot(), mount()).
  • Docker default profile: disables ~44 dangerous syscalls, reducing impact of kernel exploits like Dirty COW.
  • Custom whitelisting: high-security pods allow only required syscalls.
  • Kernel exposure (A7): Seccomp is the primary defense against kernel exploits (E35).
  • Enforced globally via Kubelet or per Pod using securityContext.
AppArmor / SELinux & Read-Only Filesystems
  • Mandatory Access Control (MAC): profiles restrict access to files, network ports, and devices per process.
  • AppArmor (path-based): limits container access to specific paths (e.g. write /tmp, block /etc/shadow).
  • SELinux (label-based): uses security labels for deep multi-tenant isolation of files and processes.
  • Defense in depth: even with root access, MAC prevents damage to the host OS.
  • Runtime protection (M23): stops breaches from the Application layer (L2) reaching the Host layer (L4).
Secrets Management
  • Avoid hardcoding: never store secrets in code, Dockerfiles, or Git (a common cause of E5).
  • Environment variable leakage: avoid env vars; secrets can leak via /proc/self/environ or logs.
  • External vaults (M16): use Vault or Azure Key Vault to inject secrets at runtime.
  • Encryption at rest: Kubernetes secrets in etcd are Base64 by default and require manual encryption.
  • Centralized rotation: vaults enable secret rotation without rebuilding or redeploying images.
Network Security: Bridge, Overlay & Policies
  • Bridge networks: single-host isolation, but allow all containers on the bridge to communicate by default.
  • Overlay networks: multi-node communication; require mTLS to prevent sniffing (E6).
  • Network segmentation: group pods to avoid flat networks and restrict access to sensitive services.
  • Zero-trust networking: treat every pod as external; enforce authentication and explicit authorization.
  • Traffic manipulation (E7): without strict policies, attackers can redirect traffic for data exfiltration.

5. Kubernetes & Advanced Topics

Kubernetes Security Overview
  • Attack-Exploit tree: trace how an L1 (Orchestration) exploit propagates downward.
  • Control plane: API Server, etcd, Scheduler, Controller Manager, the cluster's brain.
  • API centrality: all actions pass through the API Server; without auth (E2) the cluster is compromised.
  • Kubelet hardening: disable anonymous access; enforce certificate-based authentication.
Auth & AuthZ: RBAC, Service Accounts, OIDC
  • RBAC fundamentals (M13): defines who (User/ServiceAccount) can do what (verb) on which resource (Pod, Secret, Node).
  • Service accounts: pod identities must follow least privilege and be namespace-restricted.
  • OIDC integration: connect Kubernetes to providers (Okta, AD) to enforce MFA.
  • Avoid cluster-admin for users or CI; use scoped roles instead.
  • Token security: do not auto-mount ServiceAccount tokens unless required.
Pod Security & Admission Controllers
  • Pod Security Standards (PSS): three levels, Privileged (unrestricted), Baseline (default), Restricted (hardened).
  • Admission controllers intercept API requests before persistence (the guardrail).
  • OPA / Gatekeeper: uses Rego to enforce custom policies (e.g. mandatory memory limits).
  • Standard enforcement: rejects pods not meeting the Restricted profile.
  • Policy enforcement (M15): blocks control plane bypass (E2) by enforcing security baselines.
Monitoring & Auditing: Falco & Prometheus
  • Audit logs: record "who did what, when, where" at API level to detect unauthorized actions and drift.
  • Runtime monitoring (Falco): uses eBPF to detect suspicious container activity in real time.
  • Detection scenarios: alerts on shell access, unauthorized writes to /etc, or connections to malicious IPs.
  • Resource monitoring (Prometheus): detects anomalies like CPU spikes indicating crypto-jacking or DoS.
  • Visualizing security (Grafana): dashboards for failed logins, blocked traffic, and security alerts.
Holistic Security: Confidential Computing & Unikernels
  • Zero-trust model: treats cloud infrastructure as untrusted and keeps the application environment trusted/encrypted.
  • TCB optimization: Unikernels reduce the Trusted Computing Base, removing shared kernel attack surface.
  • Confidential VMs (CVMs): use hardware encryption (Intel TDX, AMD SEV) to protect data in-use in memory.
  • Encapsulated full-stack isolation: each container runs as a single-concern unit within a unikernel, preventing lateral movement.
  • Hardware-rooted integrity: uses TPM and Secure Boot to verify full stack integrity before execution.

Conclusion

  • Containers are not VMs: they share the host kernel, making them faster but less isolated, so security is critical.
  • Security happens in layers: a secure system needs defense-in-depth across image, network, runtime, and Kubernetes.
  • Use tools to automate: Trivy, Kube-bench, and Falco help detect and fix vulnerabilities at scale.
  • Shared responsibility: developers secure code and images; administrators secure the cluster and host.
  • The Golden Rule: follow the Principle of Least Privilege to prevent most attacks.
High-yield items: shared kernel risk vs VM isolation; the L1 to L5 layer model; the four Pod Security Context fields (runAsNonRoot, allowPrivilegeEscalation, readOnlyRootFilesystem, capabilities); namespaces vs cgroups (what vs how much); Trivy / Kube-bench / Falco roles; PSS levels; OPA/Gatekeeper; defense-in-depth and least privilege.
