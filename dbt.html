Parallel & Distributed Databases

3.1

Parallel Databases — Introduction

Database operations are frequently time-consuming and involve large volumes of data — they can profit significantly from parallel processing.

Parallel Database System: A system that seeks to improve performance through parallelization of various operations such as loading data, building indexes, and evaluating queries.

Parallel databases improve processing and I/O speeds by using multiple CPUs and disks in parallel. The key goals are:

  • Speedup: Doing the same task faster with more hardware.
  • Scaleup: Handling larger tasks in the same time as the hardware scales.
3.2

Models of Parallelism

Shared-Memory

Single physical address space for the entire machine. Every processor can access all memory directly. Large machines are of the NUMA (Non-Uniform Memory Access) type — accessing another processor's memory takes slightly more time. Critical bottleneck: whether data is in its own cache.

Shared-Disk

Every processor has its own private memory, but all disks are accessible from any processor via communication network. Disk controllers manage competing requests. Appears in two forms: NAS (Network Attached Storage — transfers files) and SAN (Storage Area Networks — transfers disk blocks).

Shared-Nothing

Each processor has its own memory and its own disk(s). All communication is via the network. Most commonly used architecture for database systems. Relatively inexpensive to build using commodity machines in racks. Key challenge: data shipping between processors is costly.

3.3

Query Parallelism

Two distinct methods used to leverage multiple processors and disks to enhance DBMS performance:

Intra-Query Parallelism

Speeds up execution of a single, long-running query by breaking it into smaller tasks run concurrently.

  • Goal: Reduce response time for one complex query (OLAP).
  • Mechanism: Query divided into sub-queries on different data partitions; results combined.
  • Intra-operation: Same operation (scan, sort, join) split across data subsets.
  • Inter-operation: Different operations in the same query run concurrently (pipeline or bushy-tree parallelism).
  • Benefit: Improves individual query latency.
Inter-Query Parallelism

Executes multiple independent queries simultaneously on different processors.

  • Goal: Increase overall system throughput; handle high-volume concurrent requests (OLTP).
  • Mechanism: DBMS allocates resources to different queries, schedules parallel execution, maximizing CPU and I/O utilization.
  • Benefit: Enhances ability to handle concurrent workloads (not individual query speed).
  • Complexity: Relatively simpler to achieve than intra-query parallelism.
FeatureIntra-QueryInter-Query
FocusA single queryMultiple queries/transactions
Primary GoalMinimize single query response timeMaximize overall system throughput
Use CaseComplex, long-running queries (OLAP)High volume of concurrent requests (OLTP)
ComplexityMore complex to implement efficientlyRelatively simpler to achieve
3.4

Parallel Algorithms

Tuple-at-a-Time Operations (on Shared-Nothing)

For p processors, divide relation R's tuples evenly among p processor's disks:

  • For Selection σC(R): Each processor examines tuples on its own disk. Result σC(R) is divided among processors the same way R is.
  • For Projection πL(R): The number of tuples at each processor stays the same. If R is distributed evenly, so is the projection.
  • Note: A selection can radically change distribution of tuples compared to R (uneven results). Projection preserves distribution.

Parallel Algorithms for Full-Relation Operations

  • Deduplication δ(R): Use a hash function to distribute tuples — duplicates land at the same processor. Apply standard algorithm at each processor in parallel.
  • Union / Intersection / Difference (R ∪ S, R ∩ S, R − S): Use the same hash function on both R and S. Work in parallel on portions at each processor.
  • Join R(X,Y) ⋈ S(Y,Z): Hash tuples of both R and S on Y-attributes to p buckets (one per processor). Each processor joins its portion independently.
  • Aggregation (GROUP BY): Hash tuples on grouping attributes to p buckets. Each processor aggregates its bucket.

Performance of Parallel Algorithms

Key Insight: Total work (disk I/Os + CPU cycles) is never less for parallel than for uniprocessor. The advantage is in elapsed time — p processors can do work in 1/p of the time.
  • A unary operation like σC(R) completes in 1/p of the uniprocessor time (when R is evenly distributed).
  • Number of disk I/Os is essentially the same as for uniprocessor selection.

Performance Analysis for Parallel Join

For join R ⋈ S using hash-distribution to p processors:

Total reads (first pass): B(R) + B(S) disk I/Os Tuples to ship: ((p−1)/p) × (B(R) + B(S)) blocks Disk I/Os for initial read + store = 2(B(R) + B(S)) / p Disk I/Os by naive parallel join at each processor = 3(B(R) + B(S)) / p Total disk I/Os per processor = 5(B(R) + B(S)) / p Uniprocessor: 3(B(R) + B(S))
3.5

Solved Example: Parallel Join Performance

This numeric example frequently appears in exams. Know all the steps and formulas.
Problem: R(X,Y) ⋈ S(Y,Z) — B(R) = 1000, B(S) = 500, M = 101 buffers, p = 10 processors
1Hash each tuple of R and S to one of 10 "buckets" using hash function h on join attribute Y. Ship each tuple to the processor corresponding to its bucket.
2Total disk I/Os to read R and S: B(R) + B(S) = 1000 + 500 = 1500 I/Os → 150 per processor.
3Each processor has ~150 blocks total. Of these, ~15 blocks belong to each of the other 9 processors, so each processor ships 15 × 9 = 135 blocks to other processors. Total network communication = 135 × 10 = 1350 blocks.
4Each processor receives ~50 blocks of S-tuples. Ship S first, store in memory (using 50 of 101 buffers). Then stream R-tuples through, comparing against in-memory S for join output.
5Cost of the join = 1500 disk I/Os total. Elapsed time dominated by 150 disk I/Os per processor + tuple shipping + in-memory join computation.
✓ Elapsed time ≈ 1/10th of the uniprocessor time (far less than 1500 I/Os on one machine)
3.6

Map-Reduce Framework & Hadoop

MapReduce: A high-level parallel programming system that divides large input datasets into smaller independent chunks and processes them concurrently across multiple nodes. The framework automatically handles task scheduling, monitoring, and inter-machine communication.

How MapReduce Works

  • User writes code for two functions: map() and reduce().
  • A master controller divides input data into chunks and assigns different processors to execute map on each chunk.
  • Other processors (or the same) perform reduce on pieces of the map output.

Three Phases

Mapping

Indexing and sorting data into the desired structure based on specified key-value pairs. Each mapper processes one input split independently.

Shuffling

Redistributes the mapper's output across nodes depending on the key. All values for the same key are grouped to the same reducer.

Reducing

Produces a summary of previously mapped and shuffled data. Each reducer processes all values for its assigned keys and produces final output.

The Storage Model (Hadoop)

  • Massively parallel machine, typically shared-nothing.
  • Commodity computers mounted in racks, connected by simple network.
  • Data stored in large files divided into chunks (typically many megabytes, e.g., complete disk cylinders).
  • Each chunk is replicated several times for fault tolerance.

MapReduce Execution Flow (Hadoop)

  1. Client submits the job (JAR file, Mapper & Reducer logic, input/output paths).
  2. Job Tracker accepts the job, splits into tasks, assigns to Task Trackers based on data locality.
  3. Task Trackers execute Map Tasks on local input splits.
  4. Intermediate results are shuffled and sorted.
  5. Reduce Tasks process grouped results, generating final output.
  6. Final output is written back to HDFS.

MapReduce Advantages

AdvantageDescription
ParallelismTasks executed simultaneously across hundreds/thousands of nodes
ScalabilityHandles increasing data volumes by adding more nodes
Fault ToleranceAutomatically recovers from task/node failures using replicated data
EfficiencyReduces network traffic by processing data locally (data locality)
SimplicityDevelopers implement only map() and reduce(); framework handles the rest
3.7

Distributed Database Concepts

Parallel Database System

Data stored at a single site. Used to improve query performance through parallel implementation. Processors are closely coupled (shared memory/disk/nothing).

Distributed Database System (DDB)

Data stored across several sites, each managed by a DBMS capable of running independently. Sites are loosely coupled via a general network.

DDB (Distributed Database): A collection of multiple logically related databases distributed over a computer network.
DDBMS (Distributed DBMS): A software system that manages a DDB while making the distribution transparent to the user.
3.8

Homogeneous vs Heterogeneous Distributed Databases

Homogeneous DDB
  • All sites have identical software.
  • Sites are aware of each other and agree to cooperate in processing user requests.
  • Each site surrenders part of its autonomy (right to change schemas or software).
  • Appears to user as a single system.
Heterogeneous DDB
  • Different sites may use different schemas and software.
  • Difference in schema → major problem for query processing.
  • Difference in software → major problem for transaction processing.
  • Sites may not be aware of each other; provide limited cooperation facilities.
3.9

Advantages & Additional Functions of DDB

Advantages of Distributed Databases

  1. Management with Transparency: Hides data location details.
    • Distribution/Network transparency: Freedom from operational network details (location + naming transparency).
    • Replication transparency: User unaware copies exist.
    • Fragmentation transparency: User unaware of fragments.
  2. Increased Reliability and Availability:
    • Reliability: Probability system is running at a specific time point.
    • Availability: Probability system is continuously available during a time interval.
  3. Improved Performance: DDBMS fragments the database keeping data close to where it's needed most (data localization reduces CPU and I/O).
  4. Easier Expansion: Adding more data, increasing database sizes, or adding processors is much easier.

Additional Functions Required by a DDBMS

  1. Keeping track of data: Track data distribution, fragmentation, and replication.
  2. Distributed query processing: Access remote sites via communication network.
  3. Distributed transaction management: Execute transactions accessing data from many sites.
  4. Replicated data management: Decide which replica to access; maintain consistency of copies.
  5. Distributed database recovery: Recover from individual site crashes or communication failure.
  6. Security: Proper management of data security and user authorization/access privileges.
  7. Distributed directory (catalog) management: Design and policy for directory placement.

3.11

Distributed Query Processing

4 Stages of Distributed Query Processing

  1. Query Mapping: Input query on distributed data → algebraic query on global relations. Done by referring to global conceptual schema. Does NOT consider actual data distribution.
  2. Localization: Maps distributed query on global schema → separate queries on individual fragments using data distribution and replication info.
  3. Global Query Optimization: Selects a strategy from candidates closest to optimal. DDBs are connected by network, so communication costs are most significant.
  4. Local Query Optimization: Standard query optimization at each individual site.

The Distributed Join Problem

For join R(A,B) ⋈ S(B,C) where R and S reside at different nodes:

  • Option 1: Send copy of R to site of S and compute join there.
  • Option 2: Send copy of S to site of R and compute join there.

Cost factors to consider: (1) cost of data transmission over network; (2) potential gain from parallel processing at multiple sites.

3.12

Solved Examples: Distributed Join & Semi-Join

Both the Distributed Join example and Semi-Join example are high-priority exam topics. Know the byte calculations.

Solved Example 1: Distributed Join Cost Comparison

Given: EMPLOYEE at Site 1, DEPARTMENT at Site 2. Query submitted at Site 3 (result site).

Data sizes: EMPLOYEE = 100 bytes/record × 10,000 records = 1,000,000 bytes. DEPARTMENT = 35 bytes/record × 100 records = 3,500 bytes. Query result = 40 bytes/record × 10,000 = 400,000 bytes.

Query: πFname,Lname,Dname(EMPLOYEE ⋈Dno=Dnumber DEPARTMENT)

1Option 1 — Transfer both to Site 3: Transfer EMPLOYEE (1,000,000) + DEPARTMENT (3,500) = 1,003,500 bytes
2Option 2 — Transfer EMPLOYEE to Site 2, join, send result to Site 3: Transfer EMPLOYEE (1,000,000) + result to Site 3 (400,000) = 1,400,000 bytes
3Option 3 — Transfer DEPARTMENT to Site 1, join, send result to Site 3: Transfer DEPARTMENT (3,500) + result to Site 3 (400,000) = 403,500 bytes
✓ Option 3 is best: 403,500 bytes — always prefer transferring the smaller relation

Solved Example 2: Semi-Join Optimization

Semi-Join: Only the relevant part (join attributes projected out) of one relation is shipped to the site of the other, reducing transmission cost.

Algorithm: To compute R(X,Y) ⋈ S(Y,Z):

  1. Project join attributes of S → send πY(S) to site of R.
  2. Compute R ⋈ πY(S) at site of R — this filters R to only tuples that will match.
  3. Send the result to site of S, compute full join R ⋈ S there.
Semi-Join Example: π(EMPLOYEE ⋈ DEPARTMENT) — result at Site 2
1Project join attribute Dnumber from DEPARTMENT (Site 2): F = πDnumber(DEPARTMENT). Size = 4 bytes × 100 records = 400 bytes. Send F to Site 1.
2At Site 1, join F with EMPLOYEE: R = πDno,Fname,Lname(F ⋈Dnumber=Dno EMPLOYEE). Size = (4+15+15) × 10,000 = 34 × 10,000 = 340,000 bytes. Transfer R to Site 2.
3At Site 2, join transferred R with DEPARTMENT to get the final result.
✓ Total bytes transferred = 400 + 340,000 = 340,400 bytes (far better than Option 3's 403,500!)

Joins of Many Relations

When joining 3+ relations at different sites, several semi-joins may be needed to eliminate dangling tuples before shipping. Important: some schemas have no finite sequence of semi-joins that eliminates all dangling tuples.

Example: Natural join R(A,B) ⋈ S(B,C) ⋈ T(C,A) — while R ⋈ S produces n tuples {(1,1,1),(2,2,2),...,(n,n,n)}, the full R ⋈ S ⋈ T may be empty if T = {(1,2),(2,3),...,(n-1,n),(n,1)} — no matching (C,A) pairs.
3.13

Distributed Transactions, 2PC & 3PC

Distributed Transaction Components

A transaction may access data at several sites. Two roles per site:

Local Transaction Manager
  • Maintains a log for recovery purposes.
  • Participates in coordinating concurrent execution of transactions at that site.
Transaction Coordinator
  • Starts execution of transactions originating at the site.
  • Distributes sub-transactions to appropriate sites.
  • Coordinates termination — either committed at all sites or aborted at all sites.
Key Rule: It is NOT acceptable to have a transaction committed at one site and aborted at another. Commit protocols ensure atomicity.

Two-Phase Commit Protocol (2PC) — Phase 1: Voting

  1. Coordinator logs <Prepare T> at its site.
  2. Coordinator sends prepare T message to all participant sites.
  3. Each site decides whether to commit or abort its component of T.
  4. If site wants to commit: Enters pre-committed state (cannot abort without coordinator). Performs necessary steps, logs <Ready T>, flushes log to disk, sends ready T to coordinator.
  5. If site wants to abort: Logs <Don't Commit T>, sends don't commit T to coordinator.

Two-Phase Commit Protocol (2PC) — Phase 2: Decision

  1. If coordinator received ready T from all components → logs <Commit T> and sends commit T to all sites.
  2. If coordinator received don't commit T from any site → logs <Abort T> and sends abort T to all sites.
  3. Site receiving commit T: commits its component, logs <Commit T>.
  4. Site receiving abort T: aborts T, logs <Abort T>.

Three-Phase Commit Protocol (3PC)

Addresses some drawbacks of 2PC (e.g., blocking when coordinator fails in Phase 2).

  • Phase 1: Identical to 2PC Phase 1 (obtain preliminary votes). Every site ready to commit if instructed.
  • Phase 2 (new): Coordinator makes pre-commit decision (as in 2PC) and records it in multiple (at least K) sites. This ensures the decision survives coordinator crash.
  • Phase 3: Coordinator sends final commit/abort message to all participating sites.
Tradeoff: 3PC is more complex and expensive but avoids the blocking problem of 2PC when the coordinator crashes after Phase 1.

Concurrency Control Requirements

All distributed concurrency mechanisms must:

  • Be resilient to site and communication link failures.
  • Allow parallelism to enhance performance.
  • Incur optimal cost and minimize communication delays.
  • Place correct constraints on atomic actions.

Distributed Locking & Catalog Management

Distributed Locking Approaches

MethodHow It WorksProsCons
Centralized Lock Systems One designated lock site maintains a lock table for all logical elements. Transactions send requests to this one site. Simple to implement. 3 messages per lock (request, grant, release). Single lock site is a bottleneck. If it crashes, no locks can be granted anywhere.
Primary-Copy Locking Each logical element X has one copy designated as the "primary copy." Transactions request lock from primary copy's site. Distributes lock function. Avoids bottleneck. Same simplicity as centralized. 3 messages per lock. Primary copy site still a single point of failure for that element.
Global Locks from Local Locks (Majority Locking) No copy is "primary" — all copies symmetric. Transaction must obtain local locks from a majority of copies. Broadcasts single request to all sites. No single point of failure. Release by single broadcast message. More messages per lock. Complex deadlock detection.

The Schema Architecture of a distributed database has layers: User → External View → Global Conceptual Schema (GCS) → Local Conceptual Schema (LCS) → Local Internal Schema (LIS) → Stored Data.

Peer-to-Peer Networks & Chord Circles

Peer-to-Peer Network Characteristics

A P2P network is a collection of nodes that are:

  1. Autonomous: No central control; join/leave at will.
  2. Loosely coupled: Communicate over general-purpose network (Internet).
  3. Equal in functionality: No leader or controlling node.
  4. Resource sharing: Share resources with each other.

Each node has a subset of data elements, but no centralized index says where something is located.

Distributed Hashing Problem

Given a large set of key-value pairs (K,V), constraints:

  1. At any time, only one node knows the value for any given key K.
  2. Key-value pairs distributed roughly equally among peers.
  3. Any node can find V for key K with messages growing much slower than number of peers.
  4. Routing info at each node must also grow much slower than number of nodes.

Chord Circles

Chord: Arranges peers in a "chord circle." Nodes are placed on a circle using a hash function h that maps both keys and node IDs to m-bit numbers (2m positions).
  • To place node with ID i: compute h(i) → place at position Nh(i).
  • Successor = next higher node clockwise. Predecessor = next counter-clockwise.
  • Key-value pair (K,V): stored at lowest-numbered node Nj such that h(K) ≤ j.
  • If m = 6 → 64 positions. Example: pair with h(K) = 45 → stored at N48.

Finger Tables

Each node maintains a finger table with m entries giving the first node found at distances 1, 2, 4, 8, ..., 2m-1 around the circle (powers of two).

Distance12481632
Node (for N8)N14N14N14N21N32N42

Solved Example: Search Using Finger Tables

N8 wants to find key K where h(K) = 54. Circle: N1, N8, N14, N21, N32, N38, N42, N48, N51, N56
1N8 checks if h(K)=54 is in range [9,14] (successor N14) — No. Examines finger table: all entries below 54. Takes largest entry = N42. Sends message to N42 to find K.
2N42 checks if 54 is in range [43,48] (successor N48) — No. Examines own finger table: N48, N48, N48, N51, N1, N14. Last entry below 54 (circularly) = N51. Sends request to N51.
3N51 finds that 54 < successor N56. So if (K,V) exists, it's at N56. N51 sends request to N56, asking it to reply to N8.
✓ Found in 3 messages (≤ m = 6 max). Search uses at most m+1 messages for any key.

Adding a New Node (Example: N26)

Add node N26 to the circle (N21 ... N32)
1N26 contacts any existing peer. That peer finds N26's successor = N32.
2N26 sets successor = N32, predecessor = nil.
3N21 runs the stabilization protocol → updates its successor to N26 and N32 updates predecessor to N26.
4N32 transfers all (K,V) pairs where h(K) ≤ 26 to N26 (these now belong to N26).

Node Departure & Failure

Graceful departure: Node (1) notifies predecessor and successor so they become each other's neighbors, and (2) transfers its data to successor.

Node failure handling:

  1. Replicate each (K,V) at 3 nodes: correct node, its predecessor, and its successor.
  2. Cluster nodes into groups of 3+ that replicate data and substitute for each other.
  3. Large clusters can be split; small clusters can be merged with neighbors.

Stream Processing, Big Data Architectures & Emerging Technologies

4.1

Stream Processing — Introduction

Stream Processing: The discipline and related techniques used to extract information from unbounded data — datasets that are theoretically infinite in size.
Bounded Dataset

A dataset of known, finite size. We can count the number of elements. Traditional databases and files are bounded.

Unbounded Dataset

Theoretically infinite. Elements arrive continuously. Processing must happen continuously as long as the stream delivers new data.

Business Opportunity: The need to process data and create actionable insights in a short period from when data was received. Examples: fraud detection, real-time recommendations, IoT monitoring.

4.3

Notion of Time in Stream Processing

Event Time

The time when the event was created. Provided by the local clock of the device generating the event. Reflects the real-world occurrence time.

Example: A temperature sensor records "25°C at 14:00:05" — 14:00:05 is the event time.

Processing Time

The time when the event is handled by the stream-processing system (server clock). Useful for computing processing lag or detecting duplicate output.

Example: The same temperature reading arrives at the server at 14:00:12 — 14:00:12 is processing time.

The Gap: There is always a time difference between event creation and when it becomes available for processing. Events may arrive out of order, late, or get lost entirely. This delta is the delivery delay.

Uncertainty in Stream Processing

  • Streaming systems do not require input at regular intervals.
  • Predicting peak load is challenging — sudden bursts may cause delays, resource constriction, or failure.
  • Dealing with uncertainty (late arrivals, out-of-order events, duplicates) is a central challenge.
Delivery Delay: The time elapsed between the production of an event and its eventual consumption by the stream processor. Visualized as the vertical distance between the diagonal (ideal) line and the processing time on an event-time vs. processing-time chart.
4.4

Window Aggregations (Tumbling & Sliding)

Window: A bounded, recurring time-based aggregation in stream processing. Provides a "view" of a finite portion of an otherwise infinite stream.
Tumbling Windows

Fixed, non-overlapping, consecutive time periods. Each window is independent from others. Natural grouping of "x events per period."

  • Example: "Max temperature each hour" — 00:00–01:00, 01:00–02:00, 02:00–03:00, etc.
  • Example: "Total energy consumption each 15 minutes."

Use when: periods are independent, non-overlapping summaries needed.

Sliding Windows

Windows that overlap. Defined by a window size and a slide interval. Each window moves forward by the slide amount.

  • Example: "Average temperature over the last 10 minutes, updated every 5 minutes" — overlapping windows that share data.

Use when: you need continuous updates and recent data matters across window boundaries.

Event Time vs Processing Time in Windowing

Processing-time windows: Use server clock to define 10-second intervals. All events arriving in that interval belong to the window. Boundaries are well-defined. But window contents are arbitrary — unrelated to when events were generated. A late-arriving event (generated at 00:36 but arriving after 00:40) falls out of the window.

Event-time windows: Use event creation time as window criteria. Window 00:30–00:40 contains all events created in that period — but has no natural upper boundary when it ends. Event created at 00:36 but arriving at 01:00 is "late by 20+ seconds." To close event-time windows, we need watermarks.

4.5

Watermarks

Watermark: At any given moment, the oldest timestamp the stream processor will still accept on the data stream. Any events older than the watermark are not included in results (or processed in a late-arrivals channel).

The watermark closes the open boundary of event-time windows — it provides a deadline that decides what events belong to a window and what are "too late."

Two Processing Modes After Watermark

  1. Final Output: All events in the output are older than the watermark → output is final; no further older events will ever change it.
  2. Provisional Output: Output covers data before watermark but a delayed element newer than the watermark could still arrive → output is provisional and may change.
Memory Cost: To handle watermarks, the stream processor stores a lot of intermediate data. Memory needed ≈ watermark length × arrival rate × message size.

Stream processors using watermarks that want unique, final results for each interval must delay output for at least the length of the watermark.

4.6

Lambda Architecture

Lambda Architecture: A data-processing architecture designed to handle massive quantities of data using both batch and stream-processing methods. Balances latency, throughput, and fault-tolerance.

Three Layers

Batch Layer

Receives all new data. Processes entire dataset on predefined schedule (once/twice a day). Corrects data in the stream layer. Functions: (1) manage master dataset, (2) pre-compute batch views.

Speed (Stream) Layer

Handles data not yet in the batch view due to batch latency. Only deals with recent data. Provides real-time views to complement batch views for a complete picture.

Serving Layer

Receives batch views and near-real-time views. Indexes batch views for low-latency ad-hoc queries. Users query this layer to get combined view of batch + stream results.

Tools Used

Apache Hadoop & HDFS (batch/storage), Apache Spark (streaming, batch, graph), Apache Cassandra (real-time views), Apache Kafka (speed layer streaming), Apache Storm (stream layer), Apache HBase (serving layer).

Benefits & Challenges

Benefits
  • No server management required.
  • Flexible Scaling (auto-scale).
  • Automated High Availability and fault tolerance.
  • Business Agility — react in real-time.
Challenges
  • Complexity: Two separate code bases for batch and stream layers.
  • Debugging is difficult — logic duplicated in two systems.
  • Higher maintenance overhead.
4.7

Kappa Architecture

Kappa Architecture: A streamlined data processing approach focused on real-time stream processing. Treats all data as a stream — both real-time AND historical data processed using a single processing engine. Simpler alternative to Lambda.

Kappa vs Lambda

AspectLambda ArchitectureKappa Architecture
Batch LayerYes — separate batch processingNo — stream only
Code BasesTwo (batch + stream)One (stream for both)
ComplexityHigherLower
Historical ReprocessingVia batch layerEvent replay from stream
LatencyEventual (batch) + real-time (speed)Consistently low latency

Kappa Architecture Components

  1. Data Source: Real-time streams from IoT devices, logs, user interactions.
  2. Stream Processing Engine: Kafka, Flink, or Samza — filter, transform, aggregate on the fly.
  3. Data Storage: NoSQL (Cassandra, HBase) or distributed file systems (HDFS, S3) — also handles event replay.
  4. Serving Layer: Serves processed data to users, dashboards, downstream applications.
  5. Reprocessing/Replay Mechanism: No batch layer — replay past events from original stream if reprocessing needed (e.g., after code changes).

Benefits of Kappa

  • Simplicity: Single stream-processing logic for both real-time and historical.
  • Low Latency: All data processed in real-time.
  • Unified Codebase: Reduces maintenance overhead.
  • Scalability: Kafka + Flink can distribute across many nodes.
  • Event Replayability: Easy historical reprocessing without batch jobs.
  • Fault Tolerance: Built-in replication in stream engines.

Best Practices

  1. Choose right engine (Kafka for large-scale, Flink for complex stateful).
  2. Implement event sourcing — all changes as immutable events.
  3. Ensure idempotency — handle duplicate events safely.
  4. Prefer stateless operations when possible.
  5. Use windowing for stateful aggregations.
  6. Handle data skew through proper partitioning.
  7. Implement fault tolerance via checkpointing.
  8. Design for horizontal scaling.

Real-World Use Cases

  • LinkedIn (Kafka): Processes billions of events/day — activity feeds, messaging, monitoring.
  • Netflix: Real-time personalization, recommendations, operational monitoring.
  • Uber: GPS, pricing, routing, demand prediction in real-time.
  • Spotify: User behavior tracking, music recommendations, real-time ads.
  • Pinterest: Real-time pin/repin/click processing for analytics and recommendations.
4.8

Apache Spark

MapReduce Limitations (Why Spark?)

  • Performance: MapReduce query execution often takes hours — orders of magnitude slower than modern DBMSs. Prevents interactive analysis.
  • Programming Model: Requires advanced skills and deep understanding of system architecture.
  • Configuration: Numerous sensitive configuration parameters.
  • Batch Only: No support for streaming, iterative algorithms, or interactive queries.
Apache Spark: A cluster-computing platform developed at UC Berkeley (AMPLab). Replaces MapReduce execution framework. Provides API for distributed programming but designed for fast interactive queries and iterative algorithms by caching data in memory. 10–20× faster than MapReduce.

Resilient Distributed Datasets (RDDs)

RDD: Spark's core data model. Distributed, immutable collections stored in memory across cluster nodes. Eliminates expensive intermediate disk writes of MapReduce. Because Spark knows the data source and all operations performed, it can reconstitute lost data from a crashed executor from scratch.

Typical Spark data flow:

  1. Define RDDs — access stored data (HDFS, Cassandra, S3), parallelize collections, transform existing RDDs, or cache.
  2. Invoke operations — pass closures (functions) to each element. Spark offers many high-level operators beyond map and reduce.
  3. Aggregate results — actions like count, collect, save trigger actual cluster computation.

Key Spark Features

Lazy Evaluation

Most operations are lazy — no computation happens until a final eager output (action) is called. This allows optimization of the entire execution plan. Intermediate results disappear after being consumed by the next step.

Cache Hints

Users can specify that an intermediate RDD should be cached for later reuse. Spark can spill the cache to secondary storage if memory runs out, preserving functional aspects even under peak loads.

Polyglot API: Scala, Java, Python, and R. Built-in modules for machine learning (MLlib), streaming data (Spark Streaming), graph algorithms (GraphX), and SQL-like queries (Spark SQL).

4.9

Spark Streaming & DStreams

Spark Streaming: First stream-processing framework built on Spark. Uses microbatching — converts a continuous stream into small discrete batches (at fixed time intervals) that Spark can process using its batch engine.

DStream (Discretized Stream)

DStream: Represents a stream in terms of discrete blocks of data, represented as RDDs over time. A DStream is a sequence of RDDs — each RDD contains data from a specific time interval.
  • DStreams are immutable — only through transformations can we process them.
  • All DStream transformations are lazy — they define a plan of action for when streaming starts.
  • DStream programming requires ending with an output operation (e.g., pprint(), saveAsTextFiles()) to materialize results.

Structure of a Spark Streaming Application (4 Steps)

  1. Create a SparkStreamingContext
  2. Define one or more DStreams from data sources or other DStreams
  3. Define one or more output operations to materialize results
  4. Start the SparkStreamingContext

Solved Example: Stateless Spark Streaming (Word Count)

Streaming Word Count — processes each batch independently (stateless)
from pyspark.streaming import StreamingContext

# Step 1: Create StreamingContext with 2 threads, 1-second batch interval
ssc = StreamingContext(sparkContext, 1)

# Step 2: Create DStream from socket (hostname:port)
lines = ssc.socketTextStream("localhost", 9999)

# Step 3: Split each line into words (flatMap: one-to-many)
words = lines.flatMap(lambda line: line.split(" "))

# Step 4: Count words in each batch (stateless - resets each batch)
pairs = words.map(lambda word: (word, 1))
wordCounts = pairs.reduceByKey(lambda a, b: a + b)

# Step 5: Print output (output operation to materialize DStream)
wordCounts.pprint()

# Step 6: Start computation
ssc.start()

# Step 7: Wait for termination
ssc.awaitTermination()

Solved Example: Stateful Spark Streaming (Running Word Count)

Maintains state across batches using updateStateByKey
from pyspark.streaming import StreamingContext

# Step 1: StreamingContext, 1-second interval
ssc = StreamingContext(sparkContext, 1)

# Step 2: Set checkpoint directory for state recovery on failure
ssc.checkpoint("/tmp/spark_streaming_checkpoint")

# Step 3: Define state update function
def update_function(new_values, running_count):
    if running_count is None:
        running_count = 0
    return sum(new_values, running_count)

# Step 4: Create DStream from socket
lines = ssc.socketTextStream("localhost", 9999)

# Steps 5-8: Process and maintain state
words = lines.flatMap(lambda line: line.split(" "))
pairs = words.map(lambda word: (word, 1))
running_counts = pairs.updateStateByKey(update_function)

running_counts.pprint()
ssc.start()
ssc.awaitTermination()

Latency Characteristics

  • Microbatching creates a baseline latency — any event in a batch is delayed by at least the batch interval. Common lower bound: 1 second.
  • Spark excels at throughput-oriented processing — large amounts of data per unit time, not ultra-low latency.
4.10

Structured Streaming

Structured Streaming: Extension of Spark's Dataset/DataFrame SQL-oriented model to handle data in motion. Treats streaming data as rows continuously appended to an unbounded table. Queries over this table produce an output stream.

Model Concepts

  • Data arrives from a source stream with a defined schema.
  • Stream events = rows appended to an unbounded table.
  • Apply SQL-like queries to the table → output stream of processed events.
  • Results offered to an output sink (storage, another stream, or application).
  • Introduces support for event time, watermarking, and different output modes.

Solved Example: Structured Streaming in PySpark

Streaming DataFrame with Schema
from pyspark.sql import SparkSession
from pyspark.sql.types import StructType, StringType, IntegerType

# Step 1: Start SparkSession
spark = SparkSession.builder.appName("StructuredStreaming").getOrCreate()

# Step 2: Define schema (REQUIRED for structured streaming)
schema = StructType().add("name", StringType()).add("age", IntegerType())

# Step 3: Load data into streaming DataFrame using readStream
streamingDF = spark.readStream.schema(schema).json("/path/to/source")

# Step 4: Check if stream is active
print(streamingDF.isStreaming)  # True

# Step 5: Apply transformations (SQL-like)
result = streamingDF.select("name").where("age > 25")

# Step 6: Write to sink with output mode
query = result.writeStream \
    .format("console") \
    .outputMode("append") \
    .start()

query.awaitTermination()
4.11

Apache Kafka

The Data Pipeline Problem

In modern systems, many data sources stream at varied rates, and many backends need to process the same data. Direct point-to-point connections create a fragile "maze" — O(M×N) connections for M sources and N backends. Kafka acts as an intermediary that decouples producers and consumers.

Apache Kafka: An open-source distributed event streaming platform. Combines three key capabilities: (1) publish and subscribe to event streams, (2) store streams durably and reliably, (3) process streams in real-time or retrospectively.

Three Essential Components

Producer (Publisher)

Defines what data to send. Packages data into messages and publishes them to the communication infrastructure. Does not need to know who the consumers are.

Consumer (Subscriber)

Tells infrastructure what type of messages it wants to receive (not who to receive from). Messages delivered by infrastructure. Often organized into consumer groups for load balancing.

Communication Infrastructure (Broker)

Most complex part. Routes messages to subscribers. Supports topic-based routing. Kafka uses a cluster of brokers with distributed partitioned logs.

Routing Models

Topic-Based Routing

Publishers send messages with topic labels. Subscribers subscribe to topics and receive all messages on that topic.

Example: "Subscribe to all fire sensors in B block" → topic = "fire-sensors-B".

Kafka supports this natively. Also supports pattern-based (wildcard) topic matching.

Content-Based Routing

Subscribers define matching criteria and receive all messages matching those criteria. More expressive but more complex.

Example: "Subscribe to all ads featuring Virat Kohli".

NOT supported natively by Kafka. Kafka only supports topic-based and pattern-based routing.

Kafka Performance

  • Sequential I/O: Sequential reads by consumers and sequential writes by producers — much faster than random I/O.
  • Zero-Copy I/O: Uses DMA (Direct Memory Access) — no copy from kernel to user space.
  • Write Batching: Batches multiple messages together for efficiency.

Consumer Groups & Partitions

A topic with multiple partitions can be consumed by a consumer group (multiple instances of the same application). Each partition delivers messages to one member of the consumer group — achieving load balancing. Different consumer groups each get their own copy of all messages.

Multi-model Databases

Multi-model Database: A database that can store, index, and query data in more than one model (relational, document, graph, key-value, object). Combines multiple data models in a single backend.
Polyglot Persistence: Using multiple data storage technologies for varying data storage needs within a single application. Different data requires different stores — pick the right tool for each use case.

OrientDB (Key Multi-model Database)

  • Open-source, written in Java. Supports graph, document, key/value, and object models.

4.13

Data Warehousing & OLAP

Data Warehouse: A storage architecture designed to hold data extracted from transaction systems, operational stores, and external sources. Combines data in aggregate/summary form for enterprise-wide analysis. It is subject-oriented, integrated, nonvolatile, time-variant.
OLAP (Online Analytical Processing): The analysis of complex data from the data warehouse, as opposed to OLTP (transactional processing).

Building a Data Warehouse

Data Acquisition Steps: Extraction from heterogeneous sources → Data formatting for consistency → Data cleaning for validity → Fit into data model → Load into warehouse.

Design Considerations: Usage projections, data model fit, source characteristics, metadata component design, modular component design, manageability, distributed/parallel architecture.

Multi-Dimensional Modeling

  • Takes advantage of inherent relationships in data to populate data cubes.
  • Dimension Tables: Consist of tuples of attributes of a dimension (e.g., Time, Location, Product).
  • Fact Tables: Contain measured/observed variables with pointers to dimension tables. One tuple per recorded fact.
Star Schema

Fact table at center with a single table for each dimension. Simple, fast queries. A fact constellation is a set of fact tables sharing some dimension tables. Uses bitmap indexing for high-performance access.

Snowflake Schema

Variation of star schema where dimensional tables are organized into a hierarchy by normalizing them. Reduces data redundancy but requires more joins — may be slower for some queries.

Data Mining

Data Mining: The process of discovering meaningful correlations, patterns, and trends by sifting through large amounts of data stored in repositories. Employs pattern recognition, statistical, and mathematical techniques.

Why Mine Data?

  • Data collected at enormous speeds (GB/hour) — remote sensors, telescopes, gene expression arrays, scientific simulations.
  • Traditional techniques infeasible for raw data at this scale.
  • Data mining helps classify, segment data, and formulate hypotheses.

KDD Process (Knowledge Discovery in Databases)

Input Data → Data Preprocessing → Data Mining → Pattern Evaluation → Knowledge Output. The mining step discovers: association rules, sequential patterns, classification trees, clustering, etc.

Types of Knowledge Discovered

TypeDescriptionExample
Association RulesCorrelate presence of items with other variables"Customer buys video equipment → also buys another electronic gadget" (if-then rules with confidence/support)
Sequential PatternsTime-ordered sequences of item purchases"Buys camera → within 3 months buys photo supplies → within 6 months buys accessory"
Classification TreesClassify customers by frequency, financing, purchase amountDecision tree to categorize shoppers: discount-seeking, loyal, name-brand attached

Goals of Data Mining

  • Prediction: Analyze buying transactions to predict what consumers buy under certain discounts.
  • Identification: Identify intruders by programs executed, files accessed, CPU time per session.
  • Classification: Categorize customers (discount-seeking, rush shoppers, loyal, brand-attached).
  • Optimization: Optimize limited resources (time, space, money) to maximize output (sales, profits) under constraints.
4.15

Data Lakes, Lakehouse & Data Fabric

Data Lake

Data Lake: A centralized data repository storing both traditional structured (row/column) data AND unstructured/non-tabular raw data in its native format (videos, images, binary files). Uses inexpensive object storage and open formats.
  • Optimized for quick ingestion of raw, detailed source data.
  • Supports on-the-fly processing for exploration, analytics, and operations.
  • Collection of data organized by user-designed patterns.

Data Lake vs Data Warehouse

AspectData WarehouseData Lake
Data FormatStructured (schema-on-write)Raw, native format (schema-on-read)
StorageRelational, optimized for queriesCheap object storage (HDFS, S3)
UsersBusiness analystsData scientists, ML engineers
ProcessingPre-defined reports, OLAPAd-hoc, ML, data exploration
AgilityLess agile (fixed schema)Highly flexible (raw data)

Key Data Lake Tools

  • Apache Kafka: Real-time data ingestion — publishes, subscribes, and stores event streams durably.
  • Apache Hudi: Incremental processing framework for data lakes. Joined Apache incubator 2019, top project 2020.
  • AWS Data Lake: Amazon S3 (core storage) + Athena (ad-hoc SQL queries on S3) + Redshift (data warehousing) + CloudWatch (monitoring) + Cognito (auth).
  • Apache Iceberg: Open-source table format for large datasets. Metadata-driven (snapshot-based), separates physical storage from logical table structure. Supports transactional ACID operations on data lakes.

Data Lakehouse

Data Lakehouse: A new paradigm combining the best of data lakes and data warehouses — flexible data storage of data lakes + high-performance analytics and data management features of data warehouses — implemented directly on low-cost lake storage.

Data Fabric

Data Fabric: A modern data architecture designed to democratize data access across an organization. A loosely coupled collection of distributed services enabling the right data to be available in the right shape, at the right time, from heterogeneous sources, across any cloud and on-premises platform — via self-service.
  • A network of data nodes (platforms + databases) interacting to provide greater value across the enterprise's hybrid/multicloud ecosystem.
  • Security and governance policies enforced whenever data travels or is accessed.
  • Promotes data discoverability through an enterprise-wide data marketplace with metadata and knowledge graph search.
Evolution: Data Lakes → Data Warehouses → Data Lakehouse → Data Fabric (next stage). Data fabrics simplify management of lakehouses and improve access, sharing, automation, governance, and self-service.

Case Study: Netflix Architecture

The Netflix case study synthesizes all topics from Unit 4 — streaming, Kafka, Spark, NoSQL, CDN, microservices. Know the data flow end to end.

Scale Facts

  • 222+ million subscribers in 190 countries.
  • 6 billion hours of content per month globally.
  • ~15% of the world's internet bandwidth traffic.
  • Average 3.2 hours/day per subscriber.
  • Engineering goal: "Delivering content at scale."

CDN Architecture (Open Connect)

  1. Netflix performs compression and transcoding of original video files.
  2. Files split into chunks using parallel workers in AWS for faster processing.
  3. Each chunk subdivided across several resolutions and internet speeds.
  4. Chunks stored on Amazon S3.
  5. During off-peak periods, files transferred to Open Connect boxes (Netflix's custom CDN spread globally).
  6. Open Connect boxes can communicate and share content between themselves.

Backend Architecture

Netflix operates across two clouds: AWS (backend) and Open Connect (CDN). All user-facing requests (login, recommendations, home page, history, billing) handled by AWS backend.
