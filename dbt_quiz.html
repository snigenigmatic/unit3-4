Written Questions 4M each
W14M
Compare shared-memory, shared-disk, and shared-nothing architectures for parallel databases. For each, describe the architecture and mention one advantage and one disadvantage.

Shared-Memory: All processors access a single shared physical address space. Each processor has local main memory but can directly access other processors' memory. Large machines are NUMA type (non-uniform memory access). Advantage: simple programming model since all data is in a common address space. Disadvantage: limited scalability — the shared bus/interconnect becomes a bottleneck as processors increase, and cache coherence overhead grows.

Shared-Disk: Each processor has its own private memory (not accessible to others), but all processors can access all disks through a communication network. Disk controllers manage competing requests. Two forms: NAS (file-level transfers) and SAN (block-level transfers). Advantage: good fault tolerance — if a processor fails, others can still access all data. Disadvantage: disk contention and the overhead of distributed cache coherence across processors.

Shared-Nothing: Each processor has its own private memory and its own disk(s). All communication is via network messages between processors. Advantage: highly scalable and inexpensive to build using commodity hardware in racks. Disadvantage: data transfer between processors is costly, and algorithms must be carefully designed to minimize inter-node communication.

W24M
Explain the Two-Phase Commit (2PC) protocol in detail. Cover both phases, the role of the coordinator and participants, and the log records involved at each step.

Phase 1 (Voting/Prepare): (1) The coordinator writes <Prepare T> to its log. (2) It sends a "prepare T" message to all participant sites. (3) Each participant decides whether it can commit its local component. (4) If yes, the participant enters a pre-committed state — it ensures no local abort will be needed, writes <Ready T> to its local log, flushes the log to disk, and sends "ready T" to the coordinator. Once pre-committed, a site cannot unilaterally abort. (5) If a site wants to abort, it logs <Don't Commit T> and sends "don't commit T" to the coordinator.

Phase 2 (Decision): (1) If the coordinator receives "ready T" from ALL participants, it decides to commit — logs <Commit T> and sends "commit T" to all sites. (2) If even one participant sends "don't commit T", the coordinator logs <Abort T> and sends "abort T" to all sites. (3) Each site receiving "commit T" commits locally and logs <Commit T>. (4) Each site receiving "abort T" aborts and logs <Abort T>.

Key property: Atomicity is guaranteed — the transaction is either committed at all sites or aborted at all sites. The protocol is blocking: if the coordinator crashes after Phase 1 but before Phase 2, participants in the pre-committed state must wait until the coordinator recovers.

W34M
Using the EMPLOYEE-DEPARTMENT distributed join example from the slides, compute and compare the total bytes transferred for: (a) sending both relations to the result site, (b) sending DEPARTMENT to EMPLOYEE's site and forwarding the result. Show your calculations.

Given: EMPLOYEE at Site 1: 10,000 records × 100 bytes = 1,000,000 bytes. DEPARTMENT at Site 2: 100 records × 35 bytes = 3,500 bytes. Result has 10,000 records × 40 bytes (Fname 15 + Lname 15 + Dname 10) = 400,000 bytes. Query submitted at Site 3.

(a) Send both to Site 3: Transfer EMPLOYEE (1,000,000) + DEPARTMENT (3,500) = 1,003,500 bytes. Join is performed at Site 3.

(b) Send DEPARTMENT to Site 1, join there, send result to Site 3: Transfer DEPARTMENT to Site 1 = 3,500 bytes. Join at Site 1 produces 10,000 result tuples. Transfer result to Site 3 = 400,000 bytes. Total = 3,500 + 400,000 = 403,500 bytes.

Comparison: Option (b) transfers only 403,500 bytes vs 1,003,500 for option (a) — about 60% less. The key insight: always move the smaller relation to the larger one's site, then ship only the (projected) result to the query site. There's also option (c): send EMPLOYEE to Site 2 and result to Site 3 = 1,000,000 + 400,000 = 1,400,000 bytes — the worst option.

W44M
Describe how a key lookup works in a Chord circle using finger tables. Use the example from the slides where N8 looks up a key K with h(K) = 54. Trace each hop.

Setup: m = 6, so 64 positions (0–63). Nodes: N₁, N₈, N₁₄, N₂₁, N₃₂, N₃₈, N₄₂, N₄₈, N₅₁, N₅₆. Each node has a finger table with entries at distances 1, 2, 4, 8, 16, 32.

Hop 1 — N₈: N₈'s successor is N₁₄. Since 54 is not in range [9, 14], the key is not at N₁₄. N₈ checks its finger table: {1→N₁₄, 2→N₁₄, 4→N₁₄, 8→N₂₁, 16→N₃₂, 32→N₄₂}. The largest entry ≤ 54 (in circular sense) is N₄₂. N₈ forwards the request to N₄₂.

Hop 2 — N₄₂: N₄₂'s successor is N₄₈. 54 is not in [43, 48]. N₄₂ checks its finger table: {1→N₄₈, 2→N₄₈, 4→N₄₈, 8→N₅₁, 16→N₁, 32→N₁₄}. The largest entry ≤ 54 is N₅₁. N₄₂ forwards to N₅₁.

Hop 3 — N₅₁: N₅₁'s successor is N₅₆. Since 54 ≤ 56, if (K,V) exists it must be at N₅₆. N₅₁ sends a request to N₅₆ asking it to reply directly to the originator N₈.

Hop 4 — N₅₆: N₅₆ looks up key K locally and sends the value V (or "not found") back to N₈. Total: 3 forwarding messages + 1 response = 4 messages. In general, at most m+1 messages are needed.

W54M
Compare Lambda and Kappa architectures. Discuss their layers, data flow, advantages, and when you would choose one over the other. Give one real-world example for each.

Lambda Architecture: Has three layers — (1) Batch layer: manages the master dataset (immutable, append-only) and pre-computes batch views using tools like Hadoop/HDFS; runs on a schedule (e.g., daily). (2) Speed/Stream layer: processes only recent data not yet in batch views, creating real-time views using tools like Apache Storm or Kafka. (3) Serving layer: indexes batch views for low-latency ad-hoc queries (e.g., HBase). New data is fed to both batch and stream layers simultaneously. Advantages: comprehensive accuracy from batch + low latency from stream; fault-tolerant. Disadvantage: complexity of maintaining two separate codebases with different logic for batch and stream.

Kappa Architecture: Eliminates the batch layer entirely. All data is treated as a stream processed by a single stream processing engine (e.g., Kafka, Flink, Samza). Historical reprocessing is done by replaying events from the immutable log. Results go to a serving layer or storage. Advantages: simpler unified codebase, lower latency, easier maintenance. Disadvantage: reprocessing large historical data can be resource-intensive; not ideal for workloads that are inherently batch-oriented.

When to choose: Lambda when you need both highly accurate historical analysis and real-time views, and can afford the operational complexity. Kappa when real-time processing dominates and you want simplicity. Example — Lambda: a data warehouse that runs nightly batch jobs for reporting while also showing live dashboards. Example — Kappa: LinkedIn's activity feed processing (billions of events/day via Kafka, all stream-based).

W64M
Explain the concept of watermarks in stream processing. How do they resolve the open-boundary problem of event-time windows? What are the trade-offs between watermark length and output freshness?

The problem: When using event-time windows (e.g., aggregate events from 00:30–00:40 based on when they were created), there is no natural upper boundary for when the window closes. Events can arrive late due to network delays — an event created at 00:36 might not arrive until 01:00. Without a cutoff, the system would wait indefinitely for stragglers.

Watermark definition: A watermark is an arbitrary deadline — at any given moment, it represents the oldest event timestamp the system will still accept. Any event older than the current watermark is considered "too late" and is either discarded or routed to a late-arrivals side channel.

How it resolves the open boundary: The watermark closes the event-time window by defining which events belong to the window and which are too late. Once the watermark passes the window's end time, the window can be finalized and results emitted. Without it, event-time windows would never close.

Two modes of output: (1) If all data being processed is older than the watermark → output is final and definitive. (2) If data before the watermark boundary is still arriving → output is provisional and may be revised when delayed events arrive.

Trade-offs: Longer watermark → more complete results (fewer missed late events), but higher latency (output is delayed by at least the watermark length) and higher memory consumption (memory ≈ watermark_length × arrival_rate × message_size, as intermediate state must be stored). Shorter watermark → fresher output with lower latency, but more events will be classified as "late" and potentially dropped, leading to less accurate results.

W74M
Explain the three distributed locking strategies: centralized, primary-copy, and majority locking. For each, state the number of messages required per lock and discuss bottleneck/failure implications.

1. Centralized Locking: A single designated site maintains the lock table for all logical elements across the system. To lock element X, a transaction sends a request to the lock site, which grants or denies it. Messages: 3 per lock (request → grant → release), unless the transaction is at the lock site itself (then 0 messages). Bottleneck: the single lock site can become a performance bottleneck under heavy load. Failure: if the lock site crashes, no transaction anywhere can obtain locks — single point of failure.

2. Primary-Copy Locking: Each logical element X has one of its physical copies designated as the "primary copy." The site holding the primary copy is responsible for X's lock. This distributes lock management across multiple sites. Messages: still 3 per lock (request → grant → release), except when the transaction is at the primary copy's site. Bottleneck: avoided because different elements have primaries at different sites, spreading the load. Failure: if a primary copy's site crashes, only locks for elements whose primary is at that site are affected — partial rather than total failure.

3. Majority Locking: No copy is "primary." A transaction broadcasts a lock request to all sites holding a copy of X. It obtains a global lock only after receiving local locks from a majority (more than half) of the sites. Messages: 1 broadcast request + responses from majority + 1 broadcast release — approximately 2n+1 messages for n sites (more expensive). Bottleneck: no single bottleneck since all copies are symmetric. Failure: tolerates minority of site failures — as long as a majority is reachable, locks can still be obtained. Drawback: higher message overhead and potential for deadlocks if two transactions each obtain locks from different non-overlapping majorities (though this is prevented by the majority quorum intersection property).

W84M
Explain the MapReduce execution flow in Hadoop. Describe each step from job submission to final output, including the roles of JobTracker, TaskTracker, NameNode, and DataNode.

Step 1 — Job Submission: The client submits a job to the cluster, which includes the JAR file containing the Mapper and Reducer logic, the input data path, and the output data path.

Step 2 — Job Scheduling (JobTracker): The JobTracker (running on the master node alongside the NameNode) accepts the job, splits it into individual tasks, and assigns Map and Reduce tasks to TaskTrackers on slave nodes. Assignment is based on data locality — tasks are preferentially sent to nodes that already hold the relevant data blocks.

Step 3 — Map Phase (TaskTrackers + DataNodes): Each TaskTracker (on a slave node) executes its assigned Map task on the input split stored locally on its DataNode. The Map function processes each record and emits key-value pairs. DataNodes store the actual data blocks of HDFS files; the NameNode stores metadata (file names, block locations, replication info).

Step 4 — Shuffle and Sort: The intermediate key-value pairs output by mappers are shuffled across the network — all values for the same key are grouped together and sent to the same reducer. This involves sorting by key and transferring data between nodes.

Step 5 — Reduce Phase: TaskTrackers execute Reduce tasks on the grouped intermediate data. Each reducer processes all values for a given key and produces the final output key-value pairs.

Step 6 — Output: The final output from all reducers is written back to HDFS. The NameNode manages the metadata for these output files, and DataNodes store the actual output blocks with replication for fault tolerance (default replication factor is 3).

Fault tolerance: If a TaskTracker fails, the JobTracker reassigns its tasks to another node. Data is resilient because HDFS replicates each block across multiple DataNodes.
