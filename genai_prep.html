Explain the three-stage pretraining strategy of BLIP — Image-Text Contrastive (ITC), Image-Text Matching (ITM), and Image-Text Generation (ITG). How does BLIP's bootstrapped pretraining handle noisy web image-text data? 4M

Hint: Cover the goal and mechanism of each stage, and explain the self-improving loop.

ITC: Maximizes cosine similarity between correct image-text pairs and minimizes similarity between mismatched pairs (contrastive loss in shared embedding space).
ITM: Binary classification — given image-text pair, predict match or non-match. Helps fine-grained alignment.
ITG: Auto-regressive caption generation from image using causal language modeling.
Bootstrapping: BLIP's captioner generates pseudo-labels for noisy web data → filter module removes mismatched pairs using ITM → cleaner data is used to retrain iteratively, improving quality each round.

You are building a customer support chatbot for an insurance company. The naive RAG system frequently retrieves irrelevant clauses and generates incorrect answers. Explain how Hybrid Search (BM25 + Dense) and Re-ranking with Cross-Encoders would solve this problem, with concrete steps. 4M

Problem: Naive keyword search (BM25) retrieves by exact terms — misses semantically similar clauses phrased differently. Pure dense search may miss exact policy terms.
Hybrid Search: Score = α·BM25(q,d) + (1−α)·CosineSim(q,d). BM25 handles precise policy terms; dense retrieval captures paraphrased queries. Combine for both precision and recall.
Re-ranking: Initial top-k docs pass through a Cross-Encoder (processes query + document together via BERT-like model) → assigns fine-grained relevance scores → reorders results so most relevant clause appears first.
Impact: LLM receives accurate, domain-relevant clauses → reduced hallucination, higher factual grounding in insurance-specific responses.

Theoretical

Write the LoRA weight update formula and explain each term. If the weight matrix has d = 512, k = 512 and rank r = 4, calculate the number of trainable parameters in LoRA and compare it with full fine-tuning. What is the percentage reduction? 4M

Formula: W' = W + ΔW = W + (α/r) · BA, where W = frozen pretrained weight (d×k), B (d×r) and A (r×k) are trainable low-rank matrices, α = scaling factor, r = rank.
LoRA params: r(d + k) = 4(512 + 512) = 4,096
Full fine-tuning params: d × k = 512 × 512 = 262,144
Reduction: (1 − 4096/262144) × 100 ≈ 98.4% fewer parameters
This allows training on consumer GPUs, faster iterations, and multiple task-specific adapters stored cheaply alongside one frozen base model.

Compare Early Fusion, Late Fusion, and Hybrid Fusion strategies in multimodal AI models. For each, name a real model that uses it, describe the mechanism, and identify the best use case. 4M

Early Fusion: Raw features from all modalities are combined before processing. Example: CLIP (joint embedding), ImageBind. Best for tasks requiring tight feature integration (image captioning, video understanding).
Late Fusion: Each modality processed independently → outputs combined at decision level via averaging/voting. Example: VQA 1.0, HUME. Best for independent modality insights (sentiment analysis, multimodal emotion).
Hybrid Fusion: Combines early + late fusion, integrating at multiple levels. Example: Flamingo (DeepMind), GPT-4V. Best for complex tasks needing both feature-level and decision-level reasoning (autonomous driving, video QA).

Theoretical

Explain Self-RAG (Self-Reflection Mechanism). What is the role of the "critique module" and "reflection tokens"? How does Self-RAG differ from FLARE (Flexible Retrieval Augmented Response Generation)? 4M

Self-RAG Workflow: (1) LLM generates an initial answer. (2) Critique module uses special "reflection tokens" to evaluate — Is the content supported by evidence? Is it relevant? Is it useful? (3) If critique fails, model revises the answer.
Reflection tokens: Special tokens like [IsRel], [IsSup], [IsUse] that the model uses to self-assess its output against retrieved documents — no external verifier needed.
FLARE (Flexible Retrieval): Instead of pre-retrieving everything, FLARE detects knowledge gaps during generation and triggers additional retrieval dynamically. The model decides when it needs more info mid-generation.
Key difference: Self-RAG = post-generation self-critique and revision. FLARE = adaptive retrieval during generation when model detects uncertainty.

Theoretical

Describe the complete architecture and operational pipeline of OpenAI's Whisper ASR model. How does it use special tokens to achieve multitask capability in a single model? What are its three key limitations? 4M

Architecture: Encoder-Decoder Transformer. Pipeline: Audio → resample to 16kHz → split into 30-second segments → log-Mel spectrogram → Encoder (extracts acoustic features via attention) → Decoder (generates text token-by-token using encoded representation).
Special Tokens: Whisper uses special control tokens prepended to the decoder input to specify the task: e.g., [TRANSCRIBE] for speech-to-text in the same language, [TRANSLATE] for translation to English, and language detection tokens. This allows a single model to perform multiple tasks without retraining — just changing the starting tokens steers the output.
Limitations: (1) May hallucinate text, especially in noisy/unclear audio. (2) Does not support speaker diarization (cannot identify who is speaking). (3) Real-time performance is limited, particularly with the large model variants; performance varies on low-resource languages.

Theoretical

Write and explain the complete BM25 scoring formula. Define each term: f(t, D), IDF(t), k₁, b, avgD. Explain what happens to the score when (a) b = 0 and (b) b = 1. How does BM25 differ from TF-IDF? 4M

Formula: BM25(q,d) = Σ IDF(tᵢ) × [f(tᵢ,d) × (k₁+1)] / [f(tᵢ,d) + k₁×(1−b + b×|d|/avgD)]
Terms: f(t,D) = term frequency of term t in document D; IDF(t) = log((N−nₜ+0.5)/(nₜ+0.5)) (how rare the term is); k₁ = term saturation (default 1.5); b = length normalization (default 0.75); avgD = average document length in corpus; |d| = length of current document.
b=0: No length normalization — long documents are not penalized. Favors longer documents.
b=1: Full normalization — document length fully factored in, exactly like TF-IDF normalization.
vs TF-IDF: BM25 adds saturation via k₁ (term frequency impact levels off after a point) and tunable length normalization via b. TF-IDF linearly increases with TF; BM25 saturates.

Numerical

Quantize the following weight matrix using 4-bit (b=4) quantization. Then dequantize and compute the element-wise quantization error and MAE. 4M

W = [[3.0, −1.5], [0.5, −3.0]]  |  b = 4 bits

Step 1: min = −3.0, max = 3.0
Step 2: δ = (3.0 − (−3.0)) / (2⁴ − 1) = 6.0 / 15 = 0.4
Step 3 – Quantize Q = round((W − (−3)) / 0.4):
(3.0+3)/0.4 = 15 → Q=15  |  (−1.5+3)/0.4 = 3.75 → Q=4
(0.5+3)/0.4 = 8.75 → Q=9  |  (−3+3)/0.4 = 0 → Q=0
Step 4 – Dequantize Ŵ = Q×0.4 + (−3):
15×0.4−3 = 3.0; 4×0.4−3 = −1.4; 9×0.4−3 = 0.6; 0×0.4−3 = −3.0
Step 5 – Errors |W − Ŵ|:
|3.0−3.0|=0; |−1.5−(−1.4)|=0.1; |0.5−0.6|=0.1; |−3.0−(−3.0)|=0
MAE = (0 + 0.1 + 0.1 + 0) / 4 = 0.05

Scenario

A research team wants to build an enterprise knowledge assistant that must maintain context across long multi-turn conversations spanning weeks. Design a Memory-Augmented RAG system. Describe the types of memory, storage mechanisms, and retrieval strategies you would use, and how they interact. 4M

Memory Types:
• Short-Term Memory: Current conversation window stored in-context (LLM context buffer for recent exchanges).
• Long-Term Memory: Vector database (FAISS/Pinecone/Weaviate) stores embeddings of past conversations, decisions, and retrieved documents indexed by recency and relevance.
• Knowledge Graph (optional): Entities and relationships (user → project → documents) stored as a graph for structured recall.
Storage Mechanisms: Embed past interactions → store with timestamp + user metadata in vector DB. Periodic summarization: compress old turns into summary vectors.
Retrieval Strategies: (1) For each new query, embed it → similarity search in long-term memory → inject top-k relevant past interactions into context. (2) Query expansion based on session history. (3) Re-ranking retrieved memories by recency and semantic relevance.
Interaction Flow: New query → Short-term context + retrieved long-term memories → RAG retrieval from knowledge base → LLM generation grounded in full context → response stored back to long-term memory.

Theoretical

Explain the concept of Mixture of Experts (MoE) in LLMs. How does sparse activation work? Why does Mixtral 8x7B have approximately 47B effective parameters during inference rather than 56B (8×7B)? What are two challenges of MoE in production? 4M

MoE Concept: Instead of one large dense feedforward network, the model uses N expert FFN subnetworks. A gating network (router) selects Top-K experts for each input token, activating only those for computation.
Sparse Activation: At any given token, only K of N experts are active (e.g., 2 of 8 in Mixtral). The rest contribute nothing. This gives a model with more total parameters but a much lower compute cost per forward pass.
47B not 56B: In MoE architectures, only the FFN layers are replaced by expert layers. The other layers (attention, embeddings, norms) are shared and counted once. Mixtral's shared parameters + 2 active experts contribute ~47B effective compute, not the full 56B.
Challenges: (1) High memory requirement: All 47B parameters must be loaded in VRAM even though only ~12B are active — limits deployment on small hardware. (2) Expert load imbalance: Some experts may be over-used while others are under-utilized — requires auxiliary loss functions for balanced routing during training.
