Explain why GPT is a decoder-only architecture. List its core components (embedding dimension, attention heads, FFN size, etc.) from the original design.

Why GPT is Decoder-Only

GPT is designed specifically for text generation, where the model must predict future tokens without seeing them. This drives the architectural choice:

  • Sequential generation: While generating a sentence, the model can only use previously generated tokens.
  • Encoder limitation: The Transformer encoder uses bidirectional attention (token sees both past and future). This is ideal for understanding but unsuitable for generation.
  • Decoder advantage: The decoder uses unidirectional (causal) attention, so predictions depend only on past context.
  • Architectural simplification: Since GPT doesn't need full-sequence understanding before generating output, the encoder is unnecessary — only the decoder stack is retained.

Core Components (Original Design)

  • Embedding dimension d_model = 512 (varies by GPT version)
  • Number of attention heads = 8
  • Each head: Query/Key dimension d_k = 64, Value dimension d_v = 64
  • Feed-Forward Network hidden size = 2048
  • Decoder block contains: masked multi-head self-attention + FFN + residual connections + layer normalization
  • Final output: linear layer + softmax over vocabulary

Differentiate between self-attention and cross-attention with respect to Q, K, V sources, purpose, and where they appear in the Transformer architecture.
AspectSelf-AttentionCross-Attention
Q sourceSame input sequenceDecoder hidden states
K, V sourceSame input sequenceEncoder output
PurposeCapture relationships among tokens within one sequenceAllow decoder to focus on relevant encoder tokens
Used inBoth encoder & decoderDecoder only

Self-attention: Each token computes weighted sum of all tokens in the same sequence — produces contextualized representations. Used in encoder (bidirectional) and decoder (masked/causal).

Cross-attention: The decoder Query attends over encoder Keys and Values, bridging the input sequence to the target sequence. It appears as the middle sublayer of each decoder block (between masked self-attention and FFN).

Computation is identical to self-attention; only the source of Q vs (K,V) changes.

Why is the dot-product attention score scaled by √dk before softmax? What happens if you skip this step?

Purpose: To stabilize softmax and improve gradient flow during training.

As d_k grows, dot-product values Q·K grow in magnitude (variance scales with d_k). Large values pushed through softmax produce extremely sharp (peaked) distributions where one entry ≈ 1 and others ≈ 0.

If skipped:

  • Softmax saturates → gradients become vanishingly small (the gradient of softmax in saturated regions is near zero).
  • Training becomes unstable; the model struggles to learn.
  • Numerical overflow risk when exponentiating large scores.

Dividing by √d_k keeps the variance of the dot products approximately constant (≈ 1), so softmax produces well-behaved attention distributions regardless of embedding dimension.

Define prompt engineering. Differentiate zero-shot, one-shot, and few-shot prompting with one example each.

Prompt engineering is the practice of designing and structuring inputs (prompts) to effectively guide an LLM's behavior and outputs. It reduces vague/incorrect responses, improves control, and optimizes performance across tasks.

Zero-Shot Prompting

Only an instruction is given, no examples. Relies entirely on pre-trained knowledge.

Example: "Classify this review as positive or negative: 'The product is amazing.'"

One-Shot Prompting

One example is given along with the instruction to clarify task format.

Example: "Review: 'Great lectures' → Positive. Now classify: 'The pace was too fast' → ?"

Few-Shot Prompting

Several examples shown before the actual task; the model learns the pattern.

Example: "Apple → Manzana, Banana → Plátano, Orange → ?" → "Naranja"

(a) What is RAG? How does it differ from a traditional LLM? (3)
(b) Explain the RAG Evaluation Triad (TruEra model). (3)

(a) RAG vs Traditional LLM

Retrieval-Augmented Generation (RAG) combines information retrieval with text generation. It augments an LLM with access to external knowledge stores (vector DBs, document repositories) at inference time.

Differences from a traditional LLM:

  • Traditional LLM relies only on parametric memory (knowledge baked into weights at training).
  • RAG fetches relevant documents at query time → answers are grounded in external data.
  • RAG handles knowledge cut-off, reduces hallucinations, and supports private/domain-specific data.

(b) RAG Evaluation Triad (TruEra)

Three relationships evaluated:

  • Context Relevance (Query ↔ Context): Is the retrieved data relevant to the question? Bad retrieval → bad answers.
  • Groundedness (Context ↔ Answer): Is the generated answer actually based on the retrieved context? If the LLM ignores context → hallucination.
  • Answer Relevance (Query ↔ Answer): Does the answer address the user's question? Even with good retrieval, answer may be off-topic.

What is Parameter-Efficient Fine-Tuning (PEFT)? Why is it needed for modern LLMs?

PEFT is a class of techniques that fine-tune a small subset of model parameters while keeping the bulk of the pretrained model frozen.

Why PEFT is Needed

  • Cost: Modern LLMs have billions of parameters; full fine-tuning requires huge GPU memory and is expensive.
  • Time: Training all parameters takes very long.
  • Storage: Storing one full-model copy per downstream task is impractical.
  • Hardware: Difficult on consumer or modest GPUs.

PEFT approach: Freeze base model → train only small "adapters" (e.g. LoRA matrices, prefix vectors, prompt tokens). Faster, cheaper, achieves comparable performance to full fine-tuning. Examples: LoRA, QLoRA, prompt tuning, prefix tuning, adapter layers.

Theory5 marks
Explain the three key techniques of QLoRA: NF4 quantization, double quantization, and paged optimizer.

1. NF4 Quantization (NormalFloat 4-bit)

A special 4-bit numerical format designed for neural network weights. Weights typically follow approximately a normal distribution, and NF4 places its 16 quantization levels in a distribution-aware way so that values near zero get finer resolution.

  • Better than standard int4
  • Preserves important weight magnitudes
  • Reduces accuracy loss after quantization

2. Double Quantization

Quantization itself produces scaling/zero-point constants per block, which also consume memory. Double quantization quantizes the quantization constants, compressing them further.

  • Saves an additional ~0.4 bits per parameter on average
  • Improves storage efficiency without significant accuracy loss

3. Paged Optimizer

During training, optimizer states (Adam moments) can exceed GPU memory, especially with large gradient batches. Paged optimizers swap states between CPU RAM and GPU VRAM on-demand (using unified memory).

  • Prevents out-of-memory crashes
  • Enables training large models on small/consumer GPUs (e.g. 24 GB)
  • Improves stability for long training runs

Role of α/r Scaling Factor

The LoRA update is: W' = W + (α/r)·BA

  • Controls magnitude of the LoRA update on the original weights.
  • Acts like a learning-rate scale specifically for the adapter — letting you tune r without re-tuning the overall update strength.
  • Maintains training stability: small r with appropriate α keeps updates well-conditioned.
  • Without this scaling, changing rank would unintentionally change the effective step size of fine-tuning.

Explain the ReAct pattern in agentic AI with a step-by-step example. Why is it preferred over a one-shot plan-then-execute approach for dynamic tasks?

ReAct (Reason + Act)

The agent alternates between reasoning and action. It thinks about what to do, performs an action (tool call, API request, search), observes the result, then reasons again. The loop continues until the goal is reached.

Step-by-Step Example

Task: "Find the cheapest flight from New York to Seattle next weekend."

  1. Reason: I need flight prices for next weekend.
  2. Act: Call flight search API.
  3. Observe: Flights found at $280, $310, $355.
  4. Reason: $280 is cheapest, but I should check baggage fees.
  5. Act: Call airline details API.
  6. Observe: $280 flight has $70 baggage fee → total $350.
  7. Reason: The $310 option may actually be cheaper overall.
  8. Final: Recommend the $310 flight.

Why ReAct > Plan-then-Execute

  • Adaptive: Plans formed at the start may be invalidated by new observations. ReAct adjusts step-by-step.
  • Lower hallucination: Grounding each step in real tool output reduces fabricated reasoning.
  • Handles uncertainty: When intermediate results surprise the model, it can recover gracefully.
  • Better for tool-heavy tasks: Search, code, retrieval — where the next action depends on what just happened.

Explain Whisper's architecture and how it performs multiple tasks (transcription, translation, language detection) using a single model.

Architecture

Whisper uses an encoder-decoder Transformer trained on ~680,000 hours of diverse multilingual audio.

  • Input: Raw audio is converted into a log-Mel spectrogram (numerical time-frequency features).
  • Encoder: Processes spectrogram features and produces hidden audio representations.
  • Decoder: Autoregressively generates text tokens, attending over the encoder's audio context.

Multi-task via Special Tokens

Whisper uses special control tokens in the decoder input to switch behavior:

  • <|transcribe|> → output text in the same language as audio
  • <|translate|> → output English translation
  • <|en|>, <|es|>, … → language identifier tokens
  • Timestamp tokens for time-aligned outputs

One unified model handles transcription, translation, language detection, and voice activity detection — selected by the prompt structure rather than separate models.

Key strength: Large diverse training data → robust across accents, noise, and 90+ languages.

  • Cherry-picking examples instead of using representative test sets
  • Ignoring statistical significance — small improvements may be noise
  • (Others: using reference-based metrics for open-ended tasks, assuming humans are always right, demographic bias in test sets)

Explain Causal Language Modeling (CLM) vs Masked Language Modeling (MLM). Which models use each, and what's the trade-off?

Causal Language Modeling (CLM)

  • Predicts the next word from previous words.
  • One-directional (left-to-right) context.
  • Used in: GPT family, decoder-only models.
  • Strength: Natural fit for text generation.

Masked Language Modeling (MLM)

  • Some tokens are masked; model predicts them.
  • Bidirectional context (uses both left and right).
  • Used in: BERT, encoder-only models.
  • Strength: Stronger language understanding (richer context).

Trade-off

  • CLM → great for generation, weaker pure-understanding tasks.
  • MLM → great for understanding (classification, NER), cannot directly generate fluent long text.

Describe the role of (a) positional encoding, (b) residual connections, (c) layer normalization in the Transformer architecture.

(a) Positional Encoding (PE)

  • Transformers process tokens in parallel, so they have no inherent sense of order.
  • PE injects position information by adding sine/cosine values (or learned vectors) to embeddings.
  • PE(pos, 2i) = sin(pos / 10000^(2i/d)) and PE(pos, 2i+1) = cos(...)
  • Without PE: "I love AI" and "AI love I" look identical to the model.

(b) Residual Connections

  • Add the input of a sublayer to its output: x + Sublayer(x).
  • Prevents vanishing gradients in deep networks.
  • Allows information to flow even if a layer learns near-zero updates.
  • Enables stacking many layers (encoder/decoder depth = 6+).

(c) Layer Normalization

  • Normalizes activations across feature dimensions for each token.
  • Stabilizes training, reduces internal covariate shift.
  • Speeds up convergence.
  • Applied around each sublayer together with residual connections.

Explain Chain-of-Thought (CoT) and Tree-of-Thought (ToT) prompting with one example each. When is ToT preferred?

Chain-of-Thought (CoT)

Model is prompted to break problems into intermediate reasoning steps before the final answer. A single linear chain of reasoning.

Example: "If a train travels at 60 km/h for 2 h, then 80 km/h for 3 h, total distance?"
Reasoning: 60×2 = 120 km; 80×3 = 240 km; Total = 360 km.

Tree-of-Thought (ToT)

Model explores multiple reasoning paths in parallel (branching structure), evaluates each, and selects the best. Supports backtracking.

Example: "Best way to allocate $10,000 for school development?" → Explores branches: renovation / technology / extracurriculars → compares trade-offs → picks 50/30/20 split.

When ToT is Preferred

  • Complex decision-making (planning, optimization)
  • Multiple plausible strategies must be compared
  • Problems where one mistake on a single path is hard to recover from
  • Games, puzzles, strategic reasoning

What are temperature, top-k sampling, and context window? How do they affect LLM output?

Temperature

  • Controls randomness/creativity in token sampling.
  • Low (0.1–0.3): more predictable, factual, deterministic.
  • High (0.7–1.2): more diverse, creative, sometimes unstable.

Top-k Sampling

  • Restricts choice to the k most probable next tokens.
  • Lower k (e.g. 10): focused output.
  • Higher k (e.g. 100): more variety.

Context Window

  • Fixed maximum number of tokens the model can process at once.
  • Tokens beyond this limit are dropped/truncated (older context "forgotten").
  • Affects long-document summarization, multi-turn chat, and RAG chunk sizing.

Combined effect: Temperature + top-k shape output style/creativity; context window determines how much info the model can reason over.

Explain the architecture of a RAG system. Describe each layer (indexing, generation, evaluation, monitoring, orchestration).

A production RAG system has several coordinated layers:

1. Indexing Layer

  • Documents are chunked and embedded into vectors.
  • Stored in a vector database (FAISS, Pinecone, Weaviate, Chroma).
  • Inverted indexes built for sparse retrieval (BM25).

2. Generation Layer

  • User query is embedded and used to retrieve top-k relevant chunks.
  • Retrieved context + query is sent to the LLM as a prompt.
  • LLM generates the final grounded answer.

3. Evaluation Layer

  • Measures retrieval quality (Precision@k, Recall@k), groundedness, and answer relevance.
  • Frameworks: TruLens, DeepEval, RAGAS.

4. Monitoring Layer

  • Tracks latency, hallucination rates, user feedback in production.
  • Logs failures for analysis.

5. Orchestration Layer

  • Central controller that routes the flow between layers.
  • Manages retries, fallbacks, security checks, and component interaction.

Flow: Query → Embed → Retrieve → Re-rank → Filter → Build Prompt → LLM → Answer → (Evaluate + Monitor).

Write the LoRA weight update formula. Explain each term (W, A, B, r, α).

LoRA update:

W' = W + ΔW = W + (α/r) · B·A

Terms

  • W: Original pretrained weight matrix (frozen, dimension d × k).
  • A: Trainable matrix of shape r × k (down-projection).
  • B: Trainable matrix of shape d × r (up-projection).
  • r: Rank — typically very small (r ≪ d, k); controls capacity of the adapter.
  • α: Scaling factor that controls the magnitude of the update; α/r acts as an effective learning-rate scale.

Since BA is low-rank, the number of trainable parameters is r·(d + k) — far fewer than the d·k of full fine-tuning.

Theory5 marks
How does rank (r) affect LoRA performance? What's the trade-off between small and large r?

Effect of Rank r

r controls the expressive capacity of the LoRA adapter (BA is rank-r at most).

Small r (e.g. 4–8)

  • Very few trainable parameters → fast, cheap, low memory.
  • May underfit complex tasks (limited capacity).
  • Best for tasks similar to pretraining domain.

Large r (e.g. 64–128)

  • More capacity → can adapt to harder, more distant domains.
  • Slower training, more memory, larger adapter to store.
  • Risk of overfitting on small datasets.

Trade-off Summary

  • r is a capacity ↔ efficiency dial.
  • Common practical range: r = 4 to 32.
  • α should be tuned together with r (often α = r or α = 2r).

Describe the practical fine-tuning workflow using QLoRA in 5 steps. Mention the role of the Hugging Face PEFT library.

QLoRA Fine-Tuning Workflow

  1. Load pretrained model — e.g. Llama-2 7B from Hugging Face Hub.
  2. Quantize to 4-bit (NF4) — base model weights stored in 4-bit; activations stay in higher precision (e.g. bfloat16).
  3. Add LoRA adapters — attach trainable low-rank matrices to attention layers (Q, K, V projections).
  4. Train adapters — using paged optimizer, only the LoRA matrices update; base weights stay frozen.
  5. Save / merge model — save adapter weights separately (small footprint) or merge into base for inference.

Role of Hugging Face PEFT Library

  • Provides LoraConfig and get_peft_model() to wrap any HF model with LoRA adapters in 2–3 lines.
  • Integrates with bitsandbytes for 4-bit quantization (NF4, double quantization).
  • Handles adapter saving/loading via PeftModel.from_pretrained().
  • Compatible with HF Trainer for standard fine-tuning loops.
  • Massively reduces boilerplate compared to manual implementation.

Explain CLIP. What is contrastive learning and why does CLIP enable zero-shot classification?

CLIP (Contrastive Language-Image Pretraining)

Developed by OpenAI, CLIP learns a shared embedding space for images and text by training on hundreds of millions of (image, caption) pairs from the web.

Contrastive Learning

  • For each training batch, the model is given many image-text pairs.
  • Maximize similarity between matching pairs.
  • Minimize similarity between non-matching pairs.
  • Result: matching images and texts end up close in embedding space.

Zero-Shot Classification

To classify an image into N categories without task-specific training:

  1. Create text prompts like "a photo of a {class}" for each class.
  2. Embed image and all text prompts.
  3. Pick the class whose text embedding has highest cosine similarity with the image.

No labeled training data needed — the joint embedding space generalizes to new classes via natural language descriptions.

What are agentic design patterns and why are they needed? List any 5 patterns.

An agentic design pattern is a reusable solution to a common system-design problem in agentic AI. It describes proven ways to organize reasoning, planning, tool use, memory, validation, and coordination among one or more agents.

Why They're Needed

  • Agentic systems are more complex than single-prompt LLM calls.
  • Without structure, workflows are unstable, expensive, and hard to debug.
  • Patterns improve reliability, reduce hallucination, separate concerns, and make systems easier to scale and monitor.

Five Popular Patterns

  1. Planning Pattern — decompose goal into sub-tasks before acting.
  2. ReAct Pattern — alternate reason ↔ act loop.
  3. Retrieval-Augmented Generation (RAG) — fetch documents before generating.
  4. Tool-Use Pattern — call APIs, code interpreter, calculators.
  5. Reflection / Self-Critique — agent reviews and improves its own output.

(Others: Router, Orchestrator-Worker, Multi-Agent, Memory, Human-in-Loop, Guardrails.)

Theory5 marks
Why is masking used in the decoder's self-attention? Show with an example how the mask prevents data leakage.

Why Masking?

During training, the decoder is fed the entire target sequence in parallel to enable fast training. Without masking, when predicting position t, the model could attend to future tokens (positions > t) — this is data leakage: the model would "cheat" by seeing the answer.

How the Mask Works

  • A causal mask M is added to the scaled scores before softmax.
  • Future positions get −∞; current/past positions get 0.
  • After softmax, attention weights for future tokens become exactly 0.

Example

Predicting "Word 3" in target "Word 1 Word 2 Word 3 Word 4". The mask matrix for 4 tokens:

t1t2t3t4
t10−∞−∞−∞
t200−∞−∞
t3000−∞
t40000

When predicting t3, attention to t4 is killed by the mask → no leakage. The model can train in parallel while preserving autoregressive integrity.

Theory6 marks
Explain the input processing pipeline of a Transformer: tokenization → embeddings → positional encoding. Why is final input = embedding + PE and not concatenation?

1. Tokenization

Raw text is segmented into discrete tokens using subword tokenization (BPE, WordPiece). Each token is mapped to an integer index from a fixed vocabulary (typically 30k–50k tokens).

2. Token Embeddings

Each token index is looked up in an embedding matrix → dense vector of dimension d_model (e.g. 512). These embeddings encode semantic and syntactic properties; they're learned during training.

3. Positional Encoding (PE)

Since transformers process tokens in parallel (no recurrence), they have no inherent order awareness. PE is added to each token embedding to inject position info using sine/cosine functions of different frequencies.

Why Addition (not Concatenation)?

  • Keeps dimension fixed at d_model — no extra parameters needed in downstream layers.
  • Allows joint learning of content + position in the same vector space.
  • Compatible with residual connections which require matching dimensions throughout the network.
  • Concatenation would double the dimension and force every subsequent weight matrix to be larger.
  • Empirically: addition works well because PE values are bounded (−1 to 1) and don't overwhelm the semantic embedding.

What is document chunking? Why is it important in RAG? Explain hierarchical chunking briefly.

Document Chunking

The process of splitting large documents into smaller meaningful segments before embedding and indexing.

Why It's Important

  • Fits LLM context window: Whole documents may exceed the model's input limit.
  • Improves retrieval precision: Smaller chunks reduce noise; the LLM gets only relevant passages.
  • Better semantic embeddings: A small focused chunk has a more concentrated meaning than a large mixed document.
  • Typical sizes: 256–1024 tokens, often with overlap (e.g. 10–20%) to preserve continuity across boundaries.

Hierarchical Chunking

Splits documents into structured levels: document → section → paragraph → sentence. Each level has its own embedding. At retrieval time, the system can:

  • Use high-level chunks for broad context.
  • Drill down to fine-grained chunks for specific facts.
  • Return parents of matching children to provide surrounding context.

Improves accuracy on long documents (legal, medical, research papers) where flat chunking loses structural context.

(a) What is parametric vs non-parametric memory in LLMs? (3)
(b) Explain how RAG bridges this gap and reduces hallucinations. (3)

(a) Parametric vs Non-Parametric Memory

Parametric memory:

  • Knowledge stored inside model weights (billions of parameters).
  • Learned during training, compressed lossy into weights.
  • Cannot be updated instantly; requires retraining or fine-tuning.
  • May recall facts incorrectly (hallucination).

Non-parametric memory:

  • External knowledge — databases, PDFs, APIs, vector stores.
  • Stored outside the model.
  • Can be updated anytime without retraining.
  • Effectively unlimited capacity.

(b) How RAG Bridges This Gap

  • At query time, RAG retrieves relevant docs from non-parametric memory.
  • Retrieved context is injected into the prompt → LLM generates an answer grounded in real, current data.
  • Reduces hallucinations because answers cite specific retrieved content instead of relying purely on (lossy) parametric recall.
  • Solves knowledge-cutoff: external corpus can include up-to-date information the base model never saw.
  • Handles private/domain-specific data without retraining.

What are the challenges of full fine-tuning of LLMs? How does PEFT solve them?

Challenges of Full Fine-Tuning

  • GPU memory: A 13B model needs ~70 GB just to hold parameters + gradients + optimizer states.
  • Training time: Updating billions of parameters takes long even with multi-GPU clusters.
  • Storage: One full fine-tuned model per downstream task → bloat.
  • Cost: High electricity and cloud GPU costs.
  • Catastrophic forgetting: Risk of losing general knowledge while specializing.

How PEFT Solves These

  • Freeze the base model → no gradient computation for 99% of parameters.
  • Train tiny adapter modules (LoRA, prefix tokens) → memory drops dramatically.
  • Adapter sizes are MBs not GBs → easy to share and version per task.
  • Base model unchanged → general knowledge preserved.
  • Multiple tasks can share one base model with different adapters loaded on demand.

Explain Soft Prompts and Prompt Tuning. How do they differ from prompt engineering?

Soft Prompts

Continuous, learnable vectors prepended to the input embeddings. Unlike discrete text prompts, they live directly in the model's embedding space and don't correspond to real words.

Prompt Tuning

The process of optimizing these soft-prompt vectors via backpropagation while keeping the base model frozen. Introduced by Lester et al. (Google, 2021).

Advantages

  • Minimal model changes — only the soft-prompt embedding is trained.
  • Lower training cost than full fine-tuning or even LoRA.
  • Multiple soft prompts can be stored separately and swapped per task.

Differences from Prompt Engineering

Prompt EngineeringPrompt Tuning (Soft Prompts)
Manual; uses natural-language textLearned automatically via gradient descent
Discrete tokens (real words)Continuous embedding vectors (no word equivalent)
No training requiredRequires labeled task data and training
Limited expressivenessCan capture task-specific signal not expressible in words

Challenges

May underperform on complex tasks needing substantial model adaptation; less effective than LoRA on smaller models.

Define an agentic loop. Show the cycle (Plan → Act → Observe → Reflect) with a simple real-world task.

Agentic Loop

A continuous cycle where an agent plans, acts, observes the result, reflects on it, and iterates until the goal is achieved. It's looping intelligence — not a one-shot response but continuous decision-making.

Example: "Book the cheapest flight to Mumbai next Friday"

  1. Plan: Search flights for Friday → compare prices → pick cheapest.
  2. Act: Call flight search API.
  3. Observe: Returns 5 options, prices $200–$450.
  4. Reflect: $200 option has 12-hour layover — bad. Should I optimize for total trip time, not just price?
  5. Loop: Re-plan → filter by max-layover 4h → re-search.
  6. Act: Re-call API with new filters.
  7. Observe: $280 direct flight available.
  8. Final Output: Recommend the $280 direct flight.

The loop continues refining until acceptable solution emerges.

Differentiate traditional AI workflow vs agentic workflow. List the 4 abilities that make a system "agentic".

Traditional AI (Linear)

Input → Model → Output

  • One-step process
  • No decision-making
  • No adaptation, no tool usage
  • Static; doesn't verify or improve

Agentic Workflow (Non-Linear)

Goal → Plan → Act → Observe → React → Repeat

  • Multi-step process with feedback loops
  • Dynamic decision-making
  • Uses external tools (APIs, code, search)
  • Adapts based on observed results

4 Abilities of an Agentic System

  1. Goal-Oriented: Works toward an explicit goal (e.g. "find best laptop under ₹80k").
  2. Planning: Decides the sequence of steps to take.
  3. Action (Execution): Performs actions like search, API calls, code execution.
  4. Observation & Reaction: Inspects results and adapts strategy if needed.

Explain any 3 agentic design patterns in detail: (a) Planning, (b) Tool-Use, (c) Reflection/Self-Critique. Give one task example per pattern.

(a) Planning Pattern

The agent transforms the goal into a sequence of sub-tasks before execution. Decomposes a complex problem so each step is manageable and dependencies are explicit.

Where to add: In a planner layer before tool execution.

Example task: "Plan a 3-day budget trip to New York."
Plan generated:

  1. Set total budget to $400
  2. Find bus/train transport options
  3. Search hostels under $60/night
  4. Group nearby attractions day-wise
  5. Add food + local travel estimate
  6. Produce final itinerary

(b) Tool-Use Pattern

The agent calls external tools (search, calculator, code interpreter, APIs) to do things it can't do reliably from text generation alone.

Where to add: In the execution layer with a tool-selection / function-calling mechanism.

Example task: "What is 25% growth on monthly revenue of $84,500, and plot 6-month projection."
Agent: uses calculator tool for exact math → uses Python/matplotlib to produce projection graph → returns numbers + chart.

(c) Reflection / Self-Critique Pattern

The agent evaluates its own output against quality criteria and refines it before returning the final answer.

Where to add: After draft generation, before final response.

Example task: "Write a formal email requesting an extension."

  1. Draft 1: Initial email.
  2. Self-critique: Is tone professional? Is reason clear? Did I propose a new deadline?
  3. Revise: Improve weak points.
  4. Return polished final email.

(a) Compute BLEU 1-gram precision and brevity penalty: (5)

Candidate: "she went market"  (length c = 3)

(a) BLEU-1 with Brevity Penalty

Step 1 — Clipped 1-gram precision:

  • Candidate tokens: {she, went, market} → 3 tokens
  • All 3 appear in reference once each → matches = 3
  • Precision = 3/3 = 1.000

Step 2 — Brevity Penalty:

c = 3, r = 6. Since c < r:

BP = exp(1 − r/c) = exp(1 − 6/3) = exp(−1) ≈ 0.368

Step 3 — Final BLEU-1:

BLEU-1 = BP · precision = 0.368 × 1.000 = 0.368

BLEU-1 ≈ 0.368 (low despite perfect precision, because the candidate is too short)

This shows why BLEU needs BP — without it, short candidates with high precision would get inflated scores.

(b) DeepEval vs TruLens

DeepEvalTruLens
Comprehensive pre-built metrics libraryOpen-source with focus on explainability
Custom metric frameworkFeedback functions + dashboards
Strong CI/CD integration for dev pipelinesStrong instrumentation for production logs
Batch processing for offline evalReal-time monitoring of LLM calls
Statistical analysis & hypothesis testingPerformance tracking over time, A/B testing

For production monitoring: TruLens is better suited — it instruments live LLM calls, provides dashboards, and tracks metrics over time, which is essential for monitoring deployed systems.

(c) Why BERTScore > BLEU for Open-Ended Generation

  • BLEU relies on exact n-gram overlap with reference. For open-ended tasks (creative writing, summarization, dialogue), many valid responses are semantically correct but use different words → BLEU penalizes them unfairly.
  • BERTScore uses contextual embeddings (BERT) to compute semantic similarity between candidate and reference at the token level. It rewards paraphrases and synonyms.
  • Result: BERTScore correlates much better with human judgment on open-ended generation.
